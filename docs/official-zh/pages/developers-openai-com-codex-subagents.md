Codex 可以通过并行生成专业代理来运行子代理工作流，然后将它们的结果整合为一个响应。这在处理复杂且高度并行的任务时尤其有用，例如代码库探索或实现多步骤特性计划。

通过子代理工作流，您还可以根据任务定义自己的自定义代理，具有不同的模型配置和指令。

有关子代理工作流背后的概念和权衡，包括上下文污染、上下文衰退和模型选择指导，请参见 [子代理概念](https://developers.openai.com/codex/concepts/subagents)。

## 可用性

当前 Codex 版本默认启用子代理工作流。

子代理活动目前在 Codex 应用和 CLI 中显示。IDE 扩展的可见性即将推出。

Codex 仅在您明确要求时才生成子代理。由于每个子代理都进行自己的模型和工具工作，子代理工作流消耗的令牌比可比的单代理运行更多。

## 典型工作流

Codex 处理代理间的编排，包括生成新的子代理、路由后续指令、等待结果以及关闭代理线程。

当许多代理正在运行时，Codex 会等待所有请求的结果可用，然后返回一个合并的响应。

Codex 仅在您明确要求时才生成新的代理。

要查看其实际运作，请在您的项目中尝试以下提示：

```
我想审查当前 PR（此分支与主分支）的以下点。为每个点生成一个代理，等待它们的完成，并总结每个点的结果。
1. 安全问题
2. 代码质量
3. 错误
4. 竞争
5. 测试不稳定性
6. 代码的可维护性


```

## 管理子代理

- 在 CLI 中使用 `/agent` 在活动代理线程之间切换并检查正在进行的线程。
- 直接询问 Codex 以引导正在运行的子代理、停止它，或关闭已完成的代理线程。

## 审批和沙盒控制

子代理继承您当前的沙盒策略。

在交互式 CLI 会话中，即使您正在查看主线程，来自非活跃代理线程的审批请求也可以浮出水面。审批覆盖显示源线程标签，您可以按 `o` 打开该线程，然后再批准、拒绝或回答请求。

在非交互流程中，或者每当运行无法浮出新的审批时，需要新审批的操作将失败，并且 Codex 会将错误信息返回给父工作流。

Codex 还会在生成子代理时重新应用父级轮次的实时运行时覆盖。这包括您在会话期间交互设置的沙盒和审批选择，例如 `/approvals` 变更或 `--yolo`，即使所选自定义代理文件设置了不同的默认值。

您还可以针对单个 [自定义代理](https://developers.openai.com/codex/subagents#custom-agents) 覆盖沙盒配置，例如明确标记一个代理以只读模式工作。

## 自定义代理

Codex 附带内置代理：

- `default`：通用回退代理。
- `worker`：专注于实施和修复的执行代理。
- `explorer`：以读为主的代码库探索代理。

要定义自己的自定义代理，请在 `~/.codex/agents/` 中为个人代理添加独立的 TOML 文件，或在 `.codex/agents/` 中为项目范围内的代理。

每个文件定义一个自定义代理。Codex 将这些文件作为生成会话的配置层加载，因此自定义代理可以覆盖与普通 Codex 会话配置相同的设置。这可能觉得比专用代理清单更繁重，且格式可能会随着撰写和共享的成熟而演变。

每个独立的自定义代理文件必须定义：

- `name`
- `description`
- `developer_instructions`

可选字段如 `nickname_candidates`、`model`、`model_reasoning_effort`、`sandbox_mode`、`mcp_servers` 和 `skills.config` 在您省略它们时继承自父会话。

### 全局设置

全局子代理设置仍然位于您的 [配置](https://developers.openai.com/codex/config-basic#configuration-precedence) 中的 `[agents]` 下。

| 字段 | 类型 | 必需 | 目的 |
| --- | --- | --- | --- |
| `agents.max_threads` | number | 否 | 同时打开的代理线程上限。 |
| `agents.max_depth` | number | 否 | 生成的代理嵌套深度（根会话从 0 开始）。 |
| `agents.job_max_runtime_seconds` | number | 否 | `spawn_agents_on_csv` 作业的每个工作者的默认超时。 |

**注释：**

- 如果您不设置，则 `agents.max_threads` 默认为 `6`。
- 如果您不设置，则 `agents.max_depth` 默认为 `1`，这允许直接子代理生成，但阻止更深的嵌套。除非您特别需要递归委派，否则请保持默认值。提高该值可能将广泛的委派指令变为重复的发散，从而增加令牌使用、延迟和本地资源消耗。`agents.max_threads` 仍然限制同时打开的线程，但并未消除深层递归的成本和可预测性风险。
- `agents.job_max_runtime_seconds` 为可选项。如果您不设置，它会将 `spawn_agents_on_csv` 的每个工作者的默认超时回退为 1800 秒。
- 如果自定义代理名称与内置代理（如 `explorer`）匹配，则您的自定义代理优先。

### 自定义代理文件架构

| 字段 | 类型 | 必需 | 目的 |
| --- | --- | --- | --- |
| `name` | string | 是 | Codex 在生成或引用此代理时使用的代理名称。 |
| `description` | string | 是 | Codex 应该使用此代理时的人机可读指南。 |
| `developer_instructions` | string | 是 | 定义代理行为的核心指令。 |
| `nickname_candidates` | string\[\] | 否 | 为生成的代理提供的可选显示昵称池。 |

您还可以在自定义代理文件中包含其他支持的 `config.toml` 键，如 `model`、`model_reasoning_effort`、`sandbox_mode`、`mcp_servers` 和 `skills.config`。

Codex 通过其 `name` 字段识别自定义代理。将文件名与代理名称匹配是最简单的约定，但 `name` 字段是事实的来源。

### 显示昵称

当您希望 Codex 为生成的代理分配更可读的显示名称时，请使用 `nickname_candidates`。这在您运行许多相同自定义代理的实例并希望用户界面显示不同标签而不是重复相同代理名称时尤其有用。

昵称仅用于呈现。Codex 仍然通过其 `name` 来识别和生成代理。

昵称候选必须是唯一名称的非空列表。每个昵称可以使用 ASCII 字母、数字、空格、连字符和下划线。

示例：

```
name = "reviewer"
description = "聚焦于正确性、安全性以及缺失测试的 PR 审查者。"
developer_instructions = """
像拥有者一样审查代码。
优先考虑正确性、安全性、行为回归和缺失的测试覆盖。
"""
nickname_candidates = ["Atlas", "Delta", "Echo"]


```

在实践中，Codex 应用和 CLI 可以在代理活动出现的地方显示昵称，而底层代理类型仍然是 `reviewer`。

### 示例自定义代理

最佳自定义代理是专注和有特定目标的。为每个代理给出明确的工作，匹配该工作的工具界面，并在事情偏离时提供指令以保持在正确的轨道上。

#### 示例 1：PR 审查

此模式将审查分成三个专注的自定义代理：

- `pr_explorer` 绘制代码库并收集证据。
- `reviewer` 寻找正确性、安全性和测试风险。
- `docs_researcher` 通过专用 MCP 服务器检查框架或 API 文档。

项目配置 (`.codex/config.toml`):

```
[agents]
max_threads = 6
max_depth = 1


```

`.codex/agents/pr-explorer.toml`:

```
name = "pr_explorer"
description = "只读代码库探索器，用于在提出更改之前收集证据。"
model = "gpt-5.3-codex-spark"
model_reasoning_effort = "medium"
sandbox_mode = "read-only"
developer_instructions = """
保持在探索模式。
追踪实际执行路径，引用文件和符号，除非父代理请求，否则避免提出修复建议。
优先快速搜索和针对文件读取而非广泛扫描。
"""


```

`.codex/agents/reviewer.toml`:

```
name = "reviewer"
description = "聚焦于正确性、安全性以及缺失测试的 PR 审查者。"
model = "gpt-5.4"
model_reasoning_effort = "high"
sandbox_mode = "read-only"
developer_instructions = """
像拥有者一样审查代码。
优先考虑正确性、安全性、行为回归以及缺失的测试覆盖。
以具体发现为主，尽可能包括重现步骤，避免仅针对风格的评论，除非它们隐藏了真实的bug。
"""


```

`.codex/agents/docs-researcher.toml`:

```
name = "docs_researcher"
description = "文档专家，使用文档 MCP 服务器验证 API 和框架行为。"
model = "gpt-5.4-mini"
model_reasoning_effort = "medium"
sandbox_mode = "read-only"
developer_instructions = """
使用文档 MCP 服务器确认 API、选项和特定版本的行为。
返回简明答案，尽可能提供链接或确切引用。
不要进行代码更改。
"""

[mcp_servers.openaiDeveloperDocs]
url = "https://developers.openai.com/mcp"


```

此设置适用于以下提示：

```
对该分支进行审查，参考主分支。让 pr_explorer 绘制受影响的代码路径，reviewer 找出真实风险，docs_researcher 验证补丁所依赖的框架 API。


```

## 处理 CSV 批次与子代理（实验性）

此工作流是实验性的，可能会随着子代理支持的演变而变化。
当您有许多相似的任务映射到每个工作项的一行时，请使用 `spawn_agents_on_csv`。Codex 读取 CSV，为每一行生成一个工作子代理，等待整个批次完成，并将合并结果导出到 CSV。

此功能适用于重复审计，例如：

- 每行审查一个文件、包或服务
- 核查一系列事件、PR 或迁移目标
- 为许多相似的输入生成结构化总结

该工具接受：

- `csv_path` 源 CSV 的路径
- `instruction` 工作者的提示模板，使用 `{column_name}` 占位符
- `id_column` 当您希望从特定列获得稳定的项 ID 时
- `output_schema` 当每个工作代理应返回具有固定结构的 JSON 对象时
- `output_csv_path`、`max_concurrency` 和 `max_runtime_seconds` 用于作业控制

每个工作代理必须调用 `report_agent_job_result` 精确一次。如果工作代理在未报告结果的情况下退出，Codex 会在导出的 CSV 中将该行标记为错误。

示例提示：

```
创建 /tmp/components.csv，其中包含路径、拥有者字段，每个前端组件一行。

然后调用 spawn_agents_on_csv，参数如下：
- csv_path: /tmp/components.csv
- id_column: path
- instruction: "审查 {path}，由 {owner} 持有。返回 JSON，包含键 path、risk、summary 和 follow_up，通过 report_agent_job_result 进行报告。"
- output_csv_path: /tmp/components-review.csv
- output_schema: 包含必需字符串字段 path、risk、summary 和 follow_up 的对象


```

当您通过 `codex exec` 运行此内容时，Codex 会在 `stderr` 上显示单行进度更新，同时批次运行。导出的 CSV 包括原始行数据以及元数据，如 `job_id`、`item_id`、`status`、`last_error` 和 `result_json`。

相关运行时设置：

- `agents.max_threads` 限制并发可打开的代理线程数。
- `agents.job_max_runtime_seconds` 设置 CSV 发散作业的每个工作代理默认超时。每次调用的 `max_runtime_seconds` 覆盖优先。
- `sqlite_home` 控制 Codex 存储用于代理作业及其导出结果的 SQLite 支持状态的位置。

#### 示例 2：前端集成调试

此模式对于 UI 回归、偶发性浏览器流程或跨应用程序代码和正在运行的产品的集成错误尤其有用。

项目配置 (`.codex/config.toml`):

```
[agents]
max_threads = 6
max_depth = 1


```

`.codex/agents/code-mapper.toml`:

```
name = "code_mapper"
description = "只读代码库探索器，用于定位相关的前端和后端代码路径。"
model = "gpt-5.4-mini"
model_reasoning_effort = "medium"
sandbox_mode = "read-only"
developer_instructions = """
绘制失败的 UI 流程所属的代码。
识别入口点、状态转换和可能的文件，避免工作代理开始编辑之前。
"""


```

`.codex/agents/browser-debugger.toml`:

```
name = "browser_debugger"
description = "使用浏览器工具重现问题并捕获证据的 UI 调试器。"
model = "gpt-5.4"
model_reasoning_effort = "high"
sandbox_mode = "workspace-write"
developer_instructions = """
在浏览器中重现该问题，捕获确切步骤并报告 UI 实际操作。
使用浏览器工具进行截图、控制台输出和网络证据。
不要编辑应用代码。
"""

[mcp_servers.chrome_devtools]
url = "http://localhost:3000/mcp"
startup_timeout_sec = 20


```

`.codex/agents/ui-fixer.toml`:

```
name = "ui_fixer"
description = "在问题理解清楚后，专注于小型、针对性的修复的实现型代理。"
model = "gpt-5.3-codex-spark"
model_reasoning_effort = "medium"
