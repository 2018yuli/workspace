非交互模式允许您从脚本（例如，持续集成（CI）作业）运行 Codex，而无需打开交互式 TUI。
您可以通过 `codex exec` 进行调用。

有关标志级详细信息，请参见 [`codex exec`](https://developers.openai.com/codex/cli/reference#codex-exec)。

## 何时使用 `codex exec`

当您希望 Codex：

- 作为管道的一部分运行（CI、预合并检查、定时作业）。
- 生成可以传递给其他工具的输出（例如，生成发布说明或总结）。
- 自然地适应将命令输出链入 Codex 并将 Codex 输出传递给其他工具的 CLI 工作流。
- 使用明确的、预设的沙箱和批准设置运行。

## 基本用法

将任务提示作为单个参数传递：

```
codex exec "总结仓库结构并列出前 5 个风险区域"
```

在 `codex exec` 运行时，Codex 将进度流式传输到 `stderr`，并仅将最终代理消息打印到 `stdout`。这使得重定向或管道最后的结果变得简单：

```
codex exec "生成最近 10 次提交的发布说明" | tee release-notes.md
```

当您不希望会话回滚文件持久化到磁盘时，使用 `--ephemeral`：

```
codex exec --ephemeral "对这个仓库进行分类并建议后续步骤"
```

如果 `stdin` 通过管道传输，并且您还提供了提示参数，Codex 会将提示视为指令，并将管道内容作为附加上下文。

这使得用一个命令生成输入并直接交给 Codex 变得简单：

```
curl -s https://jsonplaceholder.typicode.com/comments \
  | codex exec "将前 20 项格式化为 markdown 表格" \
  > table.md
```

有关更高级的 `stdin` 管道模式，请参见 [高级 stdin 管道](https://developers.openai.com/codex/noninteractive#advanced-stdin-piping)。

## 权限和安全

默认情况下，`codex exec` 在只读沙箱中运行。在自动化中，为工作流设置最低所需权限：

- 允许编辑：`codex exec --full-auto "<task>"`
- 允许更广泛的访问：`codex exec --sandbox danger-full-access "<task>"`

仅在受控环境中使用 `danger-full-access`（例如，隔离的 CI 运行器或容器）。

如果您配置了启用的 MCP 服务器且 `required = true`，而它无法初始化，`codex exec` 将以错误退出，而不是在没有该服务器的情况下继续。

## 使输出机器可读

要在脚本中使用 Codex 输出，使用 JSON Lines 输出：

```
codex exec --json "总结仓库结构" | jq
```

启用 `--json` 时，`stdout` 变为 JSON Lines（JSONL）流，因此您可以捕获 Codex 在运行时发出的每个事件。事件类型包括 `thread.started`、`turn.started`、`turn.completed`、`turn.failed`、`item.*` 和 `error`。

项目类型包括代理消息、推理、命令执行、文件更改、MCP 工具调用、网络搜索和计划更新。

示例 JSON 流（每一行都是一个 JSON 对象）：

```
{"type":"thread.started","thread_id":"0199a213-81c0-7800-8aa1-bbab2a035a53"}
{"type":"turn.started"}
{"type":"item.started","item":{"id":"item_1","type":"command_execution","command":"bash -lc ls","status":"in_progress"}}
{"type":"item.completed","item":{"id":"item_3","type":"agent_message","text":"仓库包含 docs、sdk 和 examples 目录。"}}
{"type":"turn.completed","usage":{"input_tokens":24763,"cached_input_tokens":24448,"output_tokens":122}}
```

如果您只需要最终消息，可以将其写入文件 `-o <path>`/`--output-last-message <path>`。这会将最终消息写入文件并仍然将其打印到 `stdout`（有关详细信息，请参见 [`codex exec`](https://developers.openai.com/codex/cli/reference#codex-exec)）。

## 使用模式创建结构化输出

如果您需要用于下游步骤的结构化数据，请使用 `--output-schema` 请求符合 JSON Schema 的最终响应。这对于需要稳定字段的自动化工作流很有用（例如，作业摘要、风险报告或发布元数据）。

`schema.json`

```
{
  "type": "object",
  "properties": {
    "project_name": { "type": "string" },
    "programming_languages": {
      "type": "array",
      "items": { "type": "string" }
    }
  },
  "required": ["project_name", "programming_languages"],
  "additionalProperties": false
}
```

运行 Codex 并将最终 JSON 响应写入磁盘：

```
codex exec "提取项目元数据" \
  --output-schema ./schema.json \
  -o ./project-metadata.json
```

示例最终输出（stdout）：

```
{
  "project_name": "Codex CLI",
  "programming_languages": ["Rust", "TypeScript", "Shell"]
}
```

## 在 CI 中认证

默认情况下，`codex exec` 重用保存的 CLI 认证。在 CI 中，常见的做法是明确提供凭据：

### 使用 API 密钥认证（推荐）

- 将 `CODEX_API_KEY` 作为作业的密钥环境变量设定。
- 记住提示和工具输出：它们可能包含敏感代码或数据。

要在单次运行中使用不同的 API 密钥，请按如下方式设置 `CODEX_API_KEY`：

```
CODEX_API_KEY=<api-key> codex exec --json "分类打开的错误报告"
```

`CODEX_API_KEY` 仅支持在 `codex exec` 中。

在 CI/CD 中使用 ChatGPT 管理的认证（高级）

如果您需要使用 Codex 用户帐户而不是 API 密钥运行 CI/CD 作业（例如，使用 ChatGPT 管理的 Codex 访问的企业团队或需要 ChatGPT/Codex 速率限制而不是 API 密钥使用的用户），请阅读以下内容。

API 密钥是自动化的正确默认值，因为它们更容易提供和轮换。仅在您需要作为 Codex 帐户运行时，才使用此路径。

将 `~/.codex/auth.json` 视为密码：它包含访问令牌。不要提交、粘贴到工单或分享。

请勿将此工作流程用于公共或开源仓库。如果运行器上没有 `codex login`，请通过安全存储播种 `auth.json`，在运行器上运行 Codex，以便 Codex 在原地刷新，并在运行之间持久化已更新的文件。

请参见 [在 CI/CD 中维护 Codex 账户认证（高级）](https://developers.openai.com/codex/auth/ci-cd-auth)。

## 恢复非交互会话

如果您需要继续先前的运行（例如，双阶段管道），请使用 `resume` 子命令：

```
codex exec "审核更改以检查竞争条件"
codex exec resume --last "修复您发现的竞争条件"
```

您还可以通过 `codex exec resume <SESSION_ID>` 针对特定会话 ID。

## 需要 Git 仓库

Codex 要求在 Git 仓库内运行命令，以防止破坏性更改。如果您确定环境是安全的，可以使用 `codex exec --skip-git-repo-check` 覆盖此检查。

## 常见自动化模式

### 示例：在 GitHub Actions 中自动修复 CI 失败

您可以使用 `codex exec` 在 CI 工作流失败时自动提议修复。典型模式是：

1. 当主 CI 工作流完成并出错时，触发后续工作流。
2. 检出失败的提交 SHA。
3. 安装依赖项，并使用狭窄的提示和最低权限运行 Codex。
4. 重新运行测试命令。
5. 打开一个包含结果补丁的拉取请求。

#### 使用 Codex CLI 的最小工作流程

下面的示例展示了核心步骤。根据您的堆栈调整安装和测试命令。

```
name: Codex auto-fix on CI failure

on:
  workflow_run:
    workflows: ["CI"]
    types: [completed]

permissions:
  contents: write
  pull-requests: write

jobs:
  auto-fix:
    if: ${{ github.event.workflow_run.conclusion == 'failure' }}
    runs-on: ubuntu-latest
    env:
      OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
      FAILED_HEAD_SHA: ${{ github.event.workflow_run.head_sha }}
      FAILED_HEAD_BRANCH: ${{ github.event.workflow_run.head_branch }}
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ env.FAILED_HEAD_SHA }}
          fetch-depth: 0

      - uses: actions/setup-node@v4
        with:
          node-version: "20"

      - name: 安装依赖
        run: |
          if [ -f package-lock.json ]; then npm ci; else npm i; fi

      - name: 安装 Codex
        run: npm i -g @openai/codex

      - name: 认证 Codex
        run: codex login --api-key "$OPENAI_API_KEY"

      - name: 运行 Codex
        run: |
          codex exec --full-auto --sandbox workspace-write \
            "读取该仓库，运行测试套件，识别使所有测试通过所需的最小变更，实施仅该变更，并停止。不要重构无关文件。"

      - name: 验证测试
        run: npm test --silent

      - name: 创建拉取请求
        if: success()
        uses: peter-evans/create-pull-request@v6
        with:
          branch: codex/auto-fix-${{ github.event.workflow_run.run_id }}
          base: ${{ env.FAILED_HEAD_BRANCH }}
          title: "通过 Codex 自动修复失败的 CI"
```

#### 替代：使用 Codex GitHub Action

如果您想避免自己安装 CLI，可以通过 [Codex GitHub Action](https://developers.openai.com/codex/github-action) 运行 `codex exec` 并将提示作为输入传递。

## 高级 stdin 管道

当另一个命令为 Codex 生成输入时，根据指令应该来自哪里选择 stdin 模式。当您已经知道指令并希望将管道输出作为上下文传递时，请使用 prompt-plus-stdin。当 stdin 应该成为完整提示时，请使用 `codex exec -`。

### 使用 prompt-plus-stdin

当另一个命令已经生成您要 Codex 检查的数据时，prompt-plus-stdin 是有用的。在此模式下，您自己编写指令并将输出作为上下文传递，这使得它非常适合围绕命令输出、日志和生成数据构建的 CLI 工作流。

```
npm test 2>&1 \
  | codex exec "总结失败的测试并提出最小可能的修复" \
  | tee test-summary.md
```

### 总结日志

```
tail -n 200 app.log \

```
curl -vv https://api.example.com/health 2>&1 \
  | codex exec "解释 TLS 或 HTTP 失败并建议最可能的修复" \
  > tls-debug.md
```

```
gh run view 123456 --log \
  | codex exec "撰写关于 CI 失败的简明 Slack 准备更新，包括可能的原因和下一步" \
  | pbcopy
```

### 从 CI 日志中起草拉取请求评论

```
gh run view 123456 --log \
  | codex exec "用 5 个要点总结拉取请求线程中的失败" \
  | gh pr comment 789 --body-file -
```

### 当 stdin 是提示时使用 `codex exec -`

如果您省略提示参数，Codex 将从 stdin 中读取提示。明确使用 `codex exec -` 当您希望强制该行为时。

`-` 哨兵在另一个命令或脚本动态生成完整提示时非常有用。这适合将提示存储在文件中、使用 shell 脚本组装提示，或在将整个提示交给 Codex 之前，将实时命令输出与指令结合起来。

```
cat prompt.txt | codex exec -
```

```
printf "用 3 个要点总结此错误日志：\n\n%s\n" "$(tail -n 200 app.log)" \
  | codex exec -
```

```
generate_prompt.sh | codex exec - --json > result.jsonl
