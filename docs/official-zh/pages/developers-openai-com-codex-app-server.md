Codex 应用服务器是 Codex 用于支持丰富客户端的接口（例如，Codex VS Code 扩展）。当您希望将 Codex 深入集成到自己的产品中时，请使用它：身份验证、对话历史、审批和流式代理事件。应用服务器实现是开源的，可以在 Codex GitHub 仓库中找到（ [openai/codex/codex-rs/app-server](https://github.com/openai/codex/tree/main/codex-rs/app-server)）。有关开源 Codex 组件的完整列表，请参见 [开源](https://developers.openai.com/codex/open-source) 页面。

如果您正在自动化工作或在 CI 中运行 Codex，请使用
[Codex SDK](https://developers.openai.com/codex/sdk)。

## 协议

如同 [MCP](https://modelcontextprotocol.io/)，`codex app-server` 支持使用 JSON-RPC 2.0 消息进行双向通信（在传输中省略了 `"jsonrpc":"2.0"` 头）。

支持的传输：

- `stdio` (`--listen stdio://`, 默认)：以换行分隔的 JSON（JSONL）。
- `websocket` (`--listen ws://IP:PORT`, 实验性且不支持)：每个 WebSocket 文本帧一个 JSON-RPC 消息。
- `off` (`--listen off`)：不暴露本地传输。

当您使用 `--listen ws://IP:PORT` 启动时，同一个侦听器还会提供基本的 HTTP 健康探针：

- `GET /readyz` 返回 `200 OK` 一旦侦听器接受新连接。
- `GET /healthz` 返回 `200 OK` 当请求不包含 `Origin` 头时。
- 带有 `Origin` 头的请求将被拒绝，返回 `403 Forbidden`。

WebSocket 传输是实验性且不支持的。回环侦听器，如 `ws://127.0.0.1:PORT` 适用于本地主机和 SSH 端口转发工作流。非回环 WebSocket 侦听器在推出期间允许未经身份验证的连接，因此在远程暴露之前，请先配置 WebSocket 身份验证。

支持的 WebSocket 身份验证标志：

- `--ws-auth capability-token --ws-token-file /absolute/path`
- `--ws-auth capability-token --ws-token-sha256 HEX`
- `--ws-auth signed-bearer-token --ws-shared-secret-file /absolute/path`

对于签名的承保人令牌，您还可以设置 `--ws-issuer`，`--ws-audience` 和 `--ws-max-clock-skew-seconds`。客户端在 WebSocket 握手期间以 `Authorization: Bearer <token>` 呈现凭据，且应用服务器在 JSON-RPC `initialize` 之前执行身份验证。

优先使用 `--ws-token-file` 而不是在命令行中传递原始的承保人令牌。仅当客户端将原始高熵令牌保存在单独的本地秘密存储中时，才使用 `--ws-token-sha256`；该哈希仅是验证器，客户端仍需原始令牌。

在 WebSocket 模式下，应用服务器使用有界队列。当请求流入已满时，服务器将以 JSON-RPC 错误代码 `-32001` 和消息 `"服务器超载；请稍后重试"` 拒绝新请求。客户端应以指数递增的延迟和抖动重试。

## 消息模式

请求包括 `method`，`params` 和 `id`：

```
{ "method": "thread/start", "id": 10, "params": { "model": "gpt-5.4" } }
```

响应会以 `id` 附 Echo，带有 `result` 或 `error`：

```
{ "id": 10, "result": { "thread": { "id": "thr_123" } } }
```

```
{ "id": 10, "error": { "code": 123, "message": "出错了" } }
```

通知省略 `id`，仅使用 `method` 和 `params`：

```
{ "method": "turn/started", "params": { "turn": { "id": "turn_456" } } }
```

您可以从 CLI 生成 TypeScript 模式或 JSON 模式包。每个输出特定于您运行的 Codex 版本，因此生成的工件与该版本完全匹配：

```
codex app-server generate-ts --out ./schemas
codex app-server generate-json-schema --out ./schemas
```

## 入门

1. 使用 `codex app-server` 启动服务器（默认 stdio 传输）或使用 `codex app-server --listen ws://127.0.0.1:4500`（实验性 WebSocket 传输）。
2. 通过所选传输连接客户端，然后发送 `initialize`，接着是 `initialized` 通知。
3. 启动线程和回合，然后持续读取活动传输流中的通知。

示例（Node.js / TypeScript）：

```
import { spawn } from "node:child_process";
import readline from "node:readline";

const proc = spawn("codex", ["app-server"], {
  stdio: ["pipe", "pipe", "inherit"],
});
const rl = readline.createInterface({ input: proc.stdout });

const send = (message: unknown) => {
  proc.stdin.write(`${JSON.stringify(message)}\n`);
};

let threadId: string | null = null;

rl.on("line", (line) => {
  const msg = JSON.parse(line) as any;
  console.log("服务器:", msg);

  if (msg.id === 1 && msg.result?.thread?.id && !threadId) {
    threadId = msg.result.thread.id;
    send({
      method: "turn/start",
      id: 2,
      params: {
        threadId,
        input: [{ type: "text", text: "总结这个仓库。" }],
      },
    });
  }
});

send({
  method: "initialize",
  id: 0,
  params: {
    clientInfo: {
      name: "my_product",
      title: "我的产品",
      version: "0.1.0",
    },
  },
});
send({ method: "initialized", params: {} });
send({ method: "thread/start", id: 1, params: { model: "gpt-5.4" } });
```

## 核心原语

- **线程**：用户和 Codex 代理之间的对话。线程包含回合。
- **回合**：单个用户请求及随之而来的代理工作。回合包含项目并流式更新。
- **项目**：输入或输出的单位（用户消息、代理消息、命令运行、文件更改、工具调用等）。

使用线程 API 创建、列出或归档对话。使用回合 API 驱动对话，并通过回合通知流进度。

## 生命周期概述

- **每个连接初始化一次**：在打开传输连接后立即发送包含客户端元数据的 `initialize` 请求，然后发布 `initialized`。在此握手之前，服务器将拒绝该连接上的任何请求。
- **开始（或恢复）线程**：调用 `thread/start` 创建新对话，调用 `thread/resume` 继续已有对话，或调用 `thread/fork` 将历史分支到新线程 ID。
- **开始回合**：调用 `turn/start`，传入目标 `threadId` 和用户输入。可选字段可以重写模型、个性、`cwd`、沙盒策略等。
- **引导活动回合**：调用 `turn/steer`，在不创建新回合的情况下将用户输入附加到当前进行中的回合。
- **流式事件**：在 `turn/start` 之后，继续在标准输出上读取通知：`thread/archived`、`thread/unarchived`、`item/started`、`item/completed`、`item/agentMessage/delta`、工具进度和其他更新。
- **完成回合**：当模型完成或在 `turn/interrupt` 取消后，服务器发布 `turn/completed`，并附带最终状态。

## 初始化

客户端必须在调用该连接上的任何其他方法之前，发送一次 `initialize` 请求，然后确认 `initialized` 通知。初始化之前发送的请求将收到 `Not initialized` 错误，而在同一连接上重复调用 `initialize` 将返回 `Already initialized`。

服务器返回用户代理字符串，它将呈现给上游服务，以及描述运行时目标的 `platformFamily` 和 `platformOs` 值。设置 `clientInfo` 以标识您的集成。

`initialize.params.capabilities` 也支持每个连接的通知选择退出，通过 `optOutNotificationMethods`，这是该连接中要抑制的确切方法名列表。匹配方式精确（不支持通配符/前缀）。未知的方法名被接受并被忽略。

**重要**：使用 `clientInfo.name` 来标识您的客户端，以便于 OpenAI 合规日志平台。如果您正在为企业使用开发新的 Codex 集成，请联系 OpenAI 以将其添加到已知客户端列表中。有关更多上下文，请参见 [Codex 日志参考](https://chatgpt.com/admin/api-reference#tag/Logs:-Codex)。

示例（来自 Codex VS Code 扩展）：

```
{
  "method": "initialize",
  "id": 0,
  "params": {
    "clientInfo": {
      "name": "codex_vscode",
      "title": "Codex VS Code 扩展",
      "version": "0.1.0"
    }
  }
}
```

带有通知选择退出的示例：

```
{
  "method": "initialize",
  "id": 1,
  "params": {
    "clientInfo": {
      "name": "my_client",
      "title": "我的客户端",
      "version": "0.1.0"
    },
    "capabilities": {
      "experimentalApi": true,
      "optOutNotificationMethods": ["thread/started", "item/agentMessage/delta"]
    }
  }
}
```

## 实验性 API 选择加入

某些应用服务器方法和字段有意被限制在某些 `experimentalApi` 能力后面。

- 要保持在稳定的 API 表面，省略 `capabilities`（或将 `experimentalApi` 设置为 `false`），服务器将拒绝实验性方法/字段。
- 将 `capabilities.experimentalApi` 设置为 `true` 以启用实验性方法和字段。

```
{
  "method": "initialize",
  "id": 1,
  "params": {
    "clientInfo": {
      "name": "my_client",
      "title": "我的客户端",
      "version": "0.1.0"
    },
    "capabilities": {
      "experimentalApi": true
    }
  }
}
```

如果客户端在没有选择加入的情况下发送实验性方法或字段，应用服务器将拒绝它：

`<descriptor> requires experimentalApi capability`

## API 概述

- `thread/start` \- 创建一个新线程；发布 `thread/started` 并自动订阅您到该线程的回合/项目事件。
- `thread/resume` \- 通过 ID 重新打开一个现有线程，以便后续 `turn/start` 调用附加到它。
- `thread/fork` \- 通过复制存储的历史，将线程分叉到一个新线程 ID；发布新线程的 `thread/started`。
- `thread/read` \- 按 ID 读取存储的线程而不恢复它；设置 `includeTurns` 以返回完整的回合历史。返回的 `thread` 对象包括运行时 `status`。
- `thread/list` \- 分页通过存储的线程日志；支持基于光标的分页以及 `modelProviders`、`sourceKinds`、`archived`、`cwd` 和 `searchTerm` 过滤器。返回的 `thread` 对象包括运行时 `status`。
- `thread/turns/list` \- 分页通过存储线程的回合历史而不恢复它。
- `thread/loaded/list` \- 列出当前加载在内存中的线程 ID。
- `thread/name/set` \- 设置或更新加载线程或持久化发布的线的用户面名称；发布 `thread/name/updated`。
- `thread/metadata/update` \- patch SQLite 支持的存储线程元数据；当前支持持久 `gitInfo`。
- `thread/archive` \- 将线程的日志文件移动到档案目录；成功时返回 `{}`，并发布 `thread/archived`。
- `thread/unsubscribe` \- 取消该连接对线程回合/项目事件的订阅。如果这是最后一个订阅者，服务器在没有订阅者的非活动宽限期之后卸载线程，并发布 `thread/closed`。
- `thread/unarchive` \- 将归档的线程发布恢复回活动会话目录；返回恢复的 `thread` 并发布 `thread/unarchived`。
- `thread/status/changed` \- 当加载的线程的运行时 `status` 变化时发布的通知。
- `thread/compact/start` \- 为线程触发对话历史压缩；立即返回 `{}`，同时通过 `turn/*` 和 `item/*` 通知流进行进度。
- `thread/shellCommand` \- 在线程上运行用户启动的 shell 命令。此命令在沙盒之外运行，具有完全访问权限，并且不继承线程沙盒策略。
- `thread/backgroundTerminals/clean` \- 停止线程的所有正在运行的后台终端（实验性；需要 `capabilities.experimentalApi`）。
- `thread/rollback` \- 删除内存上下文中的最后 N 个回合并持久化回滚标记；返回更新的 `thread`。
- `turn/start` \- 将用户输入添加到线程并开始 Codex 生成；响应初始 `turn` 并流式事件。对于 `collaborationMode`，`settings.developer_instructions: null` 意味着“使用所选模式的内置指令”。
- `thread/inject_items` \- 将原始响应 API 项目附加到已加载线程的模型可见历史中，而无需开始用户回合。
- `turn/steer` \- 将用户输入追加到线程的活动进行中的回合；返回已接受的 `turnId`。
- `turn/interrupt` \- 请求取消进行中的回合；成功返回 `{}`，回合以 `status: "interrupted"` 结束。
- `review/start` \- 启动线程的 Codex 审核者；发布 `enteredReviewMode` 和 `exitedReviewMode` 项。
- `command/exec` \- 在服务器沙盒下运行单个命令，而无需启动线程/回合。
- `command/exec/write` \- 将 `stdin` 字节写入正在运行的 `command/exec` 会话或关闭 `stdin`。
- `command/exec/resize` \- 调整运行 PTY 支持的 `command/exec` 会话的大小。
- `command/exec/terminate` \- 终止正在运行的 `command/exec` 会话。
- `command/exec/outputDelta`（通知）- 从流式 `command/exec` 会话中发出 base64 编码的 stdout/stderr 块。
- `model/list` \- 列出可用模型（设置 `includeHidden: true` 以包括带有 `hidden: true` 的条目），并带有努力选项、可选 `upgrade` 和 `inputModalities`。
- `experimentalFeature/list` \- 列出带有生命周期阶段元数据和光标分页的特性标志。
- `experimentalFeature/enablement/set` \- 堆积内存运行时支持针对的特征关键的启用。
- `collaborationMode/list` \- 列出协作模式预设（实验性，无分页）。
- `skills/list` \- 列出一个或多个 `cwd` 值的技能（支持 `forceReload` 和可选 `perCwdExtraUserRoots`）。
- `skills/changed`（通知）- 当被观察到的本地技能文件更改时发布。
- `marketplace/add` \- 添加远程插件市场并将其持久化到用户的市场配置中。
- `plugin/list` \- 列出发现的插件市场和插件状态，包括安装/身份验证策略元数据、市场加载错误、特色插件 ID，以及本地、Git 或远程插件源元数据。
- `plugin/read` \- 通过市场路径或远程市场名称和插件名称读取一个插件，包括在那些细节可用时捆绑的技能、应用和 MCP 服务器名称。
- `plugin/install` \- 从市场路径或远程市场名称安装一个插件。
- `plugin/uninstall` \- 卸载已安装的插件。
- `app/list` \- 列出可用的应用（连接）的列表，并支持分页加上可访问性/启用元数据。
- `skills/config/write` \- 按路径启用或禁用技能。
- `mcpServer/oauth/login` \- 开始为配置的 MCP 服务器进行 OAuth 登录；返回授权 URL，并在完成时发布 `mcpServer/oauthLogin/completed`。
- `tool/requestUserInput` \- 在工具调用中提示用户 1-3 个简短问题（实验性）；问题可以设置 `isOther` 作为自由选项。
- `config/mcpServer/reload` \- 从磁盘重新加载 MCP 服务器配置，并为加载的线程排队刷新。
- `mcpServerStatus/list` \- 列出 MCP 服务器、工具、资源和身份验证状态（光标 + 限制分页）。使用 `detail: "full"` 以获得完整数据，或使用 `detail: "toolsAndAuthOnly"` 省略资源。
- `mcpServer/resource/read` \- 通过初始化的 MCP 服务器读取单个 MCP 资源。
- `mcpServer/tool/call` \- 在线程配置的 MCP 服务器上调用一个工具。
- `mcpServer/startupStatus/updated`（通知）- 当配置的 MCP 服务器的启动状态发生变化时发布。
- `windowsSandbox/setupStart` \- 开始 Windows 沙盒设置以进行 `elevated` 或 `unelevated` 模式；快速返回，随后发布 `windowsSandbox/setupCompleted`。
- `feedback/upload` \- 提交反馈报告（分类 + 可选原因/日志 + 对话 ID，以及可选的 `extraLogFiles` 附件）。
- `config/read` \- 在解决配置分层后从磁盘获取有效配置。
- `externalAgentConfig/detect` \- 检测可以迁移的外部代理文物，可以通过 `includeHome` 和可选的 `cwds` 进行迁移；每个检测到的项目包括 `cwd`（`null` 代表主目录）。
- `externalAgentConfig/import` \- 通过传递显式 `migrationItems` 和 `cwd`（`null` 代表主目录）应用所选的外部代理迁移项目；插件导入会在 `externalAgentConfig/import/completed` 时发布。
- `config/value/write` \- 将单个配置键/值写入用户的 `config.toml`。
- `config/batchWrite` \- 原子性地将配置编辑应用于用户的 `config.toml`。
- `configRequirements/read` \- 从 `requirements.toml` 和/或 MDM 获取要求，包括允许列表、固定的 `featureRequirements` 和居住/网络要求（如果您没有设置任何，则为 `null`）。
- `fs/readFile`、`fs/writeFile`、`fs/createDirectory`、`fs/getMetadata`、`fs/readDirectory`、`fs/remove`、`fs/copy`、`fs/watch`、`fs/unwatch` 和 `fs/changed`（通知） - 通过应用服务器 v2 文件系统 API 在绝对文件路径上操作。

插件摘要包括 `source` 联合。本地插件返回
`{ "type": "local", "path": ... }`，基于 Git 的市场条目返回
`{ "type": "git", "url": ..., "path": ..., "refName": ..., "sha": ... }`，
而远程目录条目返回 `{ "type": "remote" }`。对于仅限远程的
目录项，`PluginMarketplaceEntry.path` 可以为 `null`；读取或安装这些插件时，请传递 `remoteMarketplaceName`，而不是 `marketplacePath`。

## 模型

### 列出模型 (`model/list`)

调用 `model/list` 以发现可用模型及其能力，然后再呈现模型或个性选择器。

```
{ "method": "model/list", "id": 6, "params": { "limit": 20, "includeHidden": false } }
{ "id": 6, "result": {
  "data": [{\
    "id": "gpt-5.4",\
    "model": "gpt-5.4",\
    "displayName": "GPT-5.4",\
    "hidden": false,\
    "defaultReasoningEffort": "medium",\
    "supportedReasoningEfforts": [{\
      "reasoningEffort": "low",\
      "description": "降低延迟"\
    }],\
    "inputModalities": ["text", "image"],\
    "supportsPersonality": true,\
    "isDefault": true\
  }],
  "nextCursor": null
} }
```

每个模型条目可以包括：

- `supportedReasoningEfforts` \- 该模型支持的努力选项。
- `defaultReasoningEffort` \- 建议的默认努力供客户端使用。
- `upgrade` \- 用于迁移提示的可选推荐升级模型 ID。
- `upgradeInfo` \- 用于迁移提示的可选升级元数据。
- `hidden` \- 模型是否对默认选择器列表隐藏。
- `inputModalities` \- 该模型支持的输入类型（例如 `text`、`image`）。
- `supportsPersonality` \- 模型是否支持个性特定的指令，例如 `/personality`。
- `isDefault` \- 模型是否为推荐默认。

默认情况下，`model/list` 仅返回选择器可见的模型。设置 `includeHidden: true` 如果您需要完整列表并希望使用 `hidden` 进行客户端过滤。

当 `inputModalities` 缺失（较旧的模型目录）时，出于向后兼容性考虑，将其视为 `["text", "image"]`。

### 列出实验性特性 (`experimentalFeature/list`)

使用此端点以发现带有元数据和生命周期阶段的特性标志：

```
{ "method": "experimentalFeature/list", "id": 7, "params": { "limit": 20 } }
{ "id": 7, "result": {
  "data": [{\
    "name": "unified_exec",\
    "stage": "beta",\
    "displayName": "统一执行",\
    "description": "使用统一的 PTY 支持的执行工具。",\
    "announcement": "提高命令执行可靠性的实验性推出。",\
    "enabled": false,\
    "defaultEnabled": false\
  }],
  "nextCursor": null
} }
```

`stage` 可以是 `beta`、`underDevelopment`、`stable`、`deprecated` 或 `removed`。对于非 beta 标志，`displayName`、`description` 和 `announcement` 可能为 `null`。

## 线程

- `thread/read` 读取存储的线程而不对其进行订阅；设置 `includeTurns` 以包含回合。
- `thread/turns/list` 分页通过存储线程的回合历史而不恢复它。
- `thread/list` 支持光标分页以及 `modelProviders`、`sourceKinds`、`archived`、`cwd` 和 `searchTerm` 过滤。
- `thread/loaded/list` 返回当前加载在内存中的线程 ID。
- `thread/archive` 将线程的持久化 JSONL 日志移动到归档目录。
- `thread/metadata/update` 修补已存储线程元数据，当前包括持久化的 `gitInfo`。
- `thread/unsubscribe` 取消当前连接对已加载线程的订阅，并可以在非活动宽限期后触发 `thread/closed`。
- `thread/unarchive` 恢复已归档线程发布回活动会话目录。
- `thread/compact/start` 触发压缩并立即返回 `{}`。
- `thread/rollback` 删除内存上下文中的最后 N 个回合并在线程的持久化 JSONL 日志中记录回滚标记。
- `turn/start` \- 将用户输入添加到线程并开始 Codex 生成；响应初始 `turn` 并流式事件。对于 `collaborationMode`，`settings.developer_instructions: null` 代码意思是“使用所选模式的内置指导”。
- `thread/inject_items` 将原始响应 API 项目附加到已加载线程的模型可见历史中而不开始用户回合。

### 启动或恢复线程

当您需要新的 Codex 对话时，启动一个新线程。

```
{ "method": "thread/start", "id": 10, "params": {
  "model": "gpt-5.4",
  "cwd": "/Users/me/project",
  "approvalPolicy": "never",
  "sandbox": "workspaceWrite",
  "personality": "friendly",
  "serviceName": "my_app_server_client"
} }
{ "id": 10, "result": {
  "thread": {
    "id": "thr_123",
    "preview": "",
    "ephemeral": false,
    "modelProvider": "openai",
    "createdAt": 1730910000
  }
} }
{ "method": "thread/started", "params": { "thread": { "id": "thr_123" } } }
```

`serviceName` 是可选的。当您希望应用服务器通过您集成的服务名称标记线程级指标时，请设置它。

要继续存储的会话，请调用 `thread/resume`，并使用您之前记录的 `thread.id`。响应形状与 `thread/start` 匹配。您还可以传递与 `thread/start` 支持的相同配置覆盖，例如 `personality`：

```
{ "method": "thread/resume", "id": 11, "params": {
  "threadId": "thr_123",
  "personality": "friendly"
} }
{ "id": 11, "result": { "thread": { "id": "thr_123", "name": "Bug bash notes", "ephemeral": false } } }
```

恢复线程不会自动更新 `thread.updatedAt`（或发布文件的修改时间）。时间戳在您开始回合时更新。

如果您在配置中将启用的 MCP 服务器标记为 `required`，并且该服务器未能初始化，则 `thread/start` 和 `thread/resume` 将失败，而不是在没有它的情况下继续。

在 `thread/start` 上的 `dynamicTools` 是一个实验性字段（需要 `capabilities.experimentalApi = true`）。Codex 将这些动态工具持久化在线程发布元数据中，并在您未提供新动态工具时在 `thread/resume` 时恢复它们。

如果您使用与发布中记录的模型不同的模型进行恢复，Codex 将发送警告，并在下一个回合中应用一次性模型切换指令。

要从存储的会话分支，请调用 `thread/fork`，并传入 `thread.id`。这将创建一个新线程 ID，并发布 `thread/started` 通知：

```
{ "method": "thread/fork", "id": 12, "params": { "threadId": "thr_123" } }
{ "id": 12, "result": { "thread": { "id": "thr_456" } } }
{ "method": "thread/started", "params": { "thread": { "id": "thr_456" } } }
```

当设置了用户面线程标题时，应用服务器在 `thread/list`、`thread/read`、`thread/resume`、`thread/unarchive` 和 `thread/rollback` 响应中为 `thread.name` 注入。直到稍后设置标题为止，`thread/start` 和 `thread/fork` 可能会省略 `name`（或返回 `null`）。

### 读取存储的线程（不恢复）

当您想要存储的线程数据但不想恢复线程或订阅其事件时，请使用 `thread/read`。

- `includeTurns` \- 当 `true` 时，响应包括线程的回合；当为 `false` 或省略时，仅获得线程摘要。
- 返回的 `thread` 对象包括运行时 `status`（`notLoaded`、`idle`、`systemError` 或 `active`，带有 `activeFlags`）。

```
{ "method": "thread/read", "id": 19, "params": { "threadId": "thr_123", "includeTurns": true } }
{ "id": 19, "result": { "thread": { "id": "thr_123", "name": "Bug bash notes", "ephemeral": false, "status": { "type": "notLoaded" }, "turns": [] } } }
```

与 `thread/resume` 不同，`thread/read` 不会将线程加载到内存中，也不会发布 `thread/started`。

### 列出线程回合

使用 `thread/turns/list` 分页存储线程的回合历史而不恢复它。结果默认是最新优先，因此客户端可以用 `nextCursor` 获取较旧的回合。响应也包括 `backwardsCursor`；将其作为 `cursor` 传递，并使用 `sortDirection: "asc"` 来获取比先前页面的第一个项目更新的回合。

```
{ "method": "thread/turns/list", "id": 20, "params": {
  "threadId": "thr_123",
  "limit": 50,
  "sortDirection": "desc"
} }
{ "id": 20, "result": {
  "data": [],
  "nextCursor": "older-turns-cursor-or-null",
  "backwardsCursor": "newer-turns-cursor-or-null"
} }
```

### 列出线程（带分页和过滤）

`thread/list` 允许您呈现历史 UI。结果默认是按 `createdAt` 最新优先。过滤器在分页之前应用。传递任意组合的：

- `cursor` \- 来自先前响应的不可见字符串；第一次页面省略。
- `limit` \- 如果未设置，服务器默认为合理的页面大小。
- `sortKey` \- `created_at`（默认）或 `updated_at`。
- `modelProviders` \- 限制结果为特定提供程序；未设置、为空或空数组包括所有提供程序。
- `sourceKinds` \- 限制结果为特定线程来源。省略或 `[]` 时，服务器默认为仅交互来源：`cli` 和 `vscode`。
- `archived` \- 当为 `true` 时，仅列出归档线程。为 `false` 或省略时，列出非归档线程（默认）。
- `cwd` \- 限制结果为当前工作目录与此路径完全匹配的线程。
- `searchTerm` \- 在分页之前搜索存储线程摘要和元数据。

`sourceKinds` 接受以下值：

- `cli`
- `vscode`
- `exec`
- `appServer`
- `subAgent`
- `subAgentReview`
- `subAgentCompact`
- `subAgentThreadSpawn`
- `subAgentOther`
- `unknown`

示例：

```
{ "method": "thread/list", "id": 20, "params": {
  "cursor": null,
  "limit": 25,
  "sortKey": "created_at"
} }
{ "id": 20, "result": {
  "data": [\
    { "id": "thr_a", "preview": "创建一个 TUI", "ephemeral": false, "modelProvider": "openai", "createdAt": 1730831111, "updatedAt": 1730831111, "name": "TUI 原型", "status": { "type": "notLoaded" } },\
    { "id": "thr_b", "preview": "修复测试", "ephemeral": true, "modelProvider": "openai", "createdAt": 1730750000, "updatedAt": 1730750000, "status": { "type": "notLoaded" } }\
  ],
  "nextCursor": "opaque-token-or-null"
} }
```

当 `nextCursor` 为 `null` 时，您已到达最后一页。

### 更新存储线程元数据

使用 `thread/metadata/update` 修补存储线程元数据，而不恢复线程。今天支持持久 `gitInfo`；省略的字段保持不变，显式 `null` 将清除存储的值。

```
{ "method": "thread/metadata/update", "id": 21, "params": {
  "threadId": "thr_123",
  "gitInfo": { "branch": "feature/sidebar-pr" }
} }
{ "id": 21, "result": {
  "thread": {
    "id": "thr_123",
    "gitInfo": { "sha": null, "branch": "feature/sidebar-pr", "originUrl": null }
  }
} }
```

### 跟踪线程状态变化

当加载的线程的运行时状态变化时，`thread/status/changed` 将被发布。负载包含 `threadId` 和新状态。

```
{
  "method": "thread/status/changed",
  "params": {
    "threadId": "thr_123",
    "status": { "type": "active", "activeFlags": ["waitingOnApproval"] }
  }
}
```

### 列出加载的线程

`thread/loaded/list` 返回当前加载在内存中的线程 ID。

```
{ "method": "thread/loaded/list", "id": 21 }
{ "id": 21, "result": { "data": ["thr_123", "thr_456"] } }
```

### 取消订阅已加载线程

`thread/unsubscribe` 移除当前连接对线程的订阅。响应状态为：

- `unsubscribed` 当连接被订阅并现已移除时。
- `notSubscribed` 当连接未订阅该线程时。
- `notLoaded` 当线程未加载时。

如果这是最后一个订阅者，则服务器保持线程加载，直到没有订阅者且线程活动超过 30 分钟。当宽限期过期时，应用服务器会卸载线程并发布 `thread/status/changed` 过渡到 `notLoaded` 加上 `thread/closed`。

```
{ "method": "thread/unsubscribe", "id": 22, "params": { "threadId": "thr_123" } }
{ "id": 22, "result": { "status": "unsubscribed" } }
```

如果线程稍后过期：

```
{ "method": "thread/status/changed", "params": {
    "threadId": "thr_123",
    "status": { "type": "notLoaded" }
} }
{ "method": "thread/closed", "params": { "threadId": "thr_123" } }
```

### 归档线程

使用 `thread/archive` 将持久化线程日志（存储为 JSONL 文件在磁盘上）移动到归档会话目录。

```
{ "method": "thread/archive", "id": 22, "params": { "threadId": "thr_b" } }
{ "id": 22, "result": {} }
{ "method": "thread/archived", "params": { "threadId": "thr_b" } }
```

归档的线程在未来调用 `thread/list` 时不会出现，除非您传递 `archived: true`。

### 取消归档线程

使用 `thread/unarchive` 将已经归档的线程发布放回活动会话目录。

```
{ "method": "thread/unarchive", "id": 24, "params": { "threadId": "thr_b" } }
{ "id": 24, "result": { "thread": { "id": "thr_b", "name": "Bug bash notes" } } }
{ "method": "thread/unarchived", "params": { "threadId": "thr_b" } }
```

### 触发线程压缩

使用 `thread/compact/start` 触发线程的手动历史压缩。该请求立即返回 `{}`。

应用服务器在同一 `threadId` 上流式发送进度，作为标准的 `turn/*` 和 `item/*` 通知，包括 `contextCompaction` 项生命周期（`item/started` 然后 `item/completed`）。

```
{ "method": "thread/compact/start", "id": 25, "params": { "threadId": "thr_b" } }
{ "id": 25, "result": {} }
```

### 运行线程 shell 命令

使用 `thread/shellCommand` 处理属于线程的用户启动的 shell 命令。该请求立即返回 `{}` ，同时进度通过标准的 `turn/*` 和 `item/*` 通知流。

此 API 在沙盒外部运行，具有完全访问权限，并且不继承线程沙盒策略。客户应仅向明确启动的命令公开此项。

如果线程已经具有活动回合，则该命令作为该回合的辅助操作运行，其格式化输出被注入到回合的消息流中。如果线程处于空闲状态，应用服务器会为 shell 命令启动一个独立的回合。

```
{ "method": "thread/shellCommand", "id": 26, "params": { "threadId": "thr_b", "command": "git status --short" } }
{ "id": 26, "result": {} }
```

### 清理后台终端

使用 `thread/backgroundTerminals/clean` 停止与线程关联的所有正在运行的后台终端。该方法是实验性的，并需要 `capabilities.experimentalApi = true`。

```
{ "method": "thread/backgroundTerminals/clean", "id": 27, "params": { "threadId": "thr_b" } }
{ "id": 27, "result": {} }
```

### 回滚最近的回合

使用 `thread/rollback` 从内存上下文中删除最后的 `numTurns` 条目，并在发布日志中持久化回滚标记。返回的 `thread` 包括回滚后填充的 `turns`。

```
{ "method": "thread/rollback", "id": 28, "params": { "threadId": "thr_b", "numTurns": 1 } }
{ "id": 28, "result": { "thread": { "id": "thr_b", "name": "Bug bash notes", "ephemeral": false } } }
```

## 回合

`input` 字段接受项目列表：

- `{ "type": "text", "text": "解释这个 diff" }`
- `{ "type": "image", "url": "https://.../design.png" }`
- `{ "type": "localImage", "path": "/tmp/screenshot.png" }`

您可以在每个回合中覆盖配置设置（模型、工作量、个性、`cwd`、沙盒策略、摘要）。当指定时，这些设置将成为同一线程后续回合的默认设置。`outputSchema` 仅适用于当前回合。对于 `sandboxPolicy.type = "externalSandbox"`，将 `networkAccess` 设置为 `restricted` 或 `enabled`；对于 `workspaceWrite`，`networkAccess` 保持布尔值。

对于 `turn/start.collaborationMode`，`settings.developer_instructions: null` 意味着“使用所选模式的内置指令”，而不是清除模式指令。

### 沙盒读取访问 (`ReadOnlyAccess`)

`sandboxPolicy` 支持明确的读取访问控制：

- `readOnly`：可选 `access`（默认为 `{ "type": "fullAccess" }`，或限制的根）。
- `workspaceWrite`：可选 `readOnlyAccess`（默认为 `{ "type": "fullAccess" }`，或限制的根）。

受限读取访问形状：

```
{
  "type": "restricted",
  "includePlatformDefaults": true,
  "readableRoots": ["/Users/me/shared-read-only"]
}
```

在 macOS 上，`includePlatformDefaults: true` 将附加受限读取会话的策划平台默认 Seatbelt 策略。这在不广泛允许 `/System` 下的所有内容的情况下改善了工具兼容性。

示例：

```
{ "type": "readOnly", "access": { "type": "fullAccess" } }
```

```
{
  "type": "workspaceWrite",
  "writableRoots": ["/Users/me/project"],
  "readOnlyAccess": {

## 审核

`review/start` 运行线程的 Codex 审核者并流式传输审核项目。目标包括：

- `uncommittedChanges`
- `baseBranch`（与某分支进行差异比较）
- `commit`（审核特定提交）
- `custom`（自由形式指令）

使用 `delivery: "inline"`（默认）在现有线程上运行审核，或使用 `delivery: "detached"` 将生成的新审核线程分叉。

请求/响应示例：

```
{ "method": "review/start", "id": 40, "params": {
  "threadId": "thr_123",
  "delivery": "inline",
  "target": { "type": "commit", "sha": "1234567deadbeef", "title": "美化 TUI 颜色" }
} }
{ "id": 40, "result": {
  "turn": {
    "id": "turn_900",
    "status": "inProgress",
    "items": [\
      { "type": "userMessage", "id": "turn_900", "content": [ { "type": "text", "text": "审核提交 1234567: 美化 TUI 颜色" } ] }\
    ],
    "error": null
  },
  "reviewThreadId": "thr_123"
} }
```

对于分离的审核，请使用 `"delivery": "detached"`。响应是相同的形状，但 `reviewThreadId` 将是新的审核线程的 ID（不同于原始的 `threadId`）。服务器还会为该新线程发出 `thread/started` 通知，然后再流式传输审核回合。

Codex 流式传输通常的 `turn/started` 通知，接着发布 `item/started`，其中包含 `enteredReviewMode` 项：

```
{
  "method": "item/started",
  "params": {
    "item": {
      "type": "enteredReviewMode",
      "id": "turn_900",
      "review": "当前更改"
    }
  }
}
```

当审核者完成时，服务器会发布 `item/started` 和 `item/completed`，其中包含 `exitedReviewMode` 项和最终的审核文本：

```
{
  "method": "item/completed",
  "params": {
    "item": {
      "type": "exitedReviewMode",
      "id": "turn_900",
      "review": "总体来说看起来很不错..."
    }
  }
}
```

使用此通知在您的客户端中呈现审核器输出。

## 命令执行

`command/exec` 在服务器沙盒下运行单个命令（`argv` 数组），而无需创建线程。

```
{ "method": "command/exec", "id": 50, "params": {
  "command": ["ls", "-la"],
  "cwd": "/Users/me/project",
  "sandboxPolicy": { "type": "workspaceWrite" },
  "timeoutMs": 10000
} }
{ "id": 50, "result": { "exitCode": 0, "stdout": "...", "stderr": "" } }
```

使用 `sandboxPolicy.type = "externalSandbox"` 如果您已经对服务器进程进行了沙盒配置，并希望 Codex 跳过其自身的沙盒执行。对于外部沙盒模式，将 `networkAccess` 设置为 `restricted`（默认）或 `enabled`。对于 `readOnly` 和 `workspaceWrite`，使用上面相同的可选 `access` / `readOnlyAccess` 结构。

注意：

- 服务器拒绝空的 `command` 数组。
- `sandboxPolicy` 接受与 `turn/start` 相同的形状（例如，`dangerFullAccess`、`readOnly`、`workspaceWrite`、`externalSandbox`）。
- 当省略时，`timeoutMs` 回退到服务器默认值。
- 如果您计划后续使用 `command/exec/write`、`command/exec/resize` 或 `command/exec/terminate`，请设置 `tty: true`。
- 设置 `streamStdoutStderr: true` 以在命令运行时接收 `command/exec/outputDelta` 通知。

### 读取管理员要求（`configRequirements/read`）

使用 `configRequirements/read` 检查从 `requirements.toml` 和/或 MDM 加载的有效管理员要求。

```
{ "method": "configRequirements/read", "id": 52, "params": {} }
{ "id": 52, "result": {
  "requirements": {
    "allowedApprovalPolicies": ["onRequest", "unlessTrusted"],
    "allowedSandboxModes": ["readOnly", "workspaceWrite"],
    "featureRequirements": {
      "personality": true,
      "unified_exec": false
    },
    "network": {
      "enabled": true,
      "allowedDomains": ["api.openai.com"],
      "allowUnixSockets": ["/tmp/example.sock"],
      "dangerouslyAllowAllUnixSockets": false
    }
  }
} }
```

`result.requirements` 为 `null` 时表示没有配置要求。有关支持的键和值的详细信息，请参见 `requirements.toml` 的文档。

### Windows 沙盒设置（`windowsSandbox/setupStart`）

自定义 Windows 客户端可以异步触发沙盒设置，而不是在启动检查上阻塞。

```
{ "method": "windowsSandbox/setupStart", "id": 53, "params": { "mode": "elevated" } }
{ "id": 53, "result": { "started": true } }
```

应用服务器在后台启动设置，稍后会发布完成通知：

```
{
  "method": "windowsSandbox/setupCompleted",
  "params": { "mode": "elevated", "success": true, "error": null }
}
```

模式：

- `elevated` \- 以提高的 Windows 沙盒设置路径运行。
- `unelevated` \- 运行遗留设置/预检路径。

## 文件系统

v2 文件系统 API 在绝对路径上运行。当客户端需要在文件或目录更改后使 UI 状态无效时，请使用 `fs/watch`。

```
{ "method": "fs/watch", "id": 54, "params": {
  "watchId": "0195ec6b-1d6f-7c2e-8c7a-56f2c4a8b9d1",
  "path": "/Users/me/project/.git/HEAD"
} }
{ "id": 54, "result": { "path": "/Users/me/project/.git/HEAD" } }
{ "method": "fs/changed", "params": {
  "watchId": "0195ec6b-1d6f-7c2e-8c7a-56f2c4a8b9d1",
  "changedPaths": ["/Users/me/project/.git/HEAD"]
} }
{ "method": "fs/unwatch", "id": 55, "params": {
  "watchId": "0195ec6b-1d6f-7c2e-8c7a-56f2c4a8b9d1"
} }
{ "id": 55, "result": {} }
```

监视文件将发出 `fs/changed`，针对该文件路径，包括通过替代或重命名操作提供的更新。

## 事件

事件通知是用于线程生命周期、回合生命周期及其内部项目的服务器启动流。在您启动或恢复线程后，请继续在活动传输流中读取 `thread/started`、`thread/archived`、`thread/unarchived`、`thread/closed`、`thread/status/changed`、`turn/*`、`item/*` 和 `serverRequest/resolved` 通知。

### 通知选择退出

客户端可以通过在 `initialize.params.capabilities.optOutNotificationMethods` 中发送确切的方法名称来抑制特定通知。

- 精确匹配仅：`item/agentMessage/delta` 仅抑制该方法。
- 未知的方法名将被忽略。
- 适用于当前的 `thread/*`、`turn/*`、`item/*` 和相关的 v2 通知。
- 不适用于请求、响应或错误。

### 模糊文件搜索事件（实验性）

模糊文件搜索会话 API 发出每个查询通知：

- `fuzzyFileSearch/sessionUpdated` \- `{ sessionId, query, files }`，提供当前活动查询的匹配结果。
- `fuzzyFileSearch/sessionCompleted` \- `{ sessionId }` 一旦索引和匹配该查询完成。

### Windows 沙盒设置事件

- `windowsSandbox/setupCompleted` \- `{ mode, success, error }` 在 `windowsSandbox/setupStart` 请求完成后被发布。

### 回合事件

- `turn/started` \- `{ turn }` 包含回合 ID、空 `items` 和 `status: "inProgress"`。
- `turn/completed` \- `{ turn }` 其中 `turn.status` 是 `completed`、`interrupted` 或 `failed`；失败携带 `{ error: { message, codexErrorInfo?, additionalDetails? } }`。
- `turn/diff/updated` \- `{ threadId, turnId, diff }`，包含回合中每个文件更改的最新聚合统一 diff。
- `turn/plan/updated` \- `{ turnId, explanation?, plan }`，每当代理共享或更改其计划时；每个 `plan` 项目为 `{ step, status }`，其中 `status` 为 `pending`、`inProgress` 或 `completed`。
- `thread/tokenUsage/updated` \- 活动线程的用法更新。

`turn/diff/updated` 和 `turn/plan/updated` 当前包含空的 `items` 数组，即使项目事件流式传输。使用 `item/*` 通知作为回合项目的真实来源。

### 项目

`ThreadItem` 是在回合响应和 `item/*` 通知中携带的标记联合。常见项目类型包括：

- `userMessage` \- `{id, content}` 其中 `content` 是用户输入（`text`、`image` 或 `localImage`）的列表。
- `agentMessage` \- `{id, text, phase?}` 包含累积的代理回复。当存在时，`phase` 使用响应 API 传输值（`commentary`、`final_answer`）。
- `plan` \- `{id, text}` 包含计划模式中的建议计划文本。在 `item/completed` 中处理的最后 `plan` 项可以视为权威。
- `reasoning` \- `{id, summary, content}`，其中 `summary` 包含流式推理摘要，`content` 存储原始推理块。
- `commandExecution` \- `{id, command, cwd, status, commandActions, aggregatedOutput?, exitCode?, durationMs?}`。
- `fileChange` \- `{id, changes, status}` 描述拟议编辑；`changes` 列表 `{path, kind, diff}`。
- `mcpToolCall` \- `{id, server, tool, status, arguments, result?, error?}`。
- `dynamicToolCall` \- `{id, tool, arguments, status, contentItems?, success?, durationMs?}` 用于客户端执行的动态工具调用。
- `collabToolCall` \- `{id, tool, status, senderThreadId, receiverThreadId?, newThreadId?, prompt?, agentStatus?}`。
- `webSearch` \- `{id, query, action?}`，针对代理发出的网页搜索请求。
- `imageView` \- `{id, path}`，当代理调用图像查看工具时发出。
- `enteredReviewMode` \- `{id, review}`，当审核者开始时发送。
- `exitedReviewMode` \- `{id, review}`，当审核者完成时发布。
- `contextCompaction` \- `{id}`，在 Codex 压缩对话历史时发布。

对于 `webSearch.action`，动作 `type` 可以是 `search`（`query?`、`queries?`）、`openPage`（`url?`）或 `findInPage`（`url?`、`pattern?`）。

应用服务器弃用遗留的 `thread/compacted` 通知；请改用 `contextCompaction` 项。

所有项目都发出两个共享生命周期事件：

- `item/started` \- 在新工作单元开始时发出完整的 `item`；`item.id` 与 `itemId` 使用相同的值。
- `item/completed` \- 一旦工作完成，发送最终的 `item`；将其视为权威状态。

### 项目增量

- `item/agentMessage/delta` \- 为代理消息追加流式文本。
- `item/plan/delta` \- 流式提议的计划文本。最终的 `plan` 项可能与串联增量不完全相等。
- `item/reasoning/summaryTextDelta` \- 流式可读的推理摘要；`summaryIndex` 在新摘要部分打开时增加。
- `item/reasoning/summaryPartAdded` \- 标记推理摘要部分之间的边界。
- `item/reasoning/textDelta` \- 流式原始推理文本（如果模型支持）。
- `item/commandExecution/outputDelta` \- 流式命令的 stdout/stderr；按顺序附加增量。
- `item/fileChange/outputDelta` \- 包含对底层 `apply_patch` 工具调用的工具调用响应。

## 错误

如果回合失败，服务器将发出 `error` 事件，负载包含 `{ error: { message, codexErrorInfo?, additionalDetails? } }`，然后以 `status: "failed"` 结束回合。 当上游 HTTP 状态可用时，它将出现在 `codexErrorInfo.httpStatusCode` 中。

常见的 `codexErrorInfo` 值包括：

- `ContextWindowExceeded`
- `UsageLimitExceeded`
- `HttpConnectionFailed`（4xx/5xx 上游错误）
- `ResponseStreamConnectionFailed`
- `ResponseStreamDisconnected`
- `ResponseTooManyFailedAttempts`
- `BadRequest`、`Unauthorized`、`SandboxError`、`InternalServerError`、`Other`

当上游 HTTP 状态可用时，服务器将其转发到相关的 `codexErrorInfo` 变体中的 `httpStatusCode`。

## 审批

根据用户的 Codex 设置，命令执行和文件更改可能需要批准。应用服务器向客户端发送服务器启动的 JSON-RPC 请求，客户端回应以决策负载。

- 命令执行的决策：`accept`、`acceptForSession`、`decline`、`cancel`，或 `{ "acceptWithExecpolicyAmendment": { "execpolicy_amendment": ["cmd", "..."] } }`。
- 文件更改的决策：`accept`、`acceptForSession`、`decline`、`cancel`。

- 请求包括 `threadId` 和 `turnId` \- 使用它们将 UI 状态限制为活动对话。

- 服务器恢复或拒绝工作并结束该项目，带有 `item/completed`。

### 命令执行审批

消息顺序：

1. `item/started` 展示待定的 `commandExecution` 项，带有 `command`、`cwd` 和其他字段。
2. `item/commandExecution/requestApproval` 包括 `itemId`、`threadId`、`turnId`、可选的 `reason`、可选的 `command`、可选的 `cwd`、可选的 `commandActions`、可选的 `proposedExecpolicyAmendment`、可选的 `networkApprovalContext` 和可选的 `availableDecisions`。当 `initialize.params.capabilities.experimentalApi = true` 时，负载还可以包括描述每个命令请求的额外权限的实验性 `additionalPermissions`。任何在 `additionalPermissions` 中的文件系统路径在传输中是绝对的。
3. 客户端用前面提到的命令执行批准决策之一进行响应。
4. `serverRequest/resolved` 确认待处理请求已得到回应或清除。
5. `item/completed` 返回最终的 `commandExecution` 项，状态为 `completed | failed | declined`。

当 `networkApprovalContext` 出现时，提示是针对管理网络访问的（而不是一般的 shell 命令批准）。当前的 v2 模式将公开目标 `host` 和 `protocol`；客户端应呈现网络特定的提示，而 не依赖于 `command` 被视为用户有意义的 shell 命令预览。

Codex 将并发的网络批准提示按目标（`host`、协议和端口）进行分组。因此，应用服务器可能会发送一个提示以解锁多个排队请求到同一目标，而同一主机的不同端口则被视为独立处理。

### 文件更改审批

消息顺序：

1. `item/started` 发布拟议的 `fileChange` 项，状态为 `inProgress`。
2. `item/fileChange/requestApproval` 包括 `itemId`、`threadId`、`turnId`、可选的 `reason` 和可选的 `grantRoot`。
3. 客户端用前面提到的文件更改批准决策之一进行响应。
4. `serverRequest/resolved` 确认待处理请求已得到回应或清除。
5. `item/completed` 返回最终的 `fileChange` 项，状态为 `completed | failed | declined`。

### `tool/requestUserInput`

当客户端对 `item/tool/requestUserInput` 作出响应时，应用服务器发布 `serverRequest/resolved`，负载为 `{ threadId, requestId }`。如果在客户端答复之前，待处理请求被回合开始、回合完成或回合中断清除，服务器将发出相同的通知以进行清理。

### 动态工具调用（实验性）

在 `thread/start` 上的 `dynamicTools` 和对应的 `item/tool/call` 请求或响应流是实验性 API。

在回合过程中，当动态工具被调用时，应用服务器会发布：

1. `item/started`，其 `item.type = "dynamicToolCall"`，`status = "inProgress"`，加上 `tool` 和 `arguments`。
2. `item/tool/call` 作为服务器请求发送给客户端。
3. 客户端响应负载，返回的内容项目。
4. `item/completed`，其 `item.type = "dynamicToolCall"`，包含最终 `status` 和任何返回的 `contentItems` 或 `success` 值。

### MCP 工具调用审批（应用）

应用（连接）工具调用也可能需要审批。当应用工具调用具有副作用时，服务器可能会通过 `tool/requestUserInput` 提示批准，选项包括 **接受**、**拒绝** 和 **取消**。破坏性工具注解总是触发批准，即使该工具还声明较低权限的提示。如果用户拒绝或取消，则相关的 `mcpToolCall` 项完成时将出现错误，而不是运行工具。
