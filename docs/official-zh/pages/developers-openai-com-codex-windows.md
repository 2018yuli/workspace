根据系统和您的设置，Codex 可以通过三种实际方式在 Windows 上运行：

- 在 Windows 上以更强的 `elevated` 沙盒模式本地运行，
- 在 Windows 上以回退的 `unelevated` 沙盒模式本地运行，
- 或在 [Windows 子系统 Linux 2](https://learn.microsoft.com/en-us/windows/wsl/install) (WSL2) 内运行，该实现使用 Linux 沙盒。

## Windows 沙盒

当您在 Windows 上本地运行 Codex 时，代理模式使用 Windows 沙盒来阻止工作文件夹之外的文件系统写入，并在没有您明确批准的情况下防止网络访问。

本地 Windows 沙盒支持包括两种模式，您可以在 `config.toml` 中配置：

```
[windows]
sandbox = "elevated" # 或 "unelevated"


```

`elevated` 是首选的本地 Windows 沙盒。它使用专用的低权限沙盒用户、文件系统权限边界、防火墙规则和本地策略更改，以便为在沙盒中运行的命令提供支持。

`unelevated` 是回退的本地 Windows 沙盒。它使用源自您当前用户的受限 Windows 令牌运行命令，应用基于 ACL 的文件系统边界，并使用环境级离线控制，而不是专用的离线用户防火墙规则。它比 `elevated` 弱，但在管理员批准的设置因本地或企业政策被阻止时仍然有用。

如果两种模式都可用，建议使用 `elevated`。如果默认的本地沙盒在您的环境中无法正常工作，使用 `unelevated` 作为后备，同时排除设置问题。

默认情况下，两种沙盒模式还使用私有桌面，以增强 UI 隔离。仅在需要向后兼容旧的 `Winsta0\\Default` 行为时，将 `windows.sandbox_private_desktop = false` 设置为 false。

### 沙盒权限

以完全访问模式运行 Codex 意味着 Codex 不受限于您的项目目录，并且可能会执行意外的破坏性操作，从而导致数据丢失。为了更安全的自动化，请保持沙盒边界，并使用 [规则](https://developers.openai.com/codex/rules) 进行特定例外，或者将您的 [批准政策设置为\\
never](https://developers.openai.com/codex/agent-approvals-security#run-without-approval-prompts)，以便 Codex 尝试在不请求升级权限的情况下解决问题，这取决于您的 [批准和安全设置](https://developers.openai.com/codex/agent-approvals-security)。

### Windows 版本矩阵

| Windows 版本 | 支持级别 | 备注 |
| --- | --- | --- |
| Windows 11 | 推荐 | Windows 上 Codex 的最佳基础线。如果您要标准化企业部署，请使用此版本。 |
| 最近的、完全更新的 Windows 10 | 尽力而为 | 可以工作，但比 Windows 11 不太可靠。对于 Windows 10，Codex 依赖于现代控制台支持，包括 ConPTY。实际上，要求 Windows 10 版本为 1809 或更高。 |
| 较旧的 Windows 10 版本 | 不推荐 | 更可能缺少所需的控制台组件，如 ConPTY，并且在企业设置中更可能失败。 |

其他环境假设：

- `winget` 应该可用。如果它缺失，请在设置 Codex 之前更新 Windows 或安装 Windows 安装程序。
- 推荐的本地沙盒取决于管理员批准的设置。
- 一些企业管理的设备即使操作系统版本本身可以接受，也会阻止所需的设置步骤。

### 授予沙盒读取访问

当命令因为 Windows 沙盒无法读取目录而失败时，请使用：

```
/sandbox-add-read-dir C:\absolute\directory\path


```

路径必须是现有的绝对目录。命令成功后，后续在沙盒中运行的命令可以在当前会话期间读取该目录。

默认情况下，使用本地 Windows 沙盒。本地 Windows 沙盒提供最佳性能和最高速度，同时保持相同的安全性。当您需要 Windows 上的 Linux 本地环境时，选择 WSL2，或者当工作流已经存在于 WSL2 中时，或当没有哪种本地 Windows 沙盒模式满足您的需求时。

## Windows 子系统 Linux

如果您选择 WSL2，Codex 将在 Linux 环境中运行，而不是使用本地 Windows 沙盒。如果您在 Windows 上需要 Linux 本地工具，如果您的代码仓库和开发工作流已经存在于 WSL2 中，或者当没有哪种本地 Windows 沙盒模式适合您的环境时，这都是有用的。

WSL1 通过 Codex `0.114` 支持。从 Codex `0.115` 开始，Linux 沙盒移至 `bubblewrap`，因此不再支持 WSL1。

### 从 WSL 内启动 VS Code

有关逐步说明，请参见 [官方 VS Code WSL 教程](https://code.visualstudio.com/docs/remote/wsl-tutorial)。

#### 先决条件

- 安装了 WSL 的 Windows。要安装 WSL，请以管理员身份打开 PowerShell，然后运行 `wsl --install`（Ubuntu 是常见的选择）。
- 安装了 [WSL 扩展](https://marketplace.visualstudio.com/items?itemName=ms-vscode-remote.remote-wsl) 的 VS Code。

#### 从 WSL 终端打开 VS Code

```
# 从您的 WSL shell
cd ~/code/your-project
code .


```

这将打开一个 WSL 远程窗口，如果需要，将安装 VS Code 服务器，并确保集成终端在 Linux 中运行。

#### 确认您已连接到 WSL

- 查找绿色状态栏上显示的 `WSL: <distro>`。

- 集成终端应显示 Linux 路径（例如 `/home/...`），而不是 `C:\`。

- 您可以使用以下命令进行验证：

```
echo $WSL_DISTRO_NAME


```

这将打印您的发行版名称。

如果您在状态栏中未看到 “WSL: …”，请按 `Ctrl+Shift+P`，选择 `WSL: Reopen Folder in WSL`，并将您的代码仓库保持在 `/home/...` 下（而不是 `C:\`）以获得最佳性能。

如果 Windows 应用或项目选择器未显示您的 WSL 代码仓库，请在文件选择器或资源管理器中输入 `\wsl$`，然后导航到您发行版的 home 目录。

### 在 WSL 中使用 Codex CLI

从提升的 PowerShell 或 Windows Terminal 运行以下命令：

```
# 安装默认的 Linux 发行版（例如 Ubuntu）
wsl --install

# 在 Windows 子系统 Linux 内启动一个 shell
wsl


```

然后从您的 WSL shell 运行以下命令：

```
# https://learn.microsoft.com/en-us/windows/dev-environment/javascript/nodejs-on-wsl
# 在 WSL 中安装 Node.js（通过 nvm）
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/master/install.sh | bash

# 在新选项卡中或退出后再次运行 `wsl` 安装 Node.js
nvm install 22

# 在 WSL 中安装并运行 Codex
npm i -g @openai/codex
codex


```

### 在 WSL 中处理代码

- 在 Windows 挂载路径下工作，例如 `/mnt/c/…`，可能比在 Windows 本地路径下工作要慢。将您的代码仓库保留在 Linux 主目录下（例如 `~/code/my-app`）以获得更快的 I/O 和更少的符号链接和权限问题：

```
mkdir -p ~/code && cd ~/code
git clone https://github.com/your/repo.git
cd repo


```

- 如果您需要 Windows 访问文件，它们位于资源管理器中的 `\wsl$\Ubuntu\home\<user>` 下。

## 故障排除和常见问题

如果您正在故障排除受管理的 Windows 机器，请从本地沙盒模式、Windows 版本以及 Codex 显示的任何策略错误开始。大多数本地 Windows 支持问题来自沙盒设置、登录权限或文件系统权限，而不是来自编辑器本身。

我的本地沙盒设置失败

如果 Codex 无法完成 `elevated` 沙盒设置，最常见的原因是：

- Windows UAC 或管理员提示被拒绝，
- 该机器不允许本地用户或组创建，
- 该机器不允许防火墙规则更改，
- 该机器阻止沙盒用户所需的登录权限，
- 或者其他企业政策阻止部分设置流程。

尝试以下方法：

1. 再次尝试 `elevated` 沙盒设置，如果您的环境允许，请批准管理员提示。
2. 如果您的公司笔记本阻止此操作，请询问您的 IT 团队该机器是否允许管理员批准设置，以便创建本地用户/组、防火墙配置和所需的沙盒用户登录权限。
3. 如果默认设置仍然失败，请使用 `unelevated` 沙盒，这样您就可以在调查期间继续工作。

Codex 将我切换到非提升沙盒

这意味着 Codex 无法在您的机器上完成更强的 `elevated` 沙盒设置。

- Codex 仍然可以在沙盒模式下运行。
- 它仍然应用基于 ACL 的文件系统边界，但不再使用 `elevated` 的单独沙盒用户边界，并且网络隔离较弱。
- 这是一个有用的后备，但不是首选的长期企业配置。

如果您在管理的企业笔记本上，通常最佳的长期解决方案是让 IT 团队帮助将 `elevated` 沙盒工作。

我看到 Windows 错误 1385

如果沙盒命令因错误 `1385` 失败，则 Windows 拒绝沙盒用户启动命令所需的登录类型。

通常，这意味着 Codex 成功创建沙盒用户，但 Windows 策略仍然阻止这些用户启动沙盒命令。

该怎么做：

1. 请询问您的 IT 团队该设备策略是否授予 Codex 创建的沙盒用户所需的登录权限。
2. 如果该问题仅影响某些机器或团队，请比较组策略或 OU 差异。
3. 如果您需要立即继续工作，请在调查政策问题时使用 `unelevated` 沙盒。
4. 发送 `CODEX_HOME/.sandbox/sandbox.log` 以及您的 Windows 版本和对故障的简短描述。

Codex 警告某些文件夹对 Everyone 可写

Codex 可能会警告某些文件夹对 `Everyone` 可写。

如果您看到此警告，Windows 中这些文件夹的权限对沙盒来说过于宽松，无法完全保护它们。

该怎么做：

1. 检查 Codex 在警告中列出的文件夹。
2. 如果在您的环境中合适，请从这些文件夹中删除 `Everyone` 写入访问权限。
3. 在更正这些权限后，重新启动 Codex 或重新运行沙盒设置。

如果您不确定如何更改这些权限，请向您的 IT 团队请求帮助。

沙盒命令无法访问网络

某些 Codex 任务故意在没有出站网络访问的情况下运行，具体取决于正在使用的权限模式。

如果某个任务因无法访问网络而失败：

1. 检查该任务是否应该在禁用网络的情况下运行。
2. 如果您期望有网络访问，请重新启动 Codex 并尝试再次。
3. 如果问题持续存在，收集沙盒日志，以便团队检查机器是否处于部分或损坏的沙盒状态。

沙盒之前工作过，然后停止

这可能发生在以下情况后：

- 移动了代码仓库或工作区，
- 更改了机器权限，
- 更改了 Windows 策略，
- 或其他系统配置更改。

尝试以下方法：

1. 重新启动 Codex。
2. 再次尝试 `elevated` 沙盒设置。
3. 如果这不能解决问题，请使用 `unelevated` 沙盒作为临时后备。
4. 收集沙盒日志以供审查。

我需要将诊断发送给 OpenAI

如果您仍然遇到问题，请发送：

- `CODEX_HOME/.sandbox/sandbox.log`

包含以下信息也很有帮助：

- 尝试做什么的简短描述，
- `elevated` 沙盒是否失败或使用了 `unelevated` 沙盒，
- 应用中显示的任何错误消息，
- 您是否看到 `1385` 或另一个 Windows 或 PowerShell 错误，
- 您是否使用 Windows 11 或 Windows 10。

请勿发送：

- `CODEX_HOME/.sandbox-secrets/` 的内容。

IDE 扩展已安装但无响应

您的系统可能缺少一些本地依赖项所需的 C++ 开发工具：

- Visual Studio Build Tools (C++ 工作负载)
- Microsoft Visual C++ Redistributable (x64)
- 使用 `winget`，运行 `winget install --id Microsoft.VisualStudio.2022.BuildTools -e`

然后在安装后完全重新启动 VS Code。

大型代码仓库在 WSL 中感觉缓慢

- 确保您没有在 `/mnt/c` 下工作。将代码仓库移至 WSL（例如，`~/code/…`）。
- 如有需要，增加 WSL 的内存和 CPU；将 WSL 更新到最新版本：

```
wsl --update
wsl --shutdown


```

在 WSL 中 VS Code 无法找到 codex

验证二进制文件是否存在并且在 WSL 中的 PATH 中：

```
which codex || echo "codex not found"


```

如果未找到二进制文件，请通过 [上述说明](https://developers.openai.com/codex/windows#use-codex-cli-with-wsl) 安装它。
