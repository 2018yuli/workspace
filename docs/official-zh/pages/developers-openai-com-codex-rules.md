使用规则控制 Codex 可以在沙盒外运行的命令。

规则是实验性的，可能会发生变化。

## 创建规则文件

1. 在活动配置层旁的 `rules/` 文件夹下创建一个 `.rules` 文件（例如，`~/.codex/rules/default.rules`）。

2. 添加规则。该示例在允许 `gh pr view` 在沙盒外运行之前进行提示。


```
# 在沙盒外运行以 `gh pr view` 开头的命令之前进行提示。
prefix_rule(
       # 要匹配的前缀。
       pattern = ["gh", "pr", "view"],

       # 当 Codex 请求运行匹配的命令时采取的行动。
       decision = "prompt",

       # 此规则存在的可选理由。
       justification = "允许查看 PR 经过批准",

       # `match` 和 `not_match` 是可选的“内联单元测试”，您可以在其中提供应该（或不应该）匹配此规则的命令示例。
       match = [\
           "gh pr view 7888",\
           "gh pr view --repo openai/codex",\
           "gh pr view 7888 --json title,body,comments",\
       ],
       not_match = [\
           # 不匹配，因为 `pattern` 必须是一个精确的前缀。\
           "gh pr --repo openai/codex view 7888",\
       ],
)


```

3. 重启 Codex。


Codex 在启动时会扫描每个活动配置层下的 `rules/`，包括 [团队配置](https://developers.openai.com/codex/enterprise/admin-setup#team-config) 位置和用户层 `~/.codex/rules/`。项目本地规则在 `<repo>/.codex/rules/` 下仅在项目 `.codex/` 层被信任时加载。

当您在 TUI 中将命令添加到允许列表时，Codex 会写入用户层 `~/.codex/rules/default.rules`，以便未来的运行可以跳过提示。

启用智能审批（默认设置）时，Codex 可能在升级请求期间为您建议一个
`prefix_rule`。在接受之前请仔细审查建议的前缀。

管理员还可以从 [`requirements.toml`](https://developers.openai.com/codex/enterprise/managed-configuration#admin-enforced-requirements-requirementstoml) 强制执行限制性的 `prefix_rule` 条目。

## 理解规则字段

`prefix_rule()`支持以下字段：

- `pattern` **（必需）**：一个非空列表，定义要匹配的命令前缀。每个元素可以是：

  - 一个字面字符串（例如，`"pr"`）。
  - 字面量的联合（例如，`["view", "list"]`），以匹配该参数位置的替代项。
- `decision` **（默认为 `"allow"`）**：当规则匹配时采取的行动。Codex 在同时匹配多个规则时应用最严格的决定（`forbidden` \> `prompt` \> `allow`）。

  - `allow`：在不提示的情况下在沙盒外运行命令。
  - `prompt`：在每次匹配调用之前进行提示。
  - `forbidden`：在不提示的情况下阻止请求。
- `justification` **（可选）**：规则的非空可读理由。Codex 可能会在审批提示或拒绝消息中显示它。当您使用 `forbidden` 时，在适当情况下推荐替代项应包含在理由中（例如，`"使用 \`rg\` 而不是 \`grep\`。”`）。
- `match` 和 `not_match` **（默认为 `[]`）**：在加载规则时 Codex 验证的示例。使用这些来捕捉错误，以便在规则生效之前发现问题。

当 Codex 考虑执行命令时，会将命令的参数列表与 `pattern` 进行比较。在内部，Codex 将命令视为参数列表（就像 `execvp(3)` 接收的那样）。

## Shell 包装器和复合命令

一些工具将多个 shell 命令包装成一个单一的调用，例如：

```
["bash", "-lc", "git add . && rm -rf /"]


```

由于这种命令可以在一个字符串内隐藏多个操作，Codex 特别处理 `bash -lc`、`bash -c` 及其 `zsh` / `sh` 等效项。

### 当 Codex 可以安全地拆分脚本

如果 shell 脚本是由以下构成的线性命令链：

- 普通单词（没有变量扩展，没有 `VAR=...`、`$FOO`、`*` 等）
- 通过安全操作符连接（`&&`、`||`、`;` 或 `|`）

那么 Codex 会解析它（使用 tree-sitter）并在应用规则之前将其拆分为单独的命令。

上述脚本被视为两个单独的命令：

- `["git", "add", "."]`
- `["rm", "-rf", "/"]`

Codex 然后根据您的规则评估每个命令，并选择最严格的结果。

即使您允许 `pattern=["git", "add"]`，Codex 也不会自动允许 `git add . && rm -rf /`，因为 `rm -rf /` 部分被单独评估，阻止了整个调用的自动允许。

这样做可以防止危险命令与安全命令一起被隐匿。

### 当 Codex 不拆分脚本

如果脚本使用更高级的 shell 特性，例如：

- 重定向（`>`、`>>`、`<`）
- 替换（`$(...)`、`...`）
- 环境变量（`FOO=bar`）
- 通配符模式（`*`、`?`）
- 控制流（`if`、`for`、`&&` 与赋值等）

那么 Codex 不会尝试理解或拆分它。

在这些情况下，整个调用被视为：

```
["bash", "-lc", "<完整脚本>"]


```

并且您的规则应用于该 **单一** 调用。

通过这种处理，当安全时，您获得逐命令评估的安全性，而在不安全时则采取保守行为。

## 测试规则文件

使用 `codex execpolicy check` 来测试您的规则如何应用于命令：

```
codex execpolicy check --pretty \
  --rules ~/.codex/rules/default.rules \
  -- gh pr view 7888 --json title,body,comments


```

该命令发出 JSON，显示严格的决定和任何匹配的规则，包括匹配规则的任何 `justification` 值。使用多个 `--rules` 标志组合文件，并添加 `--pretty` 格式化输出。

## 理解规则语言

`.rules` 文件格式使用 `Starlark`（请参见 [语言规范](https://github.com/bazelbuild/starlark/blob/master/spec.md)）。其语法类似于 Python，但旨在能够安全地运行：规则引擎可以在没有副作用的情况下运行它（例如，操作文件系统）。
