如果您是 Codex 或编码代理的新手，本指南将帮助您更快获得更好的结果。它涵盖了使 Codex 更有效的核心习惯，适用于 [CLI](https://developers.openai.com/codex/cli)、[IDE 扩展](https://developers.openai.com/codex/ide) 和 [Codex 应用](https://developers.openai.com/codex/app)，从提示、规划到验证、MCP、技能和自动化。

当您将 Codex 视为一个可配置和逐步改进的团队成员时，它的效果最佳。

一种有用的思路是：从合适的任务上下文开始，使用 `AGENTS.md` 提供持久的指导，配置 Codex 以匹配您的工作流程，通过 MCP 连接外部系统，将重复工作转变为技能，并自动化稳定的工作流程。

## 强大的首次使用：上下文和提示

即使您的提示不完美，Codex 已经足够强大，能够提供有用的结果。您通常可以在进行最少设置的情况下，将一个难题提交给它，并依然获得强结果。清晰的 [提示](https://developers.openai.com/codex/prompting) 并不是获取价值所必需的，但它确实使结果更可靠，尤其是在更大的代码库或更高风险的任务中。

如果您在一个大型或复杂的代码库中工作，最大解锁是在为 Codex 提供正确的任务上下文和所需完成内容的清晰结构。

一个好的默认做法是在您的提示中包含四个内容：

- **目标：** 您想要更改或构建什么？
- **上下文：** 哪些文件、文件夹、文档、示例或错误与此任务相关？您可以提到某些文件作为上下文。
- **约束：** Codex 应遵循哪些标准、架构、安全要求或约定？
- **完成的标准：** 任务完成前必须满足的条件，例如测试通过、行为改变或错误不再重现？

这有助于 Codex 保持范围明确，做更少的假设，并生产更容易审核的工作。

根据任务的难度选择推理级别，并测试哪个选项最适合您的工作流程。不同的用户和任务在不同的设置下效果最佳。

- 针对更快、范围较小的任务使用低级别
- 针对更复杂的变更或调试使用中级或高级
- 针对较长、代理驱动的推理事务使用额外高级

为了更快提供上下文，尝试在 Codex 应用中使用语音输入，而不是键入您希望 Codex 做的事情。

## 对于困难任务先规划

如果任务复杂、模糊或难以良好描述，请在 Codex 开始编码之前请求它进行规划。

以下几种方法效果良好：

**使用计划模式：** 对于大多数用户，这是最简单、最有效的选择。计划模式让 Codex 收集上下文、提问澄清问题，并在实施之前制定更强的计划。通过 `/plan` 或 `Shift` + `Tab` 切换。

**请求 Codex 采访您：** 如果您对自己想要的内容有大概想法，但不确定如何恰当地描述，可以先让 Codex 问您。告诉它挑战您的假设，并将模糊的想法转变为具体内容，然后再编写代码。

**使用 PLANS.md 模板：** 对于更高级的工作流程，您可以配置 Codex 遵循 `PLANS.md` 或执行计划模板，用于长时间运行或多步骤的工作。更多细节，请参见 [执行计划指南](https://developers.openai.com/cookbook/articles/codex_exec_plans)。

## 通过 `AGENTS.md` 使指导可重用

一旦提示模式有效，下一步是停止手动重复它。这就是 [AGENTS.md](https://developers.openai.com/codex/guides/agents-md) 的作用。

将 `AGENTS.md` 视为代理的开放格式 README。它会自动加载到上下文中，是编码您和您的团队希望 Codex 在代码库中工作方式的最佳地方。

一个好的 `AGENTS.md` 包含：

- 回购布局和重要目录
- 如何运行项目
- 构建、测试和 lint 命令
- 工程约定和 PR 期望
- 约束和禁忌规则
- 完成的定义以及如何验证工作

CLI 中的 `/init` 斜杠命令是快速启动命令，在当前目录中搭建一个起始的 `AGENTS.md`。这是一种很好的起点，但您应该编辑结果以匹配您的团队实际构建、测试、审查和发布代码的方式。

您可以在不同级别创建 `AGENTS.md` 文件：个人默认的全局 `AGENTS.md` 存放在 `~/.codex` 中，共享标准的回购级文件，以及更具体的子目录中的本地规则文件。如果当前目录中有更具体的文件，该指导优先。

保持实用。简短、准确的 `AGENTS.md` 比充满模糊规则的长文件更有用。从基础开始，然后仅在您注意到重复错误后添加新规则。

如果 `AGENTS.md` 开始变得过于庞大，则保持主文件简洁，并参考特定任务的 markdown 文件，例如计划、代码审查或架构。

当 Codex 两次犯同样的错误时，请要求它进行回顾并更新 `AGENTS.md`。指导保持实用，并基于真实摩擦。

## 为一致性配置 Codex

配置是使 Codex 在会话和表面之间行为更一致的主要方式之一。例如，您可以设置模型选择、推理工作量、沙盒模式、批准政策、配置文件和 MCP 设置的默认值。

一个好的起始模式是：

- 将个人默认设置保存在 `~/.codex/config.toml` 中（设置 → 配置 → 从 Codex 应用中打开 config.toml）
- 在 `.codex/config.toml` 中保留回购特定行为
- 仅在一次性情况下使用命令行覆盖（如果您使用 CLI）

[`config.toml`](https://developers.openai.com/codex/config-basic) 是您定义持久偏好的地方，例如 MCP 服务器、配置文件、多代理设置和功能标志。您可以直接编辑它或要求 Codex 为您更新它。

Codex 附带操作级沙盒，并具有两个关键控制。这两种模式分别决定了 Codex 什么时候需要您的权限来执行命令，和 Codex 是否可以在目录中读取或写入以及代理可以访问哪些文件。

如果您是编码代理的新手，请先使用默认权限。默认情况下，将批准和沙盒保持严格，然后在需要明确后仅对受信任的回购或特定工作流程放宽权限。

请注意，CLI、IDE 和 Codex 应用共享相同的配置层。了解更多信息，请参见 [示例配置](https://developers.openai.com/codex/config-sample) 页面。

及早为您的实际环境配置 Codex。许多质量问题实际上是设置问题，比如错误的工作目录、缺少写权限、错误的模型默认设置或缺少工具和连接器。

## 通过测试和审查提高可靠性

不要仅仅停留在要求 Codex 进行更改上。要求它在必要时创建测试、运行相关检查、确认结果，并在您接受之前审查工作。

Codex 可以为您执行这个循环，但前提是它知道“好”的样子。该指导可以来自提示或 `AGENTS.md`。

这可以包括：

- 为更改编写或更新测试
- 运行正确的测试套件
- 检查 lint、格式或类型检查
- 确认最终行为与请求相符
- 审查 diff 以查找错误、回归或风险模式

在 Codex 应用中切换 diff 面板以直接 [审查变更](https://developers.openai.com/codex/app/review)。单击特定行以提供反馈，为下一个 Codex 回合提供上下文。

这里有一个有用的选择是 `/review` 斜杠命令，它为您提供几种审查代码的方法：

- 针对基础分支进行审查，适用 PR 风格审查
- 审查未提交的更改
- 审查某次提交
- 使用自定义审查说明

如果您和团队有一个 `code_review.md` 文件并在 `AGENTS.md` 中引用它，Codex 也可以在审查过程中遵循该指南。这对希望审查行为在不同回购和贡献者之间保持一致的团队是一种强有力的模式。

Codex 不仅仅应生成代码。在正确的指令下，它还可以帮助 **测试、检查和审查**。

如果您使用 GitHub Cloud，可以将 Codex 设置为对您的 PR 进行 [代码审查](https://developers.openai.com/codex/integrations/github)。在 OpenAI，Codex 审查了 100% 的 PR。您可以启用自动审查或让 Codex 在您 @Codex 时进行逐步审查。

## 使用 MCP 获取外部上下文

当 Codex 所需的上下文存在于回购之外时，使用 MCP。这让 Codex 连接到您已经使用的工具和系统，因此您无需不断将实时信息复制粘贴到提示中。

[模型上下文协议](https://developers.openai.com/codex/mcp)，或 MCP，是一个连接 Codex 与外部工具和系统的开放标准。

当以下情况出现时使用 MCP：

- 所需的上下文存在于回购之外
- 数据频繁变化
- 您希望 Codex 使用工具而不是依赖粘贴的说明
- 您需要在用户或项目之间进行可重复的集成

Codex 支持使用 OAuth 的 STDIO 和可流式传输的 HTTP 服务器。

在 Codex 应用中，前往 设置 → MCP 服务器 查看自定义和推荐服务器。通常，Codex 可以帮助您安装所需的服务器。您只需求助于它。您还可以在 CLI 中使用 `codex mcp add` 命令以名称、URL 和其他详细信息添加您的自定义服务器。

仅当它们解锁真实的工作流程时，才添加工具。不要一开始就连接您使用的每个工具。首先从一两个明显消除的手动循环的工具开始，然后再扩展。

## 将可重复的工作转化为技能

一旦工作流程变得可以重复，停止依赖冗长的提示或重复的来回交流。使用 [技能](https://developers.openai.com/codex/skills) 将指令打包在 SKILL.md 文件中，包含上下文和 Codex 应一致应用的支持逻辑。技能适用于 CLI、IDE 扩展和 Codex 应用。

将每个技能范围限制为一个任务。首先定义 2 到 3 个具体用例，明确输入和输出，并编写描述，以说明技能的功能和使用时机。包括用户实际会说的触发短语类型。

不要试图一次涵盖每一个边缘情况。首先从一个代表性任务开始，使其运行良好，然后将该工作流程转化为技能并从中改进。仅在提高可靠性的情况下才包括脚本或额外的资源。

一个好的经验法则是：如果您持续使用同一提示或纠正相同工作流程，它很可能应成为一项技能。

技能对于重复作业尤其有用，例如：

- 日志筛选
- 发布说明草拟
- 针对检查表的 PR 审查
- 迁移规划
- 监测或事件摘要
- 标准调试流程

`$skill-creator` 技能是搭建技能第一版的最佳起点。在迭代的过程中将第一版本保留为本地版本。当它准备好进行广泛共享时，将其打包为 [插件](https://developers.openai.com/codex/plugins/build)。技能的一个最重要的部分是描述。它应该说明技能的功能以及使用场合。

个人技能存储在 `$HOME/.agents/skills` 中，而共享团队技能可以存储在回购内部的 `.agents/skills` 中。这对于新团队成员的入职培训尤其有帮助。

## 使用自动化处理重复工作

一旦工作流程稳定，您可以安排 Codex 在后台为您运行它。在 Codex 应用中， [自动化](https://developers.openai.com/codex/app/automations) 让您选择项目、提示、周期以及执行环境以进行重复任务。

一旦某个任务在您这里变得重复，您可以在 Codex 应用的自动化选项卡中创建一个自动化。您可以选择运行哪个项目、运行的提示（您可以调用技能）以及将要运行的频率。您还可以选择自动化是在专用的 git 工作树中运行还是在本地环境中运行。了解更多关于 [git 工作树](https://developers.openai.com/codex/app/worktrees)。

良好的候选任务包括：

- 总结最近的提交
- 扫描可能的错误
- 草拟发布说明
- 检查 CI 失败
- 生成站会摘要
- 定期运行可重复分析工作流程

一个有用的规则是，技能定义方法，自动化定义时间表。如果工作流程仍然需要大量引导，请先将其转变为技能。一旦它变得可预测，自动化便成为倍增器。

将自动化用于反思和维护，而不仅仅是执行。审查最近的会话，总结重复的摩擦，并随着时间的推移改进提示、指令或工作流程设置。

## 使用会话控制组织长期工作

Codex 会话不仅仅是聊天记录。它们是随着时间积累上下文、决策和行动的工作线程，因此良好的管理对质量有重大影响。

Codex 应用 UI 使线程管理变得最简单，因为您可以固定线程并创建工作树。如果您使用 CLI，这些 [斜杠命令](https://developers.openai.com/codex/cli/slash-commands) 特别有用：

- `/experimental` 切换实验功能并加入到您的 `config.toml`
- `/resume` 恢复已保存的对话
- `/fork` 创建新线程，同时保留原始记录
- `/compact` 当线程变得冗长时，您想要一个更早上下文的总结版本。请注意，Codex 会自动为您压缩对话。
- `/agent` 当您运行并行代理并希望在活动代理线程之间切换时使用
- `/theme` 选择语法高亮主题
- `/apps` 直接在 Codex 中使用 ChatGPT 应用
- `/status` 检查当前会话状态

每个工作单元保持一个线程。如果工作仍然是同一问题的一部分，保持在同一线程通常会更好，因为它保留了推理轨迹。仅在工作真正分叉时进行分支。

利用 Codex 的 [子代理](https://developers.openai.com/codex/concepts/subagents) 工作流程，将有界工作从主要线程卸载。保持主要代理专注于核心问题，并使用子代理处理探索、测试或筛选等任务。

## 常见错误

一些在首次使用 Codex 时应避免的常见错误：

- 在提示中负担过重，而不是将持久规则移动到 `AGENTS.md` 或技能中
- 没有让代理看见其工作的细节，例如未提供最佳的构建和测试命令运行方式
- 在多步骤和复杂任务中跳过规划
- 在理解工作流程之前，给予 Codex 电脑的完全权限
- 对同一文件运行实时线程而不使用 git 工作树
- 在手动尚不可靠前，将重复任务转变为自动化
- 将 Codex 视为必须逐步监控的工具，而不是与自己的工作并行使用
- 将每个项目视为一个线程，而不是每个任务保持一个线程。这会导致上下文膨胀，从而降低结果质量
