# 作为 MCP 服务器运行 Codex

您可以将 Codex 作为 MCP 服务器运行，并从其他 MCP 客户端连接它（例如，用 [OpenAI Agents SDK MCP 集成](https://developers.openai.com/api/docs/guides/agents/integrations-observability#mcp) 构建的代理）。

要将 Codex 启动为 MCP 服务器，您可以使用以下命令：

```
codex mcp-server
```

您可以使用 [模型上下文协议检查器](https://modelcontextprotocol.io/legacy/tools/inspector) 启动 Codex MCP 服务器：

```
npx @modelcontextprotocol/inspector codex mcp-server
```

发送 `tools/list` 请求以查看两个工具：

**`codex`**: 运行一个 Codex 会话。接受与 Codex `Config` 结构匹配的配置参数。`codex` 工具接受以下属性：

| 属性 | 类型 | 描述 |
| --- | --- | --- |
| **`prompt`**（必需） | `string` | 开始 Codex 对话的初始用户提示。 |
| `approval-policy` | `string` | 模型生成的 shell 命令的批准政策：`untrusted`、`on-request` 和 `never`。 |
| `base-instructions` | `string` | 用作默认指令的说明集合。 |
| `config` | `object` | 单个配置设置，覆盖 `$CODEX_HOME/config.toml` 中的内容。 |
| `cwd` | `string` | 会话的工作目录。如果是相对的，将按照服务器进程的当前目录解析。 |
| `include-plan-tool` | `boolean` | 是否在对话中包含计划工具。 |
| `model` | `string` | 可选覆盖模型名称（例如 `o3`、`o4-mini`）。 |
| `profile` | `string` | 从 `config.toml` 指定默认选项的配置文件。 |
| `sandbox` | `string` | 沙箱模式：`read-only`、`workspace-write` 或 `danger-full-access`。 |

**`codex-reply`**: 通过提供线程 ID 和提示来继续 Codex 会话。`codex-reply` 工具采用以下属性：

| 属性 | 类型 | 描述 |
| --- | --- | --- |
| **`prompt`**（必需） | string | 继续 Codex 对话的下一个用户提示。 |
| **`threadId`**（必需） | string | 要继续的线程 ID。 |
| `conversationId`（已弃用） | string | `threadId` 的已弃用别名（保留以兼容）。 |

使用来自 `tools/call` 响应中的 `structuredContent.threadId` 的 `threadId`。批准提示（exec/patch）还在其 `params` 负载中包含 `threadId`。

示例响应有效负载：

```
{
  "structuredContent": {
    "threadId": "019bbb20-bff6-7130-83aa-bf45ab33250e",
    "content": "`ls -lah`（或`ls -alh`）— 长列表，包括点文件，大小可读。"
  },
  "content": [\
    {\
      "type": "text",\
      "text": "`ls -lah`（或`ls -alh`）— 长列表，包括点文件，大小可读。"\
    }\
  ]
}
```

注意，现代 MCP 客户端通常仅报告 `"structuredContent"` 作为工具调用的结果（如果存在），尽管 Codex MCP 服务器也返回 `"content"` 以方便较旧的 MCP 客户端。

# 创建多代理工作流

Codex CLI 可以做的远不止运行临时任务。通过将 CLI 作为 [模型上下文协议](https://modelcontextprotocol.io/)（MCP）服务器公开，并使用 OpenAI Agents SDK 进行编排，您可以创建可预测、可审查的工作流，这些工作流可以从单个代理扩展到完整的软件交付管道。

本指南将演示在 [OpenAI 食谱](https://github.com/openai/openai-cookbook/blob/main/examples/codex/codex_mcp_agents_sdk/building_consistent_workflows_codex_cli_agents_sdk.ipynb) 中展示的相同工作流。您将：

- 启动 Codex CLI 作为一个长期运行的 MCP 服务器，
- 构建一个专注的单代理工作流，产生一个可玩的浏览器游戏，并
- 合成一个多代理团队，包括交接、护栏和您可以后续审查的完整跟踪。

在开始之前，请确保您已经：

- 本地安装 [Codex CLI](https://developers.openai.com/codex/cli)，以便 `npx codex` 可以运行。
- Python 3.10 及以上版本，带有 `pip`。
- Node.js 18 及以上（`npx` 所需）。
- 本地存储的 OpenAI API 密钥。您可以在 [OpenAI 仪表盘](https://platform.openai.com/account/api-keys) 中创建或管理密钥。

为本指南创建一个工作目录，并将您的 API 密钥添加到 `.env` 文件中：

```
mkdir codex-workflows
cd codex-workflows
printf "OPENAI_API_KEY=sk-..." > .env
```

## 将 Codex CLI 初始化为 MCP 服务器

首先将 Codex CLI 转换为 Agents SDK 可以调用的 MCP 服务器。服务器公开两个工具（`codex()` 开始对话，`codex-reply()` 继续对话），并保持 Codex 在多个代理轮次之间处于活动状态。

创建一个名为 `codex_mcp.py` 的文件，并添加以下内容：

```
import asyncio

from agents import Agent, Runner
from agents.mcp import MCPServerStdio

async def main() -> None:
    async with MCPServerStdio(
        name="Codex CLI",
        params={
            "command": "npx",
            "args": ["-y", "codex", "mcp-server"],
        },
        client_session_timeout_seconds=360000,
    ) as codex_mcp_server:
        print("Codex MCP server started.")
        # 更多逻辑将在后面的部分中到来。
        return

if __name__ == "__main__":
    asyncio.run(main())
```

运行脚本一次以验证 Codex 是否成功启动：

```
python codex_mcp.py
```

脚本在打印 `Codex MCP server started.` 后退出。在接下来的部分中，您将重用同一个 MCP 服务器，结合更丰富的工作流。

## 构建单代理工作流

让我们从一个使用 Codex MCP 发货的小浏览器游戏的示例开始。该工作流依赖于两个代理：

1. **游戏设计师**：编写游戏简要说明。
2. **游戏开发者**：通过调用 Codex MCP 实现游戏。

使用以下代码更新 `codex_mcp.py`。它保留了上述 MCP 服务器的设置，并添加了两个代理。

```
import asyncio
import os

from dotenv import load_dotenv

from agents import Agent, Runner, set_default_openai_api
from agents.mcp import MCPServerStdio

load_dotenv(override=True)
set_default_openai_api(os.getenv("OPENAI_API_KEY"))

async def main() -> None:
    async with MCPServerStdio(
        name="Codex CLI",
        params={
            "command": "npx",
            "args": ["-y", "codex", "mcp-server"],
        },
        client_session_timeout_seconds=360000,
    ) as codex_mcp_server:
        developer_agent = Agent(
            name="游戏开发者",
            instructions=(
                "您是使用基本的 html + css + javascript 构建简单游戏的专家。"
                "将您的工作保存在当前目录中的名为 index.html 的文件中。"
                "始终以 \"approval-policy\": \"never\" 和 \"sandbox\": \"workspace-write\" 调用 codex。"
            ),
            mcp_servers=[codex_mcp_server],
        )

        designer_agent = Agent(
            name="游戏设计师",
            instructions=(
                "您是一个独立的游戏爱好者。想出一个单页面 html + css + javascript 游戏的想法，开发者可以在大约 50 行代码内构建。"
                "把您的请求格式化为对游戏开发者的 3 句设计简报，并调用游戏开发者编码您构想的内容。"
            ),
            model="gpt-5",
            handoffs=[developer_agent],
        )

        await Runner.run(designer_agent, "实现一个有趣的新游戏！")

if __name__ == "__main__":
    asyncio.run(main())
```

执行脚本：

```
python codex_mcp.py
```

Codex 将读取设计师的简要说明，创建一个 `index.html` 文件，并将整个游戏写入磁盘。在浏览器中打开生成的文件以玩游戏。每次运行都会产生不同的设计，具有独特的游戏风格和细致度。

## 扩展到多代理工作流

现在将单代理设置转变为一个协调的、可追踪的工作流。系统增加：

- **项目经理**：创建共享需求，协调交接，并执行护栏。
- **设计师**、**前端开发者**、**服务器开发者** 和 **测试者**：每个角色都有特定的指令和输出文件夹。

创建一个名为 `multi_agent_workflow.py` 的新文件：

```
import asyncio
import os

from dotenv import load_dotenv

from agents import (
    Agent,
    ModelSettings,
    Runner,
    WebSearchTool,
    set_default_openai_api,
)
from agents.extensions.handoff_prompt import RECOMMENDED_PROMPT_PREFIX
from agents.mcp import MCPServerStdio
from openai.types.shared import Reasoning

load_dotenv(override=True)
set_default_openai_api(os.getenv("OPENAI_API_KEY"))

async def main() -> None:
    async with MCPServerStdio(
        name="Codex CLI",
        params={"command": "npx", "args": ["-y", "codex", "mcp"]},
        client_session_timeout_seconds=360000,
    ) as codex_mcp_server:
        designer_agent = Agent(
            name="设计师",
            instructions=(
                f"""{RECOMMENDED_PROMPT_PREFIX}"""
                "您是设计师。\n"
                "您唯一的真实来源是来自项目经理的 AGENT_TASKS.md 和 REQUIREMENTS.md。\n"
                "不要假设任何未写明的事项。\n\n"
                "您可以使用互联网进行额外的指导或研究。"
                "交付物（写入 /design）：\n"
                "- design_spec.md – 一页描述用户界面/用户体验布局、主要屏幕和关键视觉说明，正如 AGENT_TASKS.md 所要求的。\n"
                "- wireframe.md – 如果有指定的简单文本或 ASCII 线框。\n\n"
                "保持输出简短且易于实现。\n"
                "完成后，通过 transfer_to_project_manager 将其移交给项目经理。"
                "在创建文件时，以 {\"approval-policy\":\"never\",\"sandbox\":\"workspace-write\"} 方式调用 Codex MCP。"
            ),
            model="gpt-5",
            tools=[WebSearchTool()],
            mcp_servers=[codex_mcp_server],
        )

        frontend_developer_agent = Agent(
            name="前端开发者",
            instructions=(
                f"""{RECOMMENDED_PROMPT_PREFIX}"""
                "您是前端开发者。\n"
                "阅读 AGENT_TASKS.md 和 design_spec.md。按照其中描述的内容实现。\n\n"
                "交付物（写入 /frontend）：\n"
                "- index.html – 主页结构\n"
                "- styles.css 或如果有指定的内联样式\n"
                "- main.js 或如果有指定的 game.js\n\n"
                "遵循设计师提供的 DOM 结构和项目经理提供的任何集成点。\n"
                "不要添加超出所提供文档的功能或品牌。\n\n"
                "完成后，通过 transfer_to_project_manager_agent 将其移交给项目经理。"
                "在创建文件时，以 {\"approval-policy\":\"never\",\"sandbox\":\"workspace-write\"} 方式调用 Codex MCP。"
            ),
            model="gpt-5",
            mcp_servers=[codex_mcp_server],
        )

        backend_developer_agent = Agent(
            name="后端开发者",
            instructions=(
                f"""{RECOMMENDED_PROMPT_PREFIX}"""
                "您是后端开发者。\n"
                "阅读 AGENT_TASKS.md 和 REQUIREMENTS.md。实现其中描述的后端端点。\n\n"
                "交付物（写入 /backend）：\n"
                "- package.json – 如果有要求，请包括启动脚本\n"
                "- server.js – 按照指定的内容实现 API 端点和逻辑\n\n"
                "保持代码尽可能简单和可读。不允许外部数据库。\n\n"
                "完成后，通过 transfer_to_project_manager_agent 移交给项目经理。"
                "在创建文件时，以 {\"approval-policy\":\"never\",\"sandbox\":\"workspace-write\"} 方式调用 Codex MCP。"
            ),
            model="gpt-5",
            mcp_servers=[codex_mcp_server],
        )

        tester_agent = Agent(
            name="测试者",
            instructions=(
                f"""{RECOMMENDED_PROMPT_PREFIX}"""
                "您是测试者。\n"
                "阅读 AGENT_TASKS.md 和 TEST.md。确认其他角色的输出是否符合接受标准。\n\n"
                "交付物（写入 /tests）：\n"
                "- TEST_PLAN.md – 按要求的手动检查或自动化步骤的项目列表\n"
                "- test.sh 或如果有指定的简单自动化脚本\n\n"
                "保持简洁且易于运行。\n\n"
                "完成后，通过 transfer_to_project_manager 将其移交给项目经理。"
                "在创建文件时，以 {\"approval-policy\":\"never\",\"sandbox\":\"workspace-write\"} 方式调用 Codex MCP。"
            ),
            model="gpt-5",
            mcp_servers=[codex_mcp_server],
        )

        project_manager_agent = Agent(
            name="项目经理",
            instructions=(
                f"""{RECOMMENDED_PROMPT_PREFIX}"""
                """
                您是项目经理。

                目标：
                将输入任务列表转化为团队将在其下执行的三个项目根文件。

                交付物（写入项目根）：
                - REQUIREMENTS.md：产品目标、目标用户、关键功能和约束的简要总结。
                - TEST.md：带有 [Owner] 标签（设计师、前端、后端、测试者）和明确接受标准的任务。
                - AGENT_TASKS.md：每个角色一个部分，包括：
                  - 项目名称
                  - 所需交付物（确切的文件名称和目的）
                  - 关键技术说明和约束

                过程：
                - 通过最小的合理假设解决模糊问题。具体明确，以便每个角色可以在没有猜测的情况下行动。
                - 使用 Codex MCP 创建文件，使用 {"approval-policy":"never","sandbox":"workspace-write"}。
```
