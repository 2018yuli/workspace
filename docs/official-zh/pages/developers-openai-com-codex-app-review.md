审阅窗格帮助您了解 Codex 更改了什么，提供针对性反馈，并决定保留哪些内容。

它仅适用于存放在 Git 仓库中的项目。如果您的项目还不是 Git 仓库，审阅窗格会提示您创建一个。

## 显示的更改

审阅窗格反映您的 Git 仓库的状态，而不仅仅是 Codex 编辑的内容。这意味着它将显示：

- Codex 进行的更改
- 您自己所做的更改
- 仓库中任何其他未提交的更改

默认情况下，审阅窗格专注于 **未提交的更改**。您还可以切换范围到：

- **所有分支更改**（与您的基础分支的差异）
- **最后一次对话更改**（仅最近的助手对话）

在本地工作时，您还可以在 **未暂存** 和 **已暂存** 更改之间切换。

## 导航审阅窗格

- 单击文件名通常会在您选择的编辑器中打开该文件。您可以在 [设置](https://developers.openai.com/codex/app/settings) 中选择默认编辑器。
- 单击文件名背景可展开或收起差异。
- 按住 `Cmd` 的同时单击单行将打开该行在您选择的编辑器中。
- 如果您对更改感到满意，可以 [暂存更改或还原不喜欢的更改](https://developers.openai.com/codex/app/review#staging-and-reverting-files)。

## 反馈的行内评论

行内评论允许您将反馈直接附加到差异中的特定行。这通常是引导 Codex 进行正确修复的最快方法。

要留下行内评论：

1. 打开审阅窗格。
2. 将光标悬停在您想要评论的行上。
3. 单击出现的 **+** 按钮。
4. 写下您的反馈并提交。
5. 结束反馈后，回复该线程。

由于评论是针对特定行的，Codex 可以比一般指令更精确地作出响应。

Codex 将行内评论视为审阅指导。留下评论后，发送一条后续消息使您的意图明确，例如“处理行内评论并保持范围最小”。

## 拉取请求审阅

当 Codex 拥有您仓库的 GitHub 访问权限，并且当前项目位于拉取请求分支上时，Codex 应用可以帮助您处理拉取请求反馈，而无需离开应用。侧边栏显示拉取请求的上下文和审核者的反馈，审阅窗格显示评论和差异的并列，这样您可以要求 Codex 在同一线程中解决问题。

安装 GitHub CLI (`gh`) 并使用 `gh auth login` 进行身份验证，以便 Codex 可以加载拉取请求上下文、审阅评论和更改文件。如果 `gh` 缺失或未验证，侧边栏或审阅窗格可能不会显示拉取请求的详细信息。

在想在一个地方保持完整修复循环时，请使用以下流程：

1. 在拉取请求分支上打开审阅窗格。
2. 审阅拉取请求的上下文、评论和更改文件。
3. 请求 Codex 修复您想处理的特定评论。
4. 检查审阅窗格中的结果差异。
5. 准备好后，将更改暂存、提交并推送到 PR 分支。

有关 GitHub 触发的审阅，请参见 [在 GitHub 中使用 Codex](https://developers.openai.com/codex/integrations/github)。

## 暂存和还原文件

审阅窗格包括 Git 操作，以便您可以在提交之前调整差异。

您可以在以下级别进行暂存、取消暂存或还原更改：

- **整个差异**：使用审阅顶部的操作按钮（例如，“全部暂存”或“全部还原”）
- **每个文件**：暂存、取消暂存或还原单个文件
- **每个块**：暂存、取消暂存或还原单个块

当您想接受部分工作时使用暂存，当您想放弃时使用还原。

### 暂存和未暂存状态

Git 可以在同一文件中表示既已暂存又未暂存的更改。当发生这种情况时，窗格可能会显示“同一文件两次”的状态在已暂存和未暂存视图中。这是正常的 Git 行为。
