沙盒是一个边界，可以让 Codex 自主运作，而不需要给予其对您机器的无限制访问。当 Codex 在 **Codex 应用程序**、**IDE 扩展**或 **CLI** 中运行本地命令时，这些命令会在一个受限的环境中运行，而不是默认情况下拥有完全访问权限。

该环境定义了 Codex 自主行动的能力，比如它可以修改哪些文件，以及命令是否可以使用网络。当任务保持在这些边界内时，Codex 可以不断进行，而无需停下来确认。当它需要超越这些边界时，Codex 会回退到批准流程。

沙盒和批准是不同的控制机制，它们共同发挥作用。沙盒定义了技术边界，而批准政策决定了何时 Codex 必须停止并询问才能越过这些边界。

## 沙盒的作用

沙盒适用于生成的命令，而不仅限于 Codex 内置的文件操作。如果 Codex 运行像 `git`、包管理器或测试运行器这样的工具，这些命令也会继承相同的沙盒边界。

Codex 在每个操作系统上使用平台本地强制。实现会在 macOS、Linux、WSL2 和原生 Windows 之间有所不同，但在各个表面上的理念相同：为代理提供一个有界的工作场所，以便例行任务可以在明确的限制中自主运行。

## 重要性

沙盒减少了批准疲劳。Codex 不再要求您确认每个低风险命令，而可以在您已经批准的边界内读取文件、进行编辑并运行例行项目命令。

它还为自主工作提供了更清晰的信任模型。您不仅在信任代理的意图；您还在信任代理在强制限制内运行。这使得让 Codex 独立工作变得更容易，同时仍然可以知道它何时会停止并请求帮助。

## 开始使用

Codex 在您使用默认权限模式时自动应用沙盒。

### 前提条件

在 **macOS** 上，沙盒开箱即用，使用内置的 Seatbelt 框架。

在 **Windows** 上，当您在 PowerShell 中运行时，Codex 使用原生的 [Windows\\
沙盒](https://developers.openai.com/codex/windows#windows-sandbox)，而在 WSL2 中运行时使用 Linux 沙盒实现。

在 **Linux 和 WSL2** 中，首先使用您的包管理器安装 `bubblewrap`：

选择一个选项

Ubuntu/DebianFedora

```
sudo apt install bubblewrap
```

```
sudo dnf install bubblewrap
```

Codex 使用在 `PATH` 中找到的第一个 `bwrap` 可执行文件。如果没有可用的 `bwrap` 可执行文件，Codex 会回退到捆绑的助手，但该助手要求支持无特权用户命名空间的创建。安装提供 `bwrap` 的发行版包可以保持此设置的可靠性。

当 `bwrap` 缺失或助手无法创建所需的用户命名空间时，Codex 会显示启动警告。在限制此 AppArmor 设置的发行版上，建议加载 `bwrap` AppArmor 配置文件，以便 `bwrap` 可以在不全局禁用限制的情况下继续工作。

**Ubuntu AppArmor 注意事项：** 在 Ubuntu 25.04 上，从 Ubuntu 包存储库中安装 `bubblewrap` 应该可以在没有额外 AppArmor 设置的情况下正常工作。 `bwrap-userns-restrict` 配置文件在 `apparmor` 包中提供，路径为 `/etc/apparmor.d/bwrap-userns-restrict`。

在 Ubuntu 24.04 上，尽管安装了 `bubblewrap`，Codex 仍可能警告无法创建所需的用户命名空间。复制并加载额外的配置文件：

```
sudo apt update
sudo apt install apparmor-profiles apparmor-utils
sudo install -m 0644 \
  /usr/share/apparmor/extra-profiles/bwrap-userns-restrict \
  /etc/apparmor.d/bwrap-userns-restrict
sudo apparmor_parser -r /etc/apparmor.d/bwrap-userns-restrict
```

`apparmor_parser -r` 会在不重启的情况下将配置文件加载到内核中。您还可以重新加载所有 AppArmor 配置文件：

```
sudo systemctl reload apparmor.service
```

如果该配置文件不可用或无法解决问题，您可以通过以下命令禁用 AppArmor 无特权用户命名空间限制：

```
sudo sysctl -w kernel.apparmor_restrict_unprivileged_userns=0
```

## 如何控制它

大多数人从产品中的权限控制开始。

在 Codex 应用程序和 IDE 中，您可以从作曲器或聊天输入下方的权限选择器中选择一种模式。该选择器允许您依赖 Codex 的默认权限、切换到完全访问或使用自定义配置。

![Codex 应用程序权限选择器显示默认权限、完全访问和自定义 (config.toml)](https://developers.openai.com/images/codex/app/permissions-selector-light.webp)

在 CLI 中，使用 [`/permissions`](https://developers.openai.com/codex/cli/slash-commands#update-permissions-with-permissions) 在会话期间切换模式。

## 配置默认值

如果您希望 Codex 每次都以相同的行为开始，请使用自定义配置。Codex 将这些默认值存储在 `config.toml` 中，即其本地设置文件。[配置基础](https://developers.openai.com/codex/config-basic) 解释了它的工作原理，而 [配置参考](https://developers.openai.com/codex/config-reference) 记录了 `sandbox_mode`、`approval_policy` 和 `sandbox_workspace_write.writable_roots` 的确切键。使用这些设置来决定 Codex 默认获得多少自主权、可以写入哪些目录，以及何时应该暂停以待批准。

从高层次看，常见的沙盒模式有：

- `read-only`：Codex 可以检查文件，但不能编辑文件或在未获得批准的情况下运行命令。
- `workspace-write`：Codex 可以读取文件，在工作区内编辑，并在该边界内运行例行本地命令。这是本地工作的默认低摩擦模式。
- `danger-full-access`：Codex 无沙盒限制地运行。这消除了文件系统和网络边界，应该仅在您希望 Codex 拥有完全访问权限时使用。

常见的批准政策有：

- `untrusted`：Codex 在运行不在其受信集中的命令之前会询问。
- `on-request`：Codex 默认在沙盒内工作，并在需要超越该边界时询问。
- `never`：Codex 不会停下进行批准提示。

完全访问意味着使用 `sandbox_mode = "danger-full-access"` 结合 `approval_policy = "never"`。相对而言，`--full-auto` 是较低风险的本地自动化预设：`sandbox_mode = "workspace-write"` 和 `approval_policy = "on-request"`。

如果您需要 Codex 在多个目录之间工作，可写根允许您扩展它可以修改的地方，而不必完全移除沙盒。如果您需要更广泛或更狭窄的信任边界，请调整默认的沙盒模式和批准政策，而不是依赖一次性的例外。

对于可重用的权限集，将 `default_permissions` 设置为命名配置文件，并定义 `[permissions.<name>.filesystem]` 或 `[permissions.<name>.network]`。托管网络配置文件使用映射表，例如 `[permissions.<name>.network.domains]` 和 `[permissions.<name>.network.unix_sockets]`，用于域和套接字规则。文件系统配置文件还可以通过将匹配项设置为 `"none"` 来拒绝特定路径或通配符的读取；使用此方法可以保持本地机密文件的不可读性，而无需关闭工作区写入。

当工作流需要特定例外时，可以使用 [规则](https://developers.openai.com/codex/rules)。规则允许您在沙盒外允许、提示或禁止命令前缀，这通常比广泛扩展访问更合适。有关在应用程序中批准和沙盒行为的更高层次概述，请参见 [Codex 应用程序功能](https://developers.openai.com/codex/app/features#approvals-and-sandboxing)，有关 IDE 特定的设置入口，请参见 [Codex IDE 扩展设置](https://developers.openai.com/codex/ide/settings)。

可用时，自动审核不会改变沙盒边界。它审核批准请求，例如沙盒升级或网络访问，同时已在沙盒内允许的操作运行时无需额外审核。有关政策行为，请参见 [自动批准审核](https://developers.openai.com/codex/agent-approvals-security#automatic-approval-reviews)。

平台详细信息位于平台特定文档中。有关原生 Windows 设置、行为和故障排除，请参见 [Windows](https://developers.openai.com/codex/windows)。有关沙盒和批准的管理员要求和组织级限制，请参见 [代理批准与安全](https://developers.openai.com/codex/agent-approvals-security)。
