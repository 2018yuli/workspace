## 配置文件

配置文件允许您保存命名的配置值集，并从 CLI 切换它们。

配置文件处于实验状态，可能会在未来的版本中更改或删除。

目前不支持在 Codex IDE 扩展中使用配置文件。

在 `config.toml` 中定义配置文件`[profiles.<name>]`，然后运行 `codex --profile <name>`：

```
model = "gpt-5.4"
approval_policy = "on-request"
model_catalog_json = "/Users/me/.codex/model-catalogs/default.json"

[profiles.deep-review]
model = "gpt-5-pro"
model_reasoning_effort = "high"
approval_policy = "never"
model_catalog_json = "/Users/me/.codex/model-catalogs/deep-review.json"

[profiles.lightweight]
model = "gpt-4.1"
approval_policy = "untrusted"
```

要将配置文件设为默认值，请在 `config.toml` 的顶层添加 `profile = "deep-review"`。Codex 将加载该配置文件，除非您在命令行中覆盖。

配置文件还可以覆盖 `model_catalog_json`。当顶层和所选配置文件都设置 `model_catalog_json` 时，Codex 优先使用配置文件的值。

## CLI 的一次性覆盖

除编辑 `~/.codex/config.toml` 外，您还可以从 CLI 覆盖单次运行的配置：

- 在存在时优先使用专用标志（例如 `--model`）。
- 当需要覆盖任意键时，请使用 `-c` / `--config`。

示例：

```
# 专用标志
codex --model gpt-5.4

# 通用键/值覆盖（值为 TOML，而不是 JSON）
codex --config model='"gpt-5.4"'
codex --config sandbox_workspace_write.network_access=true
codex --config 'shell_environment_policy.include_only=["PATH","HOME"]'
```

注意：

- 键可以使用点符号来设置嵌套值（例如，`mcp_servers.context7.enabled=false`）。
- `--config` 值被解析为 TOML。对于疑惑的值，请用引号括起来，以免您的 shell 按空格分割。
- 如果值无法被解析为 TOML，Codex 将其视为字符串。

## 配置与状态位置

Codex 将其本地状态存储在 `CODEX_HOME` 下（默认为 `~/.codex`）。

您可能会看到的常见文件：

- `config.toml`（您的本地配置）
- `auth.json`（如果您使用基于文件的凭据存储）或您的操作系统密钥链/密钥环
- `history.jsonl`（如果启用了历史持久性）
- 其他每用户状态，例如日志和缓存

有关身份验证详细信息（包括凭据存储模式），请参见 [身份验证](https://developers.openai.com/codex/auth)。有关完整的配置密钥列表，请参见 [配置参考](https://developers.openai.com/codex/config-reference)。

有关已检查到仓库或系统路径中的共享默认值、规则和技能，请参见 [团队配置](https://developers.openai.com/codex/enterprise/admin-setup#team-config)。

如果您只需将内置的 OpenAI 提供者指向 LLM 代理、路由器或数据驻留启用项目，请在 `config.toml` 中设置 `openai_base_url`，而不是定义新的提供者。这将更改内置 `openai` 提供者的基本 URL，而无需单独的 `model_providers.<id>` 条目。

```
openai_base_url = "https://us.api.openai.com/v1"
```

## 项目配置文件 (`.codex/config.toml`)

除了用户配置外，Codex 还会读取项目范围的覆盖 `.codex/config.toml` 文件，位于您的仓库中。Codex 从项目根目录向您当前的工作目录遍历，并加载它发现的每一个 `.codex/config.toml` 文件。如果多个文件定义了相同的键，则离您工作目录最近的文件获胜。

出于安全原因，仅当项目被信任时，Codex 才会加载项目范围的配置文件。如果项目不被信任，Codex 将忽略项目的 `.codex/` 层，包括 `.codex/config.toml`、项目本地钩子和项目本地规则。用户层和系统层仍然保持独立并继续加载。

项目配置中的相对路径（例如，`model_instructions_file`）相对于包含 `config.toml` 的 `.codex/` 文件夹进行解析。

## 钩子（实验性）

Codex 还可以从 `hooks.json` 文件或在活动配置层旁边的 `config.toml` 文件中的嵌入 `[hooks]` 表加载生命周期钩子。

在实践中，两个最有用的位置是：

- `~/.codex/hooks.json`
- `~/.codex/config.toml`
- `<repo>/.codex/hooks.json`
- `<repo>/.codex/config.toml`

项目本地钩子仅在项目的 `.codex/` 层被信任时加载。用户级钩子与项目信任无关。

通过以下命令打开钩子：

```
[features]
codex_hooks = true
```

嵌入的 TOML 钩子与 `hooks.json` 使用相同的事件结构：

```
[[hooks.PreToolUse]]
matcher = "^Bash$"

[[hooks.PreToolUse.hooks]]
type = "command"
command = '/usr/bin/python3 "$(git rev-parse --show-toplevel)/.codex/hooks/pre_tool_use_policy.py"'
timeout = 30
statusMessage = "检查 Bash 命令"
```

如果单个层包含 `hooks.json` 和嵌入的 `[hooks]`，Codex 将加载两者并发出警告。每层优先选择一种表示。

有关当前事件列表、输入字段、输出行为和限制，请参见 [钩子](https://developers.openai.com/codex/hooks)。

## 项目根目录检测

Codex 通过从工作目录向上遍历，直到到达项目根目录，来发现项目配置（例如，`.codex/` 层和 `AGENTS.md`）。

默认情况下，Codex 将包含 `.git` 的目录视为项目根目录。要自定义此行为，请在 `config.toml` 中设置 `project_root_markers`：

```
# 当目录包含这些标记中的任何一个时，将其视为项目根目录。
project_root_markers = [".git", ".hg", ".sl"]
```

设置 `project_root_markers = []` 以跳过搜索父目录，并将当前工作目录视为项目根目录。

## 自定义模型提供者

模型提供者定义 Codex 如何连接到模型（基本 URL、线缆 API、身份验证和可选的 HTTP 头）。自定义提供者不能重用保留的内置提供者 ID：`openai`、`ollama` 和 `lmstudio`。

定义其他提供者并将 `model_provider` 指向它们：

```
model = "gpt-5.4"
model_provider = "proxy"

[model_providers.proxy]
name = "OpenAI 使用 LLM 代理"
base_url = "http://proxy.example.com"
env_key = "OPENAI_API_KEY"

[model_providers.local_ollama]
name = "Ollama"
base_url = "http://localhost:11434/v1"

[model_providers.mistral]
name = "Mistral"
base_url = "https://api.mistral.ai/v1"
env_key = "MISTRAL_API_KEY"
```

在需要时添加请求头：

```
[model_providers.example]
http_headers = { "X-Example-Header" = "example-value" }
env_http_headers = { "X-Example-Features" = "EXAMPLE_FEATURES" }
```

当提供者需要 Codex 从外部凭据助手提取 bearer 令牌时使用命令后台身份验证：

```
[model_providers.proxy]
name = "OpenAI 使用 LLM 代理"
base_url = "https://proxy.example.com/v1"
wire_api = "responses"

[model_providers.proxy.auth]
command = "/usr/local/bin/fetch-codex-token"
args = ["--audience", "codex"]
timeout_ms = 5000
refresh_interval_ms = 300000
```

身份验证命令不接受 `stdin`，必须将令牌打印到 `stdout`。Codex 剪切周围的空格，将空令牌视为错误，并在 `refresh_interval_ms` 之前主动刷新；设置 `refresh_interval_ms = 0` 以便在身份验证重试后刷新。不要将 `[model_providers.<id>.auth]` 与 `env_key`、`experimental_bearer_token` 或 `requires_openai_auth` 结合使用。

## OSS 模式（本地提供者）

Codex 可以在传递 `--oss` 时运行于本地“开源”提供者（例如，Ollama 或 LM Studio）。如果您在未指定提供者的情况下传递 `--oss`，Codex 使用 `oss_provider` 作为默认值。

```
# 默认的本地提供者与 `--oss` 一起使用
oss_provider = "ollama" # 或 "lmstudio"
```

## Azure 提供者和每个提供者的调优

```
[model_providers.azure]
name = "Azure"
base_url = "https://YOUR_PROJECT_NAME.openai.azure.com/openai"
env_key = "AZURE_OPENAI_API_KEY"
query_params = { api-version = "2025-04-01-preview" }
wire_api = "responses"
request_max_retries = 4
stream_max_retries = 10
stream_idle_timeout_ms = 300000
```

要更改内置 OpenAI 提供者的基本 URL，请使用 `openai_base_url`；不要创建 `[model_providers.openai]`，因为您无法覆盖内置提供者 ID。

## 使用数据驻留的 ChatGPT 客户

启用 [数据驻留](https://help.openai.com/en/articles/9903489-data-residency-and-inference-residency-for-chatgpt) 的项目可以创建一个模型提供者，以使用 [正确的前缀](https://platform.openai.com/docs/guides/your-data#which-models-and-features-are-eligible-for-data-residency) 更新 `base_url`。

```
model_provider = "openaidr"
[model_providers.openaidr]
name = "OpenAI 数据驻留"
base_url = "https://us.api.openai.com/v1" # 将 'us' 替换为域前缀
```

## 模型推理、冗长性和限制

```
model_reasoning_summary = "none"          # 禁用总结
```

## 审批策略与沙盒模式

选择审批严格性（影响 Codex 暂停的时机）和沙盒级别（影响文件/网络访问）。

有关编辑 `config.toml` 时需要注意的操作细节，请参见 [常见沙盒和审批组合](https://developers.openai.com/codex/agent-approvals-security#common-sandbox-and-approval-combinations)、[可写根中的受保护路径](https://developers.openai.com/codex/agent-approvals-security#protected-paths-in-writable-roots) 和 [网络访问](https://developers.openai.com/codex/agent-approvals-security#network-access)。

您还可以使用更细粒度的审批策略（`approval_policy = { granular = { ... } }`）来允许或自动拒绝单独的提示类别。这在您希望某些情况进行正常的交互式审批，但希望其他情况（例如 `request_permissions` 或技能脚本提示）自动失败时很有用。

设置 `approvals_reviewer = "auto_review"` 以通过自动审查为合格的交互式审批请求提供路由。这更改审批人，而不是沙盒边界。

使用 `[auto_review].policy` 进行本地审核员政策说明。托管的 `guardian_policy_config` 优先。

```
approval_policy = "untrusted"   # 其他选项：on-request，never 或 { granular = { ... } }
approvals_reviewer = "user"     # 或 "auto_review" 用于自动审查
sandbox_mode = "workspace-write"
allow_login_shell = false       # 可选加固：不允许 shell 工具的登录 shell

# 示例细粒度审批策略：
# approval_policy = { granular = {
#   sandbox_approval = true,
#   rules = true,
#   mcp_elicitations = true,
#   request_permissions = false,
#   skill_approval = false
# } }

[sandbox_workspace_write]
exclude_tmpdir_env_var = false  # 允许 $TMPDIR
exclude_slash_tmp = false       # 允许 /tmp
writable_roots = ["/Users/YOU/.pyenv/shims"]
network_access = false          # 选择外部网络

[auto_review]
policy = """
使用您组织的自动审查策略。
"""
```

需要完整的密钥列表（包括配置文件作用域覆盖和要求约束）吗？请参见 [配置参考](https://developers.openai.com/codex/config-reference) 和 [托管配置](https://developers.openai.com/codex/enterprise/managed-configuration)。

在工作区写入模式下，某些环境即使在其余工作区可写时，也会保持 `.git/` 和 `.codex/` 为只读。这就是为什么像 `git commit` 这样的命令可能仍然需要审批才能在沙盒外运行。如果要 Codex 跳过特定命令（例如，阻止 `git commit` 在沙盒外部），请使用 [规则](https://developers.openai.com/codex/rules)。

完全禁用沙盒（仅在您的环境已经隔离进程时使用）：

```
sandbox_mode = "danger-full-access"
```

## shell 环境策略

`shell_environment_policy` 控制 Codex 传递给它启动的任何子进程的环境变量（例如，在运行模型提议的工具命令时）。可从干净开始（`inherit = "none"`）或修整的集合（`inherit = "core"`）开始，然后加上排除、包含和覆盖，以避免泄漏机密，同时仍能提供任务所需的路径、密钥或标志。

```
[shell_environment_policy]
inherit = "none"
set = { PATH = "/usr/bin", MY_FLAG = "1" }
ignore_default_excludes = false
exclude = ["AWS_*", "AZURE_*"]
include_only = ["PATH", "HOME"]
```

## 可观察性和遥测

启用 OpenTelemetry（OTel）日志导出以跟踪 Codex 运行（API 请求、SSE/事件、提示、工具审批/结果）。默认情况下禁用；通过 `[otel]` 选择加入：

```
[otel]
environment = "staging"   # 默认值为 "dev"
exporter = "none"         # 设置为 otlp-http 或 otlp-grpc 以发送事件
log_user_prompt = false   # 默认禁用用户提示，除非显式启用
```

选择导出器：

```
[otel]
exporter = { otlp-http = {
  endpoint = "https://otel.example.com/v1/logs",
  protocol = "binary",
  headers = { "x-otlp-api-key" = "${OTLP_TOKEN}" }
}}
```

```
[otel]
exporter = { otlp-grpc = {
  endpoint = "https://otel.example.com:4317",
  headers = { "x-otlp-meta" = "abc123" }
}}
```

如果 `exporter = "none"`，Codex 会记录事件但不发送任何内容。导出器异步批处理并在关闭时刷新。事件元数据包括服务名称、CLI 版本、环境标签、会话 ID、模型、沙盒/审批设置以及每事件字段（见 [配置参考](https://developers.openai.com/codex/config-reference)）。

### 会被发出的内容

Codex 为运行和工具使用发出结构化日志事件。典型的事件类型包括：

- `codex.conversation_starts`（模型、推理设置、沙盒/审批策略）
- `codex.api_request`（尝试、状态/成功、持续时间和错误详情）
- `codex.sse_event`（流事件类型、成功/失败、持续时间，以及 `response.completed` 中的令牌计数）
- `codex.websocket_request` 和 `codex.websocket_event`（请求持续时间以及每条消息的类型/成功/错误）
- `codex.user_prompt`（长度；内容在未显式启用时进行编辑）
- `codex.tool_decision`（批准/拒绝以及决定是否来自配置或用户）
- `codex.tool_result`（持续时间、成功、输出片段）

### 指标

### 反馈控制


禁用后，`/feedback` 将显示禁用消息，Codex 将拒绝反馈提交。

### 隐藏或显示推理事件

如果您希望减少嘈杂的“推理”输出（例如在 CI 日志中），可以抑制它：

```
hide_agent_reasoning = true
```

如果您希望在模型输出时显示原始推理内容：

```
show_raw_agent_reasoning = true
```

仅在接受您的工作流程时启用原始推理。一些模型/提供者（如 `gpt-oss`）不会发出原始推理；在这种情况下，此设置不会产生可见效果。

## 通知

使用 `notify` 在 Codex 发出支持的事件时触发外部程序（目前仅支持 `agent-turn-complete`）。这对于桌面通知、聊天 Webhook、CI 更新或任何内置 TUI 通知无法覆盖的侧面警报非常方便。

```
notify = ["python3", "/path/to/notify.py"]
```

示例 `notify.py`（截断）响应 `agent-turn-complete`：

```
#!/usr/bin/env python3
import json, subprocess, sys

def main() -> int:
    notification = json.loads(sys.argv[1])
    if notification.get("type") != "agent-turn-complete":
        return 0
    title = f"Codex: {notification.get('last-assistant-message', '轮次完成！')}"
    message = " ".join(notification.get("input-messages", []))
    subprocess.check_output([\
        "terminal-notifier",\
        "-title", title,\
        "-message", message,\
        "-group", "codex-" + notification.get("thread-id", ""),\
        "-activate", "com.googlecode.iterm2",\
    ])
    return 0

if __name__ == "__main__":
    sys.exit(main())
```

该脚本接受单个 JSON 参数。常见字段包括：

- `type`（当前为 `agent-turn-complete`）
- `thread-id`（会话标识符）
- `turn-id`（轮次标识符）
- `cwd`（工作目录）
- `input-messages`（导致该轮次的用户消息）
- `last-assistant-message`（最后的助手消息文本）

将脚本放在磁盘上的某个地方，并将 `notify` 指向它。

#### `notify` 与 `tui.notifications`

- `notify` 运行外部程序（适用于 Webhook、桌面通知、CI 钩子）。
- `tui.notifications` 是内置的 TUI，可以根据事件类型选择性过滤（例如，`agent-turn-complete` 和 `approval-requested`）。
- `tui.notification_method` 控制 TUI 如何发出终端通知（`auto`、`osc9` 或 `bel`）。
- `tui.notification_condition` 控制 TUI 通知是否仅在终端为 `unfocused` 或 `always` 时触发。

在 `auto` 模式下，Codex 更偏好 OSC 9 通知（某些终端将其解释为桌面通知的终端转义序列），否则回退到 BEL（`\x07`）。

请参见 [配置参考](https://developers.openai.com/codex/config-reference) 以获取确切的密钥。

## 历史持久性

默认情况下，Codex 将本地会话记录保存在 `CODEX_HOME` 下（例如 `~/.codex/history.jsonl`）。要禁用本地历史持久性：

```
[history]
persistence = "none"
```

要限制历史文件的大小，请设置 `history.max_bytes`。当文件超出限制时，Codex 将丢弃最旧的条目并压缩文件，同时保留最新记录。

```
[history]
max_bytes = 104857600 # 100 MiB
```

## 可点击引用

如果您使用的终端/编辑器集成支持，Codex 可以将文件引用呈现为可单击的链接。配置 `file_opener` 以选择 Codex 使用的 URI 方案：

```
file_opener = "vscode" # 或 cursor, windsurf, vscode-insiders, none
```

示例：像 `/home/user/project/main.py:42` 这样的引用可以重写为一个可单击的 `vscode://file/...:42` 链接。

## 项目指令发现

Codex 会读取 `AGENTS.md`（及相关文件），并在会话的第一轮中包含有限的项目指导。有两个旋钮控制此工作：

- `project_doc_max_bytes`：要读取的每个 `AGENTS.md` 文件的大小
- `project_doc_fallback_filenames`：当在目录级别缺少 `AGENTS.md` 时尝试的其他文件名

有关详细的演练，请参见 [使用 AGENTS.md 进行自定义指令](https://developers.openai.com/codex/guides/agents-md)。

## TUI 选项

运行 `codex` 而没有子命令会启动互动终端用户界面（TUI）。Codex 在 `[tui]` 下公开一些针对 TUI 的配置，包括：

- `tui.notifications`：启用/禁用通知（或限制到特定类型）
- `tui.notification_method`：选择 `auto`、`osc9` 或 `bel` 以进行终端通知
- `tui.notification_condition`：选择 `unfocused` 或 `always` 以确定何时触发通知
- `tui.animations`：启用/禁用 ASCII 动画和闪烁效果
- `tui.alternate_screen`：控制备用屏幕使用情况（设置为 `never` 以保持终端回滚）
- `tui.show_tooltips`：在欢迎屏幕上显示或隐藏入门工具提示

`tui.notification_method` 默认为 `auto`。在 `auto` 模式下，Codex 更偏好 OSC 9 通知（某些终端将其解释为桌面通知的终端转义序列），当终端看起来支持时，将其作为首选，如果不支持则回退到 BEL（`\x07`）。

请参见 [配置参考](https://developers.openai.com/codex/config-reference) 以获取完整的密钥列表。
