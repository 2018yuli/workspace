### 构建您自己的精选插件列表

市场是插件的 JSON 目录。`$plugin-creator` 可以为单个插件生成一个，
您可以继续向该市场添加条目，以为仓库、团队或个人工作流构建自己的精选列表。

在 Codex 中，每个市场显示为插件目录中的可选择来源。使用 `$REPO_ROOT/.agents/plugins/marketplace.json` 作为仓库范围的列表，或使用 `~/.agents/plugins/marketplace.json` 作为个人列表。在 `plugins[]` 下为每个插件添加一个条目，指向插件文件夹的 `source.path` 设置为带有 `./` 前缀相对路径，并将 `interface.displayName` 设置为希望 Codex 在市场选择器中显示的标签。然后重启 Codex。之后，打开插件目录，选择您的市场，并浏览或安装该精选列表中的插件。

您不需要为每个插件单独创建市场。一个市场可以在您测试时公开单个插件，然后随着您添加更多插件而增长为更大的精选目录。

![插件目录中的自定义本地市场](https://developers.openai.com/images/codex/plugins/codex-local-plugin-light.png)![插件目录中的自定义本地市场](https://developers.openai.com/images/codex/plugins/codex-local-plugin.png)

### 从 CLI 添加市场

当您希望 Codex 为您安装和跟踪市场来源而不是手动编辑 `config.toml` 时，使用 `codex plugin marketplace add`。

```
codex plugin marketplace add owner/repo
codex plugin marketplace add owner/repo --ref main
codex plugin marketplace add https://github.com/example/plugins.git --sparse .agents/plugins
codex plugin marketplace add ./local-marketplace-root
```

市场来源可以是 GitHub 简码（`owner/repo` 或 `owner/repo@ref`），HTTP 或 HTTPS Git URL，SSH Git URL，或本地市场根目录。使用 `--ref` 钉住 Git 引用，重复 `--sparse PATH` 以对 Git 支持的市场仓库使用稀疏检出。`--sparse` 仅对 Git 市场来源有效。

要刷新或删除已配置的市场：

```
codex plugin marketplace upgrade
codex plugin marketplace upgrade marketplace-name
codex plugin marketplace remove marketplace-name
```

### 手动创建插件

从打包一个技能的最小插件开始。

1. 创建一个插件文件夹并在 `.codex-plugin/plugin.json` 中放置清单。

```
mkdir -p my-first-plugin/.codex-plugin
```

`my-first-plugin/.codex-plugin/plugin.json`

```
{
  "name": "my-first-plugin",
  "version": "1.0.0",
  "description": "可重用的问候工作流",
  "skills": "./skills/"
}
```

使用稳定插件 `name` 采用 kebab-case 格式。Codex 将其用作插件
标识符和组件命名空间。

2. 在 `skills/<skill-name>/SKILL.md` 下添加一个技能。

```
mkdir -p my-first-plugin/skills/hello
```

`my-first-plugin/skills/hello/SKILL.md`

```
---
name: hello
description: 用友好的信息问候用户。
---

热情地问候用户，并询问如何帮助您。
```

3. 将插件添加到市场中。使用 `$plugin-creator` 生成一个，或
按照 [构建您自己的精选插件列表](https://developers.openai.com/codex/plugins/build#build-your-own-curated-plugin-list)
手动将插件接入 Codex。

从那里，您可以根据需要添加 MCP 配置、应用集成或市场元数据。

### 手动安装本地插件

根据谁可以访问插件或精选列表选择使用仓库市场或个人市场。

选择一个选项

仓库个人

在 `$REPO_ROOT/.agents/plugins/marketplace.json` 中添加一个市场文件
并将插件存储在 `$REPO_ROOT/plugins/` 下。

**仓库市场示例**

步骤 1：将插件文件夹复制到 `$REPO_ROOT/plugins/my-plugin`。

```
mkdir -p ./plugins
cp -R /absolute/path/to/my-plugin ./plugins/my-plugin
```

步骤 2：添加或更新 `$REPO_ROOT/.agents/plugins/marketplace.json`，使得 `source.path` 指向该插件目录，使用前缀为 `./` 的相对路径：

```
{
  "name": "local-repo",
  "plugins": [\
    {\
      "name": "my-plugin",\
      "source": {\
        "source": "local",\
        "path": "./plugins/my-plugin"\
      },\
      "policy": {\
        "installation": "AVAILABLE",\
        "authentication": "ON_INSTALL"\
      },\
      "category": "Productivity"\
    }\
  ]
}
```

步骤 3：重启 Codex 并验证插件是否出现。

在 `~/.agents/plugins/marketplace.json` 中添加一个市场文件，并将您的插件存储在 `~/.codex/plugins/` 下。

**个人市场示例**

步骤 1：将插件文件夹复制到 `~/.codex/plugins/my-plugin`。

```
mkdir -p ~/.codex/plugins
cp -R /absolute/path/to/my-plugin ~/.codex/plugins/my-plugin
```

步骤 2：添加或更新 `~/.agents/plugins/marketplace.json`，使得插件条目的 `source.path` 指向该目录。

步骤 3：重启 Codex 并验证插件是否出现。

市场文件指向插件位置，因此这些目录是示例，而非固定要求。Codex 解析 `source.path` 是相对于市场根目录，而非相对于 `.agents/plugins/` 文件夹。请参见 [市场元数据](https://developers.openai.com/codex/plugins/build#marketplace-metadata) 以获取文件格式。

在您更改插件后，更新市场条目中指向的插件目录并重启 Codex，以便本地安装获取新文件。

### 市场元数据

如果您维护一个仓库市场，将其定义为
`$REPO_ROOT/.agents/plugins/marketplace.json`。对于个人市场，使用
`~/.agents/plugins/marketplace.json`。一个市场文件控制插件排序和其在 Codex 界面中的安装策略。它可以代表一个插件，在您测试时或是您希望 Codex 一起显示的插件的精选列表。在将插件添加到市场之前，请确保其 `version`、发布者元数据和安装表面副本已准备好供其他开发者查看。

```
{
  "name": "local-example-plugins",
  "interface": {
    "displayName": "本地示例插件"
  },
  "plugins": [\
    {\
      "name": "my-plugin",\
      "source": {\
        "source": "local",\
        "path": "./plugins/my-plugin"\
      },\
      "policy": {\
        "installation": "AVAILABLE",\
        "authentication": "ON_INSTALL"\
      },\
      "category": "Productivity"\
    },\
    {\
      "name": "research-helper",\
      "source": {\
        "source": "local",\
        "path": "./plugins/research-helper"\
      },\
      "policy": {\
        "installation": "AVAILABLE",\
        "authentication": "ON_INSTALL"\
      },\
      "category": "Productivity"\
    }\
  ]
}
```

- 使用顶级 `name` 标识市场。
- 使用 `interface.displayName` 作为在 Codex 中显示的市场标题。
- 在 `plugins` 下为每个插件添加一个对象，以构建一个 Codex 在该市场标题下展示的精选列表。
- 将每个插件条目的 `source.path` 指向您希望 Codex 加载的插件目录。对于仓库安装，通常位于 `./plugins/` 下。对于个人安装，常见模式是 `./.codex/plugins/<plugin-name>`。
- 保持 `source.path` 相对于市场根目录，以 `./` 开头，并保持在该根目录内部。
- 对于本地条目，`source` 也可以是一个简单的字符串路径，例如 `"./plugins/my-plugin"`。
- 始终在每个插件条目上包含 `policy.installation`、`policy.authentication` 和 `category`。
- 使用 `policy.installation` 值，例如 `AVAILABLE`、`INSTALLED_BY_DEFAULT` 或 `NOT_AVAILABLE`。
- 使用 `policy.authentication` 决定身份验证是在安装时还是首次使用时进行。

市场控制 Codex 从哪里加载插件。一个本地 `source.path` 可以指向别处，如果您的插件位于其他示例目录之外。市场文件可以处于您正在开发插件的仓库中，或者在一个单独的市场仓库中，一个市场文件可以指向一个插件或多个插件。

市场条目也可以指向基于 Git 的插件来源。当插件存在于仓库根目录时，使用 `"source": "url"`，而当插件存在于子目录时，使用 `"source": "git-subdir"`：

```
{
  "name": "remote-helper",
  "source": {
    "source": "git-subdir",
    "url": "https://github.com/example/codex-plugins.git",
    "path": "./plugins/remote-helper",
    "ref": "main"
  },
  "policy": {
    "installation": "AVAILABLE",
    "authentication": "ON_INSTALL"
  },
  "category": "Productivity"
}
```

基于 Git 的条目可能使用 `ref` 或 `sha` 选择器。如果 Codex 无法解析市场条目的来源，它会跳过该插件条目，而不是导致整个市场失败。

### Codex 如何使用市场

插件市场是 Codex 可以读取和安装的插件的 JSON 目录。

Codex 可以读取市场文件来自：

- 提供官方插件目录的精选市场
- 位于 `$REPO_ROOT/.agents/plugins/marketplace.json` 的仓库市场
- 位于 `$REPO_ROOT/.claude-plugin/marketplace.json` 的 Claude 风格市场
- 位于 `~/.agents/plugins/marketplace.json` 的个人市场

您可以安装通过市场公开的任何插件。Codex 安装插件到
`~/.codex/plugins/cache/$MARKETPLACE_NAME/$PLUGIN_NAME/$VERSION/`。对于本地
插件，`$VERSION` 为 `local`，Codex 从该缓存路径加载已安装的副本，而不是直接从市场条目中加载。

您可以单独启用或禁用每个插件。Codex 将每个插件的开启或关闭状态存储在 `~/.codex/config.toml` 中。

## 打包和分发插件

### 插件结构

每个插件在 `.codex-plugin/plugin.json` 中都有一个清单。它还可以包括
一个 `skills/` 目录，一个 `.app.json` 文件，指向一个或多个应用或连接器，一个 `.mcp.json` 文件，配置 MCP 服务器，以及用于在支持的表面上展示插件的资源。

- my-plugin/
  
  - .codex-plugin/
    
    - plugin.json  必需：插件清单
  
  - skills/
  
    - my-skill/
  
      - SKILL.md  可选：技能说明
  
  - .app.json  可选：应用或连接器映射
  
  - .mcp.json  可选：MCP 服务器配置
  
  - assets/  可选：图标、徽标、截图

只有 `plugin.json` 属于 `.codex-plugin/`。将 `skills/`、`assets/`、
`.mcp.json` 和 `.app.json` 保持在插件根目录下。

发布的插件通常使用比快速开始脚手架中出现的最小示例更丰富的清单。清单有三个功能：

- 标识插件。
- 指向捆绑组件，如技能、应用或 MCP 服务器。
- 提供安装表面元数据，如描述、图标和法律链接。

以下是一个完整的清单示例：

```
{
  "name": "my-plugin",
  "version": "0.1.0",
  "description": "捆绑可重用的技能和应用集成。",
  "author": {
    "name": "您的团队",
    "email": "team@example.com",
    "url": "https://example.com"
  },
  "homepage": "https://example.com/plugins/my-plugin",
  "repository": "https://github.com/example/my-plugin",
  "license": "MIT",
  "keywords": ["research", "crm"],
  "skills": "./skills/",
  "mcpServers": "./.mcp.json",
  "apps": "./.app.json",
  "interface": {
    "displayName": "我的插件",
    "shortDescription": "可重用的技能和应用",
    "longDescription": "一起分发技能和应用集成。",
    "developerName": "您的团队",
    "category": "Productivity",
    "capabilities": ["Read", "Write"],
    "websiteURL": "https://example.com",
    "privacyPolicyURL": "https://example.com/privacy",
    "termsOfServiceURL": "https://example.com/terms",
    "defaultPrompt": [\
      "使用我的插件总结新 CRM 备注。",\
      "使用我的插件处理新的客户跟进。\
    ],
    "brandColor": "#10A37F",
    "composerIcon": "./assets/icon.png",
    "logo": "./assets/logo.png",
    "screenshots": ["./assets/screenshot-1.png"]
  }
}
```

`.codex-plugin/plugin.json` 是必需的入口。其他清单字段是可选的，但发布的插件通常会使用它们。

### 清单字段

使用顶级字段定义包元数据并指向捆绑组件：

- `name`、`version` 和 `description` 标识插件。
- `author`、`homepage`、`repository`、`license` 和 `keywords` 提供发布者和发现元数据。
- `skills`、`mcpServers` 和 `apps` 相对于插件根目录指向捆绑组件。
- `interface` 控制安装表面如何展示插件。

使用 `interface` 对象提供安装表面元数据：

- `displayName`、`shortDescription` 和 `longDescription` 控制标题和描述性副本。
- `developerName`、`category` 和 `capabilities` 添加发布者和能力元数据。
- `websiteURL`、`privacyPolicyURL` 和 `termsOfServiceURL` 提供外部链接。
- `defaultPrompt`、`brandColor`、`composerIcon`、`logo` 和 `screenshots` 控制启动提示和视觉呈现。

### 路径规则

- 将清单路径相对于插件根目录，并以 `./` 开头。
- 尽可能将视觉资源（如 `composerIcon`、`logo` 和 `screenshots`）存放在 `./assets/` 下。
- 使用 `skills` 表示捆绑的技能文件夹，使用 `apps` 表示 `.app.json`，并使用 `mcpServers` 表示 `.mcp.json`。

### 发布官方公共插件

将插件加入官方插件目录的功能即将推出。

自助插件发布和管理功能即将推出。
