定制化是指如何让 Codex 按照你的团队的工作方式运行。

在 Codex 中，定制化来自几个相互协作的层次：

- **项目指导（`AGENTS.md`）** 用于持久性指令
- **[记忆](https://developers.openai.com/codex/memories)** 用于从过去工作中学习的有用上下文
- **技能** 用于可重用的工作流程和领域专业知识
- **[MCP](https://developers.openai.com/codex/mcp)** 用于访问外部工具和共享系统
- **[子代理](https://developers.openai.com/codex/concepts/subagents)** 用于将工作委派给专业子代理

这些是互补的，而不是竞争的。`AGENTS.md` 形状行为，记忆传递本地上下文，技能打包可重复的过程，而 [MCP](https://developers.openai.com/codex/mcp) 将 Codex 连接到本地工作区之外的系统。

## AGENTS 指导

`AGENTS.md` 为 Codex 提供持久的项目指导，这些指导与您的代码库一起移动并在代理开始工作之前应用。保持简短。

将其用于您希望 Codex 每次在代码库中遵循的规则，例如：

- 构建和测试命令
- 评审期望
- 特定于代码库的约定
- 特定于目录的指令

当代理对您的代码库做出错误假设时，请在 `AGENTS.md` 中纠正它们，并要求代理更新 `AGENTS.md` 以便修复永久生效。将其视为反馈循环。

**更新 `AGENTS.md`：** 只需从重要的指令开始。将重复的评审反馈进行编纂，将指导置于应用的最近目录，并在您纠正某些内容时告知代理更新 `AGENTS.md`，让未来的会话继承修复。

### 何时更新 `AGENTS.md`

- **重复错误**：如果代理反复犯同样的错误，请添加一条规则。
- **阅读过多**：如果它找到正确的文件但阅读了太多文档，请添加路由指导（优先考虑哪些目录/文件）。
- **重复的 PR 反馈**：如果您多次留下相同的反馈，请将其编纂。
- **在 GitHub 中**：在拉取请求评论中，标记 `@codex` 并请求（例如，`@codex add this to AGENTS.md`）将更新委派到云任务。
- **自动化漂移检查**：使用 [自动化](https://developers.openai.com/codex/app/automations) 运行重复检查（例如，每日）以寻找指导空白，并建议添加到 `AGENTS.md`。

将 `AGENTS.md` 结合执行这些规则的基础设施：预提交钩子、代码检查工具和类型检查器在您看到问题之前捕获问题，因此系统在防止重复错误方面变得更聪明。

Codex 可以从多个位置加载指导：您在 Codex 主目录中的全局文件（供您作为开发人员使用）和团队可以签入的特定于代码库的文件。靠近工作目录的文件优先。

- ~/.codex/

  - AGENTS.md 全局（供您作为开发人员使用）

- repo-root/

  - AGENTS.md 特定于代码库（供您的团队使用）

[使用 AGENTS.md 的自定义指令](https://developers.openai.com/codex/guides/agents-md)

## 技能

技能为 Codex 提供可重用的能力以实现可重复的工作流程。技能通常是可重用工作流程的最佳选择，因为它们支持更丰富的指令、脚本和引用，同时保持跨任务的可重用性。技能被加载并对代理可见（至少它们的元数据），因此 Codex 可以隐式发现和选择它们。这使得丰富的工作流程可用，而不会提前膨胀上下文。

使用技能文件夹在本地撰写和迭代工作流程。如果该工作流程已经存在插件，则先安装它以重用经过验证的设置。当您希望跨团队分发自己的工作流程或将其与应用程序集成捆绑时，将其打包为 [插件](https://developers.openai.com/codex/plugins/build)。技能仍然是创作格式；插件是可安装的分发单元。

技能通常是一个 `SKILL.md` 文件以及可选的脚本、引用和资产。

- my-skill/

  - SKILL.md 必需：指令 + 元数据

  - scripts/ 可选：可执行代码

  - references/ 可选：文档

  - assets/ 可选：模板、资源

技能目录可以包括一个可被 Codex 在工作流程中调用为 part 的 CLI 脚本的 `scripts/` 文件夹（例如，种子数据或运行验证）。当工作流程需要外部系统（问题跟踪器、设计工具、文档服务器）时，将技能与 [MCP](https://developers.openai.com/codex/mcp) 结合。

示例 `SKILL.md`：

```
---
name: commit
description: 按语义组的阶段并提交更改。当用户想要提交、组织提交或在推送之前清理分支时使用。
---

1. 不要运行 `git add .`。按目的在逻辑组中分阶段文件。
2. 分为单独的提交：feat → test → docs → refactor → chore。
3. 编写与更改范围匹配的简洁提交信息。
4. 保持每个提交专注且可审阅。
```

使用技能来：

- 可重用的工作流程（发布步骤、评审例程、文档更新）
- 特定于团队的专业知识
- 需要示例、引用或辅助脚本的过程

技能可以是全局（在您的用户目录中，供您作为开发人员使用）或特定于代码库（签入 `.agents/skills`，供您的团队使用）。当工作流程适用于该项目时，请将代码库技能放在 `.agents/skills` 中；对于您希望在所有代码库中使用的技能，请使用您的用户目录。

| 层 | 全局 | 代码库 |
| --- | --- | --- |
| AGENTS | `~/.codex/AGENTS.md` | 代码库根目录或嵌套目录中的 `AGENTS.md` |
| 技能 | `$HOME/.agents/skills` | 代码库中的 `.agents/skills` |

Codex 使用逐步披露的方式来处理技能：

- 开始使用元数据（`name`，`description`）进行发现
- 仅在选择技能时加载 `SKILL.md`
- 仅在需要时读取引用或运行脚本

技能可以被明确调用，Codex 也可以在任务与技能描述匹配时隐式选择它们。清晰的技能描述可以提高触发可靠性。

[代理技能](https://developers.openai.com/codex/skills)

## MCP

MCP（模型上下文协议）是将 Codex 连接到外部工具和上下文提供者的标准方法。它对于远程托管的系统尤为有用，例如 Figma、Linear、GitHub 或您的团队所依赖的内部知识服务。

当 Codex 需要本地代码库之外的功能时（例如问题跟踪器、设计工具、浏览器或共享文档系统），请使用 MCP。

可以这样思考：

- **主机**：Codex
- **客户端**：Codex 内部的 MCP 连接
- **服务器**：外部工具或上下文提供者

MCP 服务器可以公开：

- **工具**（操作）
- **资源**（可读数据）
- **提示**（可重用提示模板）

这种分离有助于您思考信任和能力边界。一些服务器主要提供上下文，而另一些则公开强大的操作。

在实践中，MCP 通常在与技能结合使用时最有用：

- 一项技能定义工作流程并命名要使用的 MCP 工具

[模型上下文协议](https://developers.openai.com/codex/mcp)

## 子代理

您可以创建不同角色的不同代理，并提示它们以不同方式使用工具。例如，一个代理可能运行特定的测试命令和配置，而另一个具有用于调试的获取生产日志的 MCP 服务器。每个子代理保持专注并使用适合其工作的正确工具。

[子代理概念](https://developers.openai.com/codex/concepts/subagents)

## 技能 + MCP 结合

技能加上 MCP 是一切汇聚的地方：技能定义可重复的工作流程，而 MCP 将它们连接到外部工具和系统。如果一项技能依赖于 MCP，请在 `agents/openai.yaml` 中声明该依赖关系，以便 Codex 可以自动安装并连接（参见 [代理技能](https://developers.openai.com/codex/skills)）。

## 下一步

按以下顺序构建：

1. [使用 AGENTS.md 的自定义指令](https://developers.openai.com/codex/guides/agents-md)，使 Codex 遵循您的代码库约定。添加预提交钩子和代码检查器以强制执行这些规则。
2. 安装一个 [插件](https://developers.openai.com/codex/plugins)，当已存在可重用的工作流程时。否则，当您想要分享时创建一个 [技能](https://developers.openai.com/codex/skills) 并将其打包为插件。
3. [MCP](https://developers.openai.com/codex/mcp)，当工作流程需要外部系统（Linear、GitHub、文档服务器、设计工具）时。
4. [子代理](https://developers.openai.com/codex/subagents)，当您准备将嘈杂或专业的任务委派给子代理时。
