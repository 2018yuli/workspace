企业管理员可以通过两种方式控制本地 Codex 行为：

- **要求**：管理员强制施加的限制，用户无法覆盖。
- **管理默认值**：Codex 启动时应用的初始值。用户仍可以在会话期间更改设置；Codex 在下次启动时重新应用管理默认值。

## 管理员强制要求 (requirements.toml)

要求限制安全敏感设置（批准政策、批准审阅者、自动审阅政策、沙盒模式、Web 搜索模式、管理钩子，以及可选用户可以启用的 MCP 服务器）。在解析配置时（例如，从 `config.toml`、配置文件或 CLI 配置覆盖），如果值与强制规则冲突，Codex 将还原到兼容值并通知用户。如果你配置了 `mcp_servers` 允许列表，Codex 仅当其名称和身份与批准的条目匹配时才启用 MCP 服务器；否则，Codex 将其禁用。

要求还可以通过 `requirements.toml` 中的 `[features]` 表限制 [功能标志](https://developers.openai.com/codex/config-basic/#feature-flags)。请注意，功能不总是安全敏感，但企业可以根据需要固定值。省略的键将保持不受限制。

关于确切的键列表，请参见 [配置参考中的 `requirements.toml` 部分](https://developers.openai.com/codex/config-reference#requirementstoml)。

### 位置和优先级

Codex 按以下顺序应用要求层（早期 wins 每个字段）：

1. 云管理要求（ChatGPT 商务版或企业版）
2. macOS 管理偏好（MDM）通过 `com.openai.codex:requirements_toml_base64`
3. 系统 `requirements.toml`（Unix 系统上的 `/etc/codex/requirements.toml`，包括 Linux/macOS，或 Windows 上的 `%ProgramData%\OpenAI\Codex\requirements.toml`）

在各层之间，Codex 按字段合并要求：如果较早的层设置了某字段（包括空列表），后来的层不会覆盖该字段，但较低的层仍可以填充未设置的字段。

为了向后兼容，Codex 还解释遗留的 `managed_config.toml` 字段 `approval_policy` 和 `sandbox_mode` 作为要求（仅允许该单一值）。

### 云管理要求

当您使用 ChatGPT 登录商务或企业计划时，Codex 还可以从 Codex 服务中获取管理员强制要求。这是另一个与 `requirements.toml` 兼容的要求来源。这适用于 Codex 的所有表面，包括 CLI、应用和 IDE 扩展。

#### 配置云管理要求

请访问 [Codex 管理配置页面](https://chatgpt.com/codex/settings/managed-configs)。

使用与 `requirements.toml` 相同的格式和键创建一个新的管理要求文件。

```
enforce_residency = "us"
allowed_approval_policies = ["on-request"]
allowed_sandbox_modes = ["read-only", "workspace-write"]

[rules]
prefix_rules = [\
  { pattern = [{ any_of = ["bash", "sh", "zsh"] }], decision = "prompt", justification = "Require explicit approval for shell entrypoints" },\
]
```

保存配置。保存后，更新的管理要求立即适用于匹配的用户。
有关更多示例，请参见 [示例 requirements.toml](https://developers.openai.com/codex/enterprise/managed-configuration#example-requirementstoml)。

#### 将要求分配给组

管理员可以为不同的用户组配置不同的管理要求，并设置默认的回退要求策略。

如果用户匹配多个组特定规则，则第一个匹配规则适用。Codex 不从后面的匹配组规则中填充未设置的字段。

例如，如果第一个匹配组规则仅设置 `allowed_sandbox_modes = ["read-only"]` 而后面的匹配组规则设置 `allowed_approval_policies = ["on-request"]`，则 Codex 仅应用第一个匹配组规则，而不填充后面规则中的 `allowed_approval_policies`。

#### Codex 如何在本地应用云管理要求

当用户启动 Codex 并以 ChatGPT 商务或企业计划登录时，Codex 会尽力应用管理要求。Codex 首先检查是否有有效的、未过期的本地管理要求缓存项，如果可用，则使用它。如果缓存缺失、过期、损坏或与当前身份验证不匹配，Codex 会尝试从服务中获取管理要求（进行重试），并在成功时写入新的签名缓存项。如果没有有效的缓存项可用且获取失败或超时，Codex 将继续而不使用管理要求层。

在缓存解析后，Codex 按上述正常要求分层的一部分强制执行管理要求。

### 示例 requirements.toml

该示例阻止 `--ask-for-approval never` 和 `--sandbox danger-full-access`（包括 `--yolo`）：

```
allowed_approval_policies = ["untrusted", "on-request"]
allowed_sandbox_modes = ["read-only", "workspace-write"]
```

### 通过主机覆盖沙盒要求

当需要在不同主机上应用不同的沙盒要求时，使用 `[[remote_sandbox_config]]`。例如，您可以为笔记本电脑保持严格的默认设置，同时允许匹配的开发盒或 CI 运行器进行工作区写入。主机特定条目目前仅覆盖 `allowed_sandbox_modes`：

```
allowed_sandbox_modes = ["read-only"]

[[remote_sandbox_config]]
hostname_patterns = ["*.devbox.example.com", "runner-??.ci.example.com"]
allowed_sandbox_modes = ["read-only", "workspace-write"]
```

Codex 将每个 `hostname_patterns` 条目与尽力解析的主机名进行比较。它在可用时优先选择完全合格的域名，并回退到本地主机名。匹配是不区分大小写的；`*` 匹配任意字符序列，`?` 匹配一个字符。

在同一要求源内，第一个匹配的 `[[remote_sandbox_config]]` 条目获胜。如果没有条目匹配，Codex 将保留顶层的 `allowed_sandbox_modes`。主机名匹配仅用于策略选择；不要将其视为经过身份验证的设备证明。

您还可以约束 Web 搜索模式：

```
allowed_web_search_modes = ["cached"] # "disabled" 保持隐式允许
```

`allowed_web_search_modes = []` 仅允许 `"disabled"`。
例如，`allowed_web_search_modes = ["cached"]` 即使在 `danger-full-access` 会话中也会阻止实时网络搜索。

您还可以固定 [功能标志](https://developers.openai.com/codex/config-basic/#feature-flags)：

```
[features]
personality = true
unified_exec = false
```

使用 `config.toml` 中 `[features]` 表的规范性功能键。Codex 将结果特征集规范化以满足这些固定值，并拒绝对 `config.toml` 或配置范围特征设置的冲突写入。

### 禁用 Codex 的特征表面

管理员可以使用 `[feature_requirements]` 为接收管理 `requirements.toml` 的用户禁用特定的 Codex 特征表面。您还可以在现有 `[features]` 表中设置相同的键。

```
[feature_requirements]
browser_use = false
in_app_browser = false
computer_use = false
```

- `in_app_browser = false` 禁用应用内浏览器面板。
- `browser_use = false` 禁用浏览器使用和浏览器代理可用性。
- `computer_use = false` 禁用计算机使用可用性及相关的安装或启用流程。

如果省略，这些特征由政策允许，受正常客户端、平台和推出可用性的限制。

### 配置自动审查政策

使用 `allowed_approvals_reviewers` 要求或允许自动审查。将其设置为 `["auto_review"]` 以要求自动审查，或在用户可以选择手动批准时包含 `"user"`。

设置 `guardian_policy_config` 以替换自动审查政策的租户特定部分。Codex 仍然使用内置审阅者模板和输出合约。管理的 `guardian_policy_config` 优先于本地的 `[auto_review].policy`。

```
allowed_approval_policies = ["on-request"]
allowed_approvals_reviewers = ["auto_review"]

guardian_policy_config = """
## 环境配置文件
- 受信任的内部目标包括 github.com/my-org、artifacts.example.com 和内部 CI 系统。

## 租户风险分类法和允许/拒绝规则
- 将上传到未批准的第三方文件共享服务视为高风险。
- 拒绝将凭据或私有源代码暴露给不受信任的目标的操作。
"""
```

### 强制拒绝读取要求

管理员可以使用 `[permissions.filesystem]` 对确切路径或全局模式拒绝读取。用户无法通过本地配置削弱这些要求。

```
[permissions.filesystem]
deny_read = [\
  "/Users/alice/.ssh",\
  "./private/**/*.txt",\
]
```

当拒绝读取要求存在时，Codex 将限制本地沙盒模式为 `read-only` 或 `workspace-write`，以便 Codex 能够强制执行它们。在本机 Windows 上，管理的 `deny_read` 适用于直接文件工具；shell 子进程读取不使用此沙盒规则。

### 从要求强制执行管理钩子

管理员还可以在 `requirements.toml` 中直接定义管理的生命周期钩子。使用 `[hooks]` 进行钩子配置，并将 `managed_dir` 指向您的 MDM 或端点管理工具安装引用脚本的目录。

```
[features]
codex_hooks = true

[hooks]
managed_dir = "/enterprise/hooks"
windows_managed_dir = 'C:\enterprise\hooks'

[[hooks.PreToolUse]]
matcher = "^Bash$"

[[hooks.PreToolUse.hooks]]
type = "command"
command = "python3 /enterprise/hooks/pre_tool_use_policy.py"
timeout = 30
statusMessage = "Checking managed Bash command"
```

注意：

- Codex 从 `requirements.toml` 强制执行钩子配置，但不会分发 `managed_dir` 中的脚本。
- 请通过您的 MDM 或设备管理解决方案单独交付这些脚本。
- 管理钩子命令应引用配置的管理目录下的绝对脚本路径。

### 从要求强制执行命令规则

管理员还可以使用 `[rules]` 表从 `requirements.toml` 强制执行限制性命令规则。这些规则与常规 `.rules` 文件合并，最限制性的决定仍然有效。

与 `.rules` 不同，要求规则必须指定 `decision`，且该决定必须为 `"prompt"` 或 `"forbidden"`（而不是 `"allow"`）。

```
[rules]
prefix_rules = [\
  { pattern = [{ token = "rm" }], decision = "forbidden", justification = "Use git clean -fd instead." },\
  { pattern = [{ token = "git" }, { any_of = ["push", "commit"] }], decision = "prompt", justification = "Require review before mutating history." },\
]
```

为了限制 Codex 能够启用的 MCP 服务器，请添加 `mcp_servers` 批准列表。对于 stdio 服务器，按 `command` 匹配；对于可流式传输的 HTTP 服务器，按 `url` 匹配：

```
[mcp_servers.docs]
identity = { command = "codex-mcp" }

[mcp_servers.remote]
identity = { url = "https://example.com/mcp" }
```

如果存在但为空的 `mcp_servers`，Codex 将禁用所有 MCP 服务器。

## 管理默认值 (`managed_config.toml`)

管理默认值与用户的本地 `config.toml` 叠加并优先于任何 CLI `--config` 覆盖，在 Codex 启动时设置初始值。用户仍然可以在会话期间更改这些设置；Codex 在下次启动时重新应用管理默认值。

请确保您的管理默认值满足您的要求；Codex 拒绝不允许的值。

### 优先级和分层

Codex 按以下顺序组装有效配置（顶部覆盖底部）：

- 管理偏好（macOS MDM；最高优先级）
- `managed_config.toml`（系统/管理文件）
- `config.toml`（用户的基础配置）

CLI `--config key=value` 覆盖适用于基础，但管理层覆盖它们。这意味着每次运行都从管理默认值开始，即使您提供本地标志。

云管理要求影响要求层（而不是管理默认值）。有关优先级，请参见上述管理员强制要求部分。

### 位置

- Linux/macOS（Unix）：`/etc/codex/managed_config.toml`
- Windows/非 Unix：`~/.codex/managed_config.toml`

如果文件缺失，Codex 将跳过管理层。

### macOS 管理偏好（MDM）

在 macOS 上，管理员可以推送设备配置文件，提供 base64 编码的 TOML 负载：

- 偏好域：`com.openai.codex`
- 键：
  - `config_toml_base64`（管理默认值）
  - `requirements_toml_base64`（要求）

Codex 解析这些“管理偏好”负载为 TOML。对于管理默认值（`config_toml_base64`），管理偏好具有最高优先级。对于要求（`requirements_toml_base64`），优先级遵循上述云管理要求顺序。同一个要求侧的 `[features]` 表也可用于 `requirements_toml_base64`；在这里也使用规范性功能键。

### MDM 设置工作流程

Codex 遵循标准的 macOS MDM 负载，因此您可以使用 `Jamf Pro`、`Fleet` 或 `Kandji` 等工具分发设置。轻量级部署大致如下：

1. 构建管理负载 TOML 并使用 `base64` 编码（无需换行）。
2. 将字符串放入您的 MDM 配置文件的 `com.openai.codex` 域下，放置在 `config_toml_base64`（管理默认值）或 `requirements_toml_base64`（要求）。
3. 推送配置文件，然后让用户重新启动 Codex 并确认启动配置摘要反映管理值。
4. 当撤销或更改政策时，更新管理负载；CLI 会在下次启动时读取已刷新偏好。

避免在负载中嵌入秘密或高变动动态值。将管理 TOML 视为其他 MDM 设置的一部分，需遵循变更控制。

### 示例 managed_config.toml

```
# 设置保守的默认值
approval_policy = "on-request"
sandbox_mode    = "workspace-write"

[sandbox_workspace_write]
network_access = false             # 默认禁用网络，除非明确允许
```

### 推荐的保护措施

- 对大多数用户偏好 `workspace-write` 需要批准；将完全访问权限保留给控制的容器。
- 除非安全审核允许收集器或您的工作流所需的域，否则保持 `network_access = false`。
- 使用管理配置固定 OTel 设置（导出器、环境），但除非您的政策明确允许存储提示内容，否则保持 `log_user_prompt = false`。
- 定期审核本地 `config.toml` 和管理政策之间的差异，以捕捉漂移；管理层应优先于本地标志和文件。
