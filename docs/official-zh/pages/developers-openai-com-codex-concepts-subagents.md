Codex 可以通过并行生成专门的代理来运行子代理工作流，以便它们可以同时探索、处理或分析工作。

本页面解释了核心概念和权衡。有关设置、代理配置和示例，请参见 [子代理](https://developers.openai.com/codex/subagents)。

## 为什么子代理工作流有帮助

即使有较大的上下文窗口，模型也是有限的。如果你用噪音的中间输出（如探索笔记、测试日志、堆栈跟踪和命令输出）淹没主对话（你在其中定义要求、约束和决策），会导致会话随着时间的推移变得不可靠。

这通常被描述为：

- **上下文污染**：有用的信息被埋没在嘈杂的中间输出之下。
- **上下文腐烂**：随着对话被不太相关的细节填充，性能下降。

有关背景，请参见 Chroma 关于 [上下文腐烂](https://research.trychroma.com/context-rot) 的文章。

子代理工作流通过将嘈杂的工作移出主线程来提供帮助：

- 保持 **主代理** 专注于要求、决策和最终输出。
- 运行专门的 **子代理**，以进行探索、测试或日志分析。
- 从子代理返回 **摘要**，而不是原始中间输出。

当工作可以独立并行运行时，它们还可以节省时间，并通过将更大形状的任务分解为有限的部分来使其更易处理。例如，Codex 可以将对一个数百万标记的文档的分析分解为更小的问题，并将精炼的摘要返回给主线程。

作为起点，使用并行代理处理重读任务，例如探索、测试、分类和总结。在并行写入密集型工作流中要更加小心，因为多个代理同时编辑代码可能会导致冲突并增加协调开销。

## 核心术语

Codex 在子代理工作流中使用几个相关术语：

- **子代理工作流**：Codex 运行并行代理并组合其结果的工作流。
- **子代理**：Codex 启动以处理特定任务的代理。
- **代理线程**：代理的 CLI 线程，你可以通过 `/agent` 检查并在其间切换。

## 触发子代理工作流

Codex 不会自动生成子代理，只有在你明确要求子代理或并行代理工作时才应使用子代理。

实际上，手动触发意味着使用直接指令，例如“生成两个代理”、“并行分配此工作”或“一点使用一个代理”。子代理工作流消耗的令牌比可比的单代理运行更多，因为每个子代理都进行自己的模型和工具工作。

一个好的子代理提示应该解释如何划分工作，Codex 是否应在继续之前等待所有代理，以及应返回的摘要或输出。

```
使用并行子代理审查此分支。为安全风险生成一个子代理，为测试缺口生成一个子代理，为可维护性生成一个子代理。等待所有三个，然后按类别总结发现并附上文件参考。
```

## 选择模型和推理

不同的代理需要不同的模型和推理设置。

如果你不固定模型或 `model_reasoning_effort`，Codex 可以选择一种平衡任务的智能、速度和价格的设置。它可能偏爱 `gpt-5.4-mini` 用于快速扫描，或在该模型可用时更高努力的 `gpt-5.5` 配置以进行更具挑战性的推理。当你需要更精细的控制时，可以在提示中引导该选择，或直接在代理文件中设置 `model` 和 `model_reasoning_effort`。

对于 Codex 中的大多数任务，当 `gpt-5.5` 可用时，请从 `gpt-5.5` 开始。如果 `gpt-5.5` 尚不可用，请在推送期间继续使用 `gpt-5.4`。当你希望为较轻的子代理工作寻求更快、更低成本的选项时，请使用 `gpt-5.4-mini`。如果你拥有 ChatGPT Pro 并希望进行近乎即时的文本迭代，`gpt-5.3-codex-spark` 在研究预览中仍然可用。

### 模型选择

- **`gpt-5.5`**：在可用时，从这里开始用于要求较高的代理。它在需要计划、工具使用、验证和在更大上下文中持续跟进的模糊、多步骤工作中表现最强。
- **`gpt-5.4`**：当 `gpt-5.5` 尚不可用或工作流固定为 GPT-5.4 时使用。它结合了强大的编码、推理、工具使用和更广泛的工作流。
- **`gpt-5.4-mini`**：用于偏向速度和效率而非深度的代理，例如探索、读写密集工作、大文件审查或处理支持文档。它适用于以主代理返回提炼结果的并行工人。
- **`gpt-5.3-codex-spark`**：如果你拥有 ChatGPT Pro，在延迟比更广泛的能力更重要的情况下，使用此研究预览模型进行近乎即时的文本迭代。

### 推理努力 (`model_reasoning_effort`)

- **`高`**：当代理需要跟踪复杂逻辑、检查假设或处理边缘情况时使用（例如，审查者或安全导向的代理）。
- **`中`**：大多数代理的均衡默认值。
- **`低`**：当任务简单且速度最重要时使用。

较高的推理努力会增加响应时间和令牌使用，但可以提高复杂工作的质量。有关详细信息，请参见 [模型](https://developers.openai.com/codex/models)、[配置基础知识](https://developers.openai.com/codex/config-basic) 和 [配置参考](https://developers.openai.com/codex/config-reference)。
