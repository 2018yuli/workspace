Codex 在执行任何工作之前会读取 `AGENTS.md` 文件。通过将全局指导与特定项目的覆盖层叠，您可以从每个任务开始时保持一致的期望，无论您打开哪个代码库。

## Codex 如何发现指导

Codex 在启动时（每次运行一次；在 TUI 中，这通常意味着每次启动会话时）构建一个指令链。发现遵循以下优先顺序：

1. **全局范围：** 在您的 Codex 主目录中（默认是 `~/.codex`，除非您设置了 `CODEX_HOME`），Codex 会读取 `AGENTS.override.md`（如果存在）。否则，Codex 会读取 `AGENTS.md`。Codex 只使用该层级中的第一个非空文件。
2. **项目范围：** 从项目根目录开始（通常是 Git 根），Codex 向下遍历到您当前的工作目录。如果 Codex 找不到项目根目录，它只检查当前目录。在路径上的每个目录中，它会检查 `AGENTS.override.md`，然后是 `AGENTS.md`，接着是任何在 `project_doc_fallback_filenames` 中的回退名称。Codex 每个目录最多包含一个文件。
3. **合并顺序：** Codex 从根目录向下连接文件，用空行分隔。距离您当前目录较近的文件会覆盖早期的指导，因为它们在合并提示中出现得较晚。

Codex 会跳过空文件，并且在合并大小达到 `project_doc_max_bytes` 定义的限制后停止添加文件（默认为 32 KiB）。有关这些设置的详细信息，请参见 [项目指令发现](https://developers.openai.com/codex/config-advanced#project-instructions-discovery)。当您达到上限时，请提高限制或在嵌套目录中分割指令。

## 创建全局指导

在您的 Codex 主目录中创建持久性默认值，以便每个代码库都可以继承您的工作协议。

1. 确保目录存在：

```
mkdir -p ~/.codex
```

2. 创建 `~/.codex/AGENTS.md` 并使用可重用的偏好设置：

```
# ~/.codex/AGENTS.md

## 工作协议

- 修改 JavaScript 文件后始终运行 `npm test`。
- 安装依赖时首选使用 `pnpm`。
- 添加新的生产依赖前需征求确认。
```

3. 在任何地方运行 Codex 来确认它加载了该文件：

```
codex --ask-for-approval never "总结当前指令。"
```

期望：Codex 在提议工作之前引用 `~/.codex/AGENTS.md` 中的条目。

当您需要临时的全局覆盖而无需删除基础文件时，请使用 `~/.codex/AGENTS.override.md`。删除覆盖以恢复共享指导。

## 层叠项目指令

存储库级文件保持 Codex 了解项目规范，同时仍然继承您的全局默认值。

1. 在您的存储库根目录中，添加一个涵盖基本设置的 `AGENTS.md`：

```
# AGENTS.md

## 存储库预期

- 在打开拉取请求之前运行 `npm run lint`。
- 当您更改行为时，在 `docs/` 中记录公共工具。
```

2. 当特定团队需要不同规则时，在嵌套目录中添加覆盖。例如，在 `services/payments/` 内创建 `AGENTS.override.md`：

```
# services/payments/AGENTS.override.md

## 支付服务规则

- 使用 `make test-payments` 而不是 `npm test`。
- 未通知安全渠道前绝不要旋转 API 密钥。
```

3. 从支付目录启动 Codex：

```
codex --cd services/payments --ask-for-approval never "列出您加载的指令源。"
```

期望：Codex 首先报告全局文件，其次是存储库根 `AGENTS.md`，最后是支付覆盖。

Codex 一旦到达当前目录就停止搜索，因此请把覆盖放置在离专门工作尽可能近的地方。

这是您添加全局文件和支付特定覆盖后的示例存储库：

- AGENTS.md  存储库预期

- services/

  - payments/

    - AGENTS.md  被覆盖文件忽略

    - AGENTS.override.md  支付服务规则

    - README.md

  - search/

    - AGENTS.md

    - …

## 自定义回退文件名

如果您的存储库已经使用不同的文件名（例如 `TEAM_GUIDE.md`），请将其添加到回退列表，以便 Codex 将其视为指令文件。

1. 编辑您的 Codex 配置：

```
# ~/.codex/config.toml
project_doc_fallback_filenames = ["TEAM_GUIDE.md", ".agents.md"]
project_doc_max_bytes = 65536
```

2. 重新启动 Codex 或运行新命令，以便更新配置加载。

现在 Codex 按以下顺序检查每个目录：`AGENTS.override.md`、`AGENTS.md`、`TEAM_GUIDE.md`、`.agents.md`。不在此列表中的文件将被忽略以进行指令发现。更大的字节限制允许在截断之前提供更多的合并指导。

在回退列表到位后，Codex 将将替代文件视为指令：

- TEAM\_GUIDE.md  通过回退列表检测

- .agents.md  根目录中的回退文件

- support/

  - AGENTS.override.md  覆盖回退指导

  - playbooks/

    - …

当您想要不同的配置文件时，请设置 `CODEX_HOME` 环境变量，例如项目特定的自动化用户：

```
CODEX_HOME=$(pwd)/.codex codex exec "列出活动指令源"
```

期望：输出将相对于自定义的 `.codex` 目录列出文件。

## 验证您的设置

- 从存储库根目录运行 `codex --ask-for-approval never "总结当前指令。“`。Codex 应该按优先顺序回显来自全局和项目文件的指导。
- 使用 `codex --cd subdir --ask-for-approval never "显示哪些指令文件处于活动状态。" ` 确认嵌套覆盖替换更广泛的规则。
- 如果您需要审核 Codex 加载了哪些指令文件，请检查 `~/.codex/log/codex-tui.log`（或如果您启用了会话日志记录的最新 `session-*.jsonl` 文件）。
- 如果指令看起来过时，请在目标目录中重新启动 Codex。Codex 在每次运行时（以及每次 TUI 会话开始时）都重新构建指令链，因此不会手动清除缓存。

## 解决发现问题

- **没有加载内容：** 验证您处于预期的存储库中，并且 `codex status` 报告您期望的工作区根目录。确保指令文件包含内容；Codex 会忽略空文件。
- **出现错误的指导：** 查找在目录树中更高层的 `AGENTS.override.md` 或在您的 Codex 主目录下。重命名或删除覆盖以回退到常规文件。
- **Codex 忽略回退名称：** 确认您在 `project_doc_fallback_filenames` 中没有拼写错误，并重新启动 Codex 以便更新配置生效。
- **指令被截断：** 提高 `project_doc_max_bytes` 或在嵌套目录中拆分大型文件以保持关键指导不变。
- **配置文件混淆：** 在启动 Codex 之前运行 `echo $CODEX_HOME`。非默认值指出 Codex 指向与您编辑的不同主目录。
