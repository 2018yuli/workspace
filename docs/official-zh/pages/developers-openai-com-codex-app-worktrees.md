在 Codex 应用中，工作树使 Codex 能够在同一个项目中运行多个独立任务，而不会相互干扰。对于 Git 仓库，[自动化](https://developers.openai.com/codex/app/automations) 在专用后台工作树上运行，以免与您正在进行的工作发生冲突。在非版本控制的项目中，自动化直接在项目目录中运行。您还可以手动在工作树上启动线程，并使用 Handoff 将线程在本地和工作树之间移动。

## 什么是工作树

工作树仅在 Git 仓库中的项目中有效，因为它们在底层使用 [Git 工作树](https://git-scm.com/docs/git-worktree)。工作树允许您创建仓库的第二个副本（“检出”）。每个工作树都有您仓库中每个文件的独立副本，但它们都共享有关提交、分支等相同的元数据（`.git` 文件夹）。这使您能够并行检出并在多个分支上工作。

## 术语

- **本地检出**：您创建的仓库。在 Codex 应用中有时仅称为 **本地**。
- **工作树**：从您在 Codex 应用中本地检出创建的 [Git 工作树](https://git-scm.com/docs/git-worktree)。
- **Handoff**：将线程在本地和工作树之间移动的流程。Codex 处理所需的 Git 操作，以安全地在它们之间移动您的工作。

## 为什么使用工作树

1. 与 Codex 并行工作，而不会干扰您当前的本地设置。
2. 在您专注于前台的同时排队后台工作。
3. 当您准备好检查、测试或更直接地协作时，将线程移动到本地。

## 入门

工作树需要一个 Git 仓库。确保您选择的项目位于其中。

1. 选择“工作树”

在新线程视图中，在撰写器下选择 **工作树**。
可选地，选择一个 [本地环境](https://developers.openai.com/codex/app/local-environments) 来运行工作树的设置脚本。

2. 选择起始分支

在撰写器下方，选择要基于的 Git 分支。这可以是您的 `main` / `master` 分支、功能分支或您当前的带有未暂存本地更改的分支。

3. 提交您的提示

提交您的任务，Codex 将基于您选择的分支创建一个 Git 工作树。默认情况下，Codex 在 [“分离 HEAD”](https://git-scm.com/docs/git-checkout#_detached_head) 状态下工作。

4. 选择在哪里继续工作

当您准备好时，您可以继续直接在工作树上工作，或将线程移交给您的本地检出。将线程移交到本地或从本地移交将移动您的线程 _和_ 代码，以便您可以在另一个检出中继续。

## 在本地和工作树之间工作

工作树的外观和感觉与您的本地检出非常相似。不同之处在于它们在您的工作流程中的位置。您可以将本地视为前台，将工作树视为后台。Handoff 允许您在它们之间移动线程。

在底层，Handoff 处理所需的 Git 操作，以安全地在两个检出之间移动工作。这是重要的，因为 **Git 只允许同一时间在一个地方检出一个分支**。如果您在工作树上检出一个分支，您 **不能** 同时在本地检出它，反之亦然。

在实践中，有两条常见的路径：

1. [专门在工作树上工作](https://developers.openai.com/codex/app/worktrees#option-1-working-on-the-worktree)。当您可以直接在工作树上验证更改时，这条路径效果最好，例如，因为您安装了依赖项和工具使用 [本地环境设置脚本](https://developers.openai.com/codex/app/local-environments)。
2. [将线程移交给本地](https://developers.openai.com/codex/app/worktrees#option-2-handing-a-thread-off-to-local)。当您希望将线程带入前台时使用此方法，例如因为您想在您通常的 IDE 中检查更改或只能运行一个实例的应用程序时。

### 选项 1：在工作树上工作

如果您想在工作树上保持您的更改，可以使用线程头部的 **在此创建分支** 按钮将工作树转变为分支。

从这里，您可以提交您的更改，将您的分支推送到远程仓库，并在 GitHub 上打开拉取请求。

您可以使用头部的 “打开” 按钮在工作树中打开您的 IDE，使用集成终端，或者从工作树目录执行您需要的任何操作。

请记住，如果您在工作树上创建分支，您不能在任何其他工作树上检出，包括您的本地检出。

### 选项 2：将线程移交给本地

如果您想将线程带入前台，请在您的线程头部单击 **移交**，将其移动到 **本地**。

当您希望在您通常的 IDE 窗口中读取更改、运行现有开发服务器或在您日常使用的相同环境中验证工作时，这条路径效果很好。

Codex 处理将线程安全地移动到工作树和本地检出所需的 Git 步骤。

每个线程随时间保持相同的关联工作树。如果您稍后将线程移交回工作树，Codex 会将其返回到那个相同的背景环境，以便您可以从上次中断的地方继续。

您也可以反方向进行。如果您已经在本地工作并希望释放前台，使用 **移交** 将线程移动到工作树。当您希望 Codex 在后台保持工作，同时您将注意力切换回本地的其他事情时，这非常有用。

由于 Handoff 使用 Git 操作，因此 `.gitignore` 文件中包含的任何文件将不会随线程一起移动。

## 高级详情

### Codex 管理和永久工作树

默认情况下，线程使用 Codex 管理的工作树。这些工作树旨在让您感觉轻巧和可丢弃。Codex 管理的工作树通常专用于一个线程，如果您稍后将该线程移交回工作树，Codex 会将其返回到相同的工作树。

如果您需要一个长期存在的环境，请通过侧边栏中的三点菜单从项目创建一个永久工作树。这会创建一个新的永久工作树，作为其自己的项目。永久工作树不会自动删除，您可以从同一工作树启动多个线程。

### Codex 如何为您管理工作树

Codex 在 `$CODEX_HOME/worktrees` 中创建工作树。起始提交将是您启动线程时选择的分支的 `HEAD` 提交。如果您选择了一个带有本地更改的分支，未提交的更改也将应用于工作树。工作树 _不会_ 被检出为分支。它将处于 [分离 HEAD](https://git-scm.com/docs/git-checkout#_detached_head) 状态。这使 Codex 能够创建多个工作树，而不会污染您的分支。

### 分支限制

假设 Codex 在工作树上完成了一些工作，而您选择使用 **在此创建分支** 创建一个 `feature/a` 分支。现在，您想在本地检出尝试。如果您尝试检出该分支，您将得到以下错误：

```
fatal: 'feature/a' is already used by worktree at '<WORKTREE_PATH>'
```

要解决此问题，您需要在工作树上检出另一个分支，而不是 `feature/a`。

如果您计划在本地检出该分支，请使用 Handoff 将线程移交到本地，而不是尝试在两个地方同时保持同一分支检出。

为什么会存在这个限制

Git 防止在同一时间在多个工作树中检出相同的分支，因为分支表示一个单一的可变引用（`refs/heads/<name>`），其含义是“当前检出状态”的工作树。

当分支被检出时，Git 将其 HEAD 视为属于该工作树，并期望诸如提交、重置、变基和合并等操作以明确定义的序列化方式推进该引用。允许多个工作树同时检出同一分支将造成模糊性和竞争条件，导致哪个工作树的操作更新分支引用，从而可能导致丢失提交、不一致的索引或不清晰的冲突解决。

通过强制执行每个工作树只有一个分支的规则，Git 确保每个分支都有一个单一的权威工作副本，同时仍允许其他工作树通过分离 HEAD 或独立分支安全地引用相同的提交。

### 工作树清理

工作树可能占用大量磁盘空间。每个工作树都有自己的一组仓库文件、依赖项、构建缓存等。因此，Codex 应用程序试图将工作树的数量保持在合理的限制内。

默认情况下，Codex 保留您最近的 15 个 Codex 管理的工作树。如果您希望管理磁盘使用，可以改变此限制或关闭自动删除。

Codex 尝试避免删除仍然重要的工作树。如果满足以下条件，Codex 管理的工作树不会自动删除：

- 有一个固定的对话与之关联
- 线程仍在进行中
- 工作树是永久工作树

当满足以下条件时，Codex 管理的工作树会自动删除：

- 您归档了关联的线程
- Codex 需要删除旧工作树，以保持在配置的限制内

在删除 Codex 管理的工作树之前，Codex 会保存其工作快照。如果您在其工作树被删除后打开对话，您会看到恢复的选项。

## 常见问题

我可以控制工作树的创建位置吗？

今天不可以。Codex 在 `$CODEX_HOME/worktrees` 中创建工作树，以便一致地管理它们。

我可以在本地和工作树之间移动线程吗？

可以。在线程头部使用 **移交** 将线程在本地检出和工作树之间移动。Codex 处理在环境之间安全移动线程所需的 Git 操作。如果您稍后将线程移交回工作树，Codex 将其返回到相同的关联工作树。

如果工作树被删除，线程会发生什么？

即使底层工作树目录被删除，线程仍然可以保留在您的历史记录中。对于 Codex 管理的工作树，如果在删除工作树之前，Codex 会保存快照，并在您重新打开关联线程时提供恢复选项。永久工作树在您归档其线程时不会自动删除。
