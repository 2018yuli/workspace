对于项目范围的自动化，应用程序需要运行，所选项目需要在磁盘上可用。

您还可以保持模型和推理努力的默认设置，或者如果您希望对自动化的运行有更多控制，也可以明确选择它们。

## 管理任务

在您的 Codex 应用程序侧边栏中的自动化窗格中寻找所有自动化及其运行情况。

“Triage” 部分充当您的收件箱。具有发现的自动化运行将显示在此处，您可以筛选收件箱以显示所有自动化运行或仅未读内容。

独立自动化按计划启动全新运行并在 Triage 中报告结果。当每个运行应该是独立的，或者当一个自动化应该在一个或多个项目中运行时，使用它们。如果您需要自定义的节奏，请选择自定义调度并输入 cron 语法。

对于 Git 仓库，每个自动化可以运行在您的本地项目中，或者在专用的后台 [工作树](https://developers.openai.com/codex/app/features#worktree-support) 上。使用工作树时，希望将自动化更改与未完成的本地工作隔离开来。使用本地模式时，希望自动化直接在您的主签出中工作，请记住它可能会更改您正在积极编辑的文件。在非版本控制的项目中，自动化直接在项目目录中运行。您可以让同一个自动化在多个项目上运行。

自动化使用您默认的沙盒设置。在只读模式下，如果工具调用需要修改文件、访问网络或与计算机上的应用程序一起工作，则会失败。启用完全访问时，后台自动化存在更高的风险。您可以在 [设置](https://developers.openai.com/codex/app/settings) 中调整沙盒设置，并通过 [规则](https://developers.openai.com/codex/rules) 有选择地允许命令。

自动化可以使用 Codex 可用的相同插件和技能。为了保持自动化在团队中的可维护性和可共享性，请使用 [技能](https://developers.openai.com/codex/skills) 定义操作并提供工具和上下文。您可以通过在自动化中使用 `$skill-name` 显式触发技能。

## 请求 Codex 创建或更新自动化

您可以从常规 Codex 线程中创建和更新自动化。描述任务、调度时间，以及自动化是否应与当前线程保持关联或启动全新运行。Codex 可以起草自动化提示，选择正确的自动化类型，并在范围或节奏变化时更新它。

例如，请求 Codex 在部署完成时在此线程中提醒您，或者请求它创建一个独立的自动化，在定期的时间表上检查项目。

技能也可以创建或更新自动化。例如，用于照看拉取请求的技能可以设置定期自动化，使用 GitHub 插件检查 PR 状态并修复新审查反馈。

## 线程自动化

线程自动化是附加到当前线程的心跳样式的定期唤醒调用。当您希望 Codex 定期返回到相同的对话中时，请使用它们。

当计划工作应保留线程的上下文而不是每次从新的提示开始时，请使用线程自动化。

线程自动化可以使用基于分钟的时间间隔进行积极的跟进循环，或者在您需要在特定时间进行检查时使用每日和每周的计划。

线程自动化非常有用：

- 检查长时间运行的命令，直到其完成
- 在结果应保持在同一线程时，对 Slack、GitHub 或其他已连接源进行轮询
- 提醒 Codex 按固定节奏继续审查循环
- 执行驱动技能的工作流，使用插件，例如检查 PR 状态并处理新的反馈
- 将聊天保持在进行中的研究或分类任务上

当每个运行应该是独立时，或者它应该跨多个项目运行时，或者当发现应该以单独的自动化运行出现在 Triage 中时，请使用独立或项目自动化。

创建线程自动化时，使提示持久。它应该描述在每次线程唤醒时 Codex 应该做什么，如何判断是否有重要内容需要报告，以及何时停止或请求您的输入。

## 测试自动化

在您安排自动化之前，首先在常规线程中手动测试提示。这有助于您确认：

- 提示清晰且范围正确。
- 选择的或默认模型、推理努力和工具的行为符合预期。
- 结果差异可供审查。

当您开始安排运行时，审查前几个输出并根据需要调整提示或节奏。

## 自动化的工作树清理

如果您选择 Git 仓库的工作树，频繁的调度可能会随着时间的推移创建许多工作树。归档您不再需要的自动化运行，并避免固定运行，除非您打算保留其工作树。

## 权限和安全模型

自动化无人值守运行，并使用默认的沙盒设置。

- 如果您的沙盒模式是 **只读**，那么如果工具调用需要修改文件、访问网络或与计算机上的应用程序一起工作，则将失败。考虑将沙盒设置更新为工作区写入。
- 如果您的沙盒模式是 **工作区写入**，那么如果工具调用需要修改工作区外的文件、访问网络或与计算机上的应用程序一起工作，则将失败。您可以使用 [规则](https://developers.openai.com/codex/rules) 有选择地允许命令在沙盒外运行。
- 如果您的沙盒模式是 **完全访问**，则后台自动化存在更高的风险，因为 Codex 可能会更改文件、运行命令，并访问网络，而无需询问。考虑将沙盒设置更新为工作区写入，并使用 [规则](https://developers.openai.com/codex/rules) 有选择地定义代理可以以完全访问方式运行的命令。

如果您处于受管环境中，管理员可以使用管理员强制要求限制这些行为。例如，他们可以禁止 `approval_policy = "never"` 或限制允许的沙盒模式。请参见 [管理员强制要求（`requirements.toml`）](https://developers.openai.com/codex/enterprise/managed-configuration#admin-enforced-requirements-requirementstoml)。

当您的组织政策允许时，自动化使用 `approval_policy = "never"`。如果管理员要求不允许 `approval_policy = "never"`，则自动化将回退到您所选模式的批准行为。

如果有我们反复做且遇到困难的事情，我们应该将其保存为技能，以加快未来的工作，让我们来做。

绝对不要觉得需要更新任何内容——只有在有充分理由的情况下才更新！

如果您更改了任何内容，请告诉我。


```

### 保持与项目的实时更新

```
查看最新的远程 origin/master 或 origin/main。然后为过去 24 小时内触及 <DIRECTORY> 的提交生成执行摘要。

格式 + 结构：

- 使用丰富的 Markdown（H1 工作流部分，斜体作为副标题，需要时使用水平规则）。
- 前言可以读作“这是 <directory> 的过去 24 小时摘要：”
- 副标题应为：“与所有者的叙述性讲解；按工作流分组。”
- 按工作流分组而不是列出每个提交。工作流标题应为 H1。
- 为每个工作流编写简短的叙述，以简单语言解释更改。
- 当使其更易读时使用项目符号和加粗。
- 可以按个人制作项目符号，但请将其名称加粗。

内容要求：

- 包括 PR 链接在内（例如，[#123](...)），而不带“PRs:” 标签。
- 不包括提交哈希或“重点提交”部分。
- 在一个工作流下可以展示多个 PR，但避免逐个提交的项目符号列表。

范围规则：

- 仅包括当前工作目录（cwd）内的更改。
- 仅包括过去 24 小时内的提交。
- 如果有帮助，请使用 `gh` 获取 PR 标题和描述。
  还可以拉取 PR 审查和评论。

```

# 最近代码修复

## 概述

查找在过去一周内由当前作者引入的错误，实施修复，并在可能的情况下验证。操作在当前工作目录内，假设代码是本地的，并确保根本原因直接与作者自己的编辑相关。

## 工作流

### 1) 确定近期更改的范围

使用 Git 确定过去一周的作者和更改文件。

- 从 `git config user.name`/`user.email` 确定作者。如果不可用，使用当前用户的名称或询问一次。
- 使用 `git log --since=1.week --author=<author>` 列出近期提交和文件。重点关注那些提交所触及的文件。
- 如果用户的提示为空，则直接按此默认范围进行。

### 2) 找到与近期更改相关的具体失败

优先考虑直接归因于作者编辑的缺陷。

- 查找近期失败（测试、代码检查、运行时错误），如果可用本地日志或 CI 输出。
- 如果没有提供失败信息，请运行最小相关验证（单个测试、文件级代码检查或针对修改文件的重现命令）。
- 确认根本原因直接与作者的更改相关，而不是无关的遗留问题。如果只发现无关的失败，请停止并报告未检测到符合条件的错误。

### 3) 实施修复

进行与项目规范一致的最小修复。

- 仅更新解决问题所需的文件。
- 避免添加额外的防御性检查或无关的重构。
- 保持与本地样式和测试的一致性。

### 4) 验证

在可能的情况下尝试进行验证。

- 优先选择最小的验证步骤（针对性测试、集中代码检查或直接重现命令）。
- 如果无法进行验证，请说明应该运行什么以及为何未执行。

### 5) 报告

总结根本原因、修复及所执行的验证。明确根本原因如何与作者的最近更改相联系。

```

之后，创建一个新的自动化：

```
检查我过去 24 小时的提交，并提交一个 $recent-code-bugfix。
