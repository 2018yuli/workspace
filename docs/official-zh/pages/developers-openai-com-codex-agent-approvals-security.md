Codex 帮助保护您的代码和数据，减少误用的风险。

本页面涵盖了如何安全地操作 Codex，包括沙箱、审批和网络访问。如果您正在寻找用于扫描连接的 GitHub 仓库的 Codex 安全产品，请参见 [Codex 安全](https://developers.openai.com/codex/security)。

默认情况下，代理在网络访问关闭的情况下运行。在本地，Codex 使用操作系统强制的沙箱，它限制它可以接触的内容（通常是当前工作区），加上一个审批政策，控制何时需要在执行操作之前停止并询问您。

有关沙箱如何在 Codex 应用程序、IDE 扩展和 CLI 中工作的高级解释，请参见 [沙箱](https://developers.openai.com/codex/concepts/sandboxing)。有关更广泛的企业安全概述，请参见 [Codex 安全白皮书](https://trust.openai.com/?itemUid=382f924d-54f3-43a8-a9df-c39e6c959958&source=click)。

## 沙箱和审批

Codex 安全控制来自两个协同工作的层次：

- **沙箱模式**：Codex 在执行模型生成的命令时技术上可以做什么（例如，它可以在哪里写作，是否可以访问网络）。
- **审批政策**：当 Codex 执行某个操作时，必须在执行之前询问您（例如，离开沙箱、使用网络或运行不在可信集合中的命令）。

Codex 根据您运行它的位置使用不同的沙箱模式：

- **Codex 云**：在隔离的 OpenAI 管理的容器中运行，防止访问您的主系统或无关数据。使用两相运行模型：设置在代理阶段之前运行，可以访问网络以安装指定的依赖项，然后代理阶段在默认情况下离线运行，除非您为该环境启用互联网访问。为云环境配置的机密仅在设置期间可用，并在代理阶段开始之前被删除。
- **Codex CLI / IDE 扩展**：操作系统级机制强制执行沙箱政策。默认包括没有网络访问和写入权限仅限于活动工作区。您可以根据风险容忍度配置沙箱、审批政策和网络设置。

在 `Auto` 预设中（例如，`--full-auto`），Codex 可以自动读取文件、进行编辑和在工作目录中运行命令。

Codex 会询问是否可以编辑工作区之外的文件或运行需要网络访问的命令。如果您想聊天或计划而不进行更改，请使用 `/permissions` 命令切换到 `read-only` 模式。

Codex 还可以在应用（连接器）工具调用中引发审批，这些调用会宣传副作用，即使该操作不是 shell 命令或文件更改。当工具宣传破坏性注释时，破坏性应用/MCP 工具调用始终需要审批，即使它也宣传其他提示（例如，只读提示）。

## 网络访问 [提高风险](https://help.openai.com/articles/20001061)

有关 Codex 云，请参见 [代理互联网访问](https://developers.openai.com/codex/cloud/internet-access) 以启用完全互联网访问或域允许列表。

对于 Codex 应用、CLI 或 IDE 扩展，默认的 `workspace-write` 沙箱模式保持网络访问关闭，除非您在配置中启用它：

```
[sandbox_workspace_write]
network_access = true
```

您还可以在不授予完全网络访问权限的情况下控制 [网络搜索工具](https://platform.openai.com/docs/guides/tools-web-search)。Codex 默认使用网络搜索缓存来访问结果。缓存是 OpenAI 维护的网络结果索引，因此缓存模式返回预先索引的结果，而不是提取实时页面。这减少了来自任意实时内容的提示注入的暴露，但您仍应将网络结果视为不可信。如果您使用 `--yolo` 或其他 [完全访问沙箱设置](https://developers.openai.com/codex/agent-approvals-security#common-sandbox-and-approval-combinations)，网络搜索默认使用实时结果。使用 `--search` 或将其设置为 `"live"` 以允许实时浏览，或将其设置为 `"disabled"` 以关闭该工具：

```
web_search = "cached"  # 默认
# web_search = "disabled"
# web_search = "live"  # 与 --search 相同
```

在 Codex 中启用网络访问或网络搜索时请谨慎。提示注入可能导致代理获取并遵循不可信的指令。

## 默认值和推荐

- 启动时，Codex 检测文件夹是否受版本控制，然后建议：
  - 受版本控制的文件夹：`Auto`（工作区写入 + 请求时审批）
  - 非版本控制的文件夹：`read-only`
- 根据您的设置，Codex 也可能在明确信任工作目录之前以 `read-only` 开始（例如，通过入职提示或 `/permissions`）。
- 工作区包括当前目录和临时目录，如 `/tmp`。使用 `/status` 命令查看哪些目录在工作区中。
- 要接受默认设置，请运行 `codex`。
- 您可以明确设置这些：
  - `codex --sandbox workspace-write --ask-for-approval on-request`
  - `codex --sandbox read-only --ask-for-approval on-request`

### 可写根中的受保护路径

在默认的 `workspace-write` 沙箱策略中，可写根仍包括受保护路径：

- `<writable_root>/.git` 不论作为目录还是文件出现，都作为只读受保护。
- 如果 `<writable_root>/.git` 是一个指针文件（`gitdir: ...`），解析后的 Git 目录路径也是只读受保护。
- `<writable_root>/.agents` 在存在为目录时受保护为只读。
- `<writable_root>/.codex` 在存在为目录时受保护为只读。
- 保护是递归的，因此这些路径下的所有内容都是只读的。

### 使用文件系统配置文件拒绝读取

命名权限配置文件也可以拒绝对精确路径或通配符模式的读取。这在工作区应该保持可写但是特定敏感文件（例如本地环境文件）必须保持不可读的情况下很有用：

```
default_permissions = "workspace"

[permissions.workspace.filesystem]
":project_roots" = { "." = "write", "**/*.env" = "none" }
glob_scan_max_depth = 3
```

对 Codex 不应读取的路径或通配符使用 `"none"`。沙箱策略为本地 macOS 和 Linux 的命令执行评估通配符。在预先扩展通配符匹配的 платформ上，在沙箱启动之前，设置 `glob_scan_max_depth` 以处理无界的 `**` 模式，或列出明确深度，如 `*.env`、`*/*.env` 和 `*/*/*.env`。

### 在没有审批提示的情况下运行

您可以使用 `--ask-for-approval never` 或 `-a never` （简写）禁用审批提示。

此选项适用于所有 `--sandbox` 模式，因此您仍可以控制 Codex 的自主级别。Codex 在您设置的约束范围内尽力而为。

如果您需要 Codex 读取文件、进行编辑并在没有审批提示的情况下运行网络访问，请使用 `--sandbox danger-full-access`（或 `--dangerously-bypass-approvals-and-sandbox` 标志）。在这样做之前请小心。

对于中间方案，`approval_policy = { granular = { ... } }` 允许您保持特定审批提示类别为交互式，同时自动拒绝其他。细粒度政策涵盖沙箱审批、执行策略规则提示、MCP 提示、`request_permissions` 提示和技能脚本审批。

### 自动审批审核

默认情况下，审批请求会路由到您：

```
approvals_reviewer = "user"
```

自动审批审核适用于交互式审批，例如 `approval_policy = "on-request"` 或细粒度审批政策。设置 `approvals_reviewer = "auto_review"` 将合格的审批请求通过审查代理路由，在 Codex 执行请求之前：

```
approval_policy = "on-request"
approvals_reviewer = "auto_review"
```

审查员仅评估已需要审批的操作，例如沙箱升级、网络请求、`request_permissions` 提示或产生副作用的应用和 MCP 工具调用。留在沙箱内的操作可继续，没有额外的审核步骤。

审查员政策检查数据外泄、凭证探测、持续安全削弱和破坏性行为。低风险和中风险行为可以在政策允许时继续。政策拒绝关键风险行为。高风险行为需要足够的用户授权且不匹配拒绝规则。超时、解析失败和审核错误将关闭失败。

[默认审查员政策](https://github.com/openai/codex/blob/main/codex-rs/core/src/guardian/policy.md) 在开源 Codex 存储库中。企业可以用 `guardian_policy_config` 替换其特定租户部分，在管理要求中。也支持本地 `[auto_review].policy` 文本，但管理要求优先。有关设置细节，请参见 [受管配置](https://developers.openai.com/codex/enterprise/managed-configuration#configure-automatic-review-policy)。

在 Codex 应用中，这些审核作为自动审核项目出现，状态如审核中、已批准、已拒绝、已停止或超时。它们还可以包含所审查请求的风险水平。

自动审核使用额外的模型调用，因此可能会增加 Codex 使用量。管理员可以用 `allowed_approvals_reviewers` 限制它。

### 常见沙箱和审批组合

| 意图 | 标志 | 效果 |
| --- | --- | --- |
| 自动（预设） | _不需要标志_ 或 `--full-auto` | Codex 可以读取文件、进行编辑并在工作区运行命令。Codex 需要审批才能编辑工作区之外的内容或访问网络。 |
| 安全只读浏览 | `--sandbox read-only --ask-for-approval on-request` | Codex 可以读取文件并回答问题。Codex 需要审批才能进行编辑、运行命令或访问网络。 |
| 只读非交互（CI） | `--sandbox read-only --ask-for-approval never` | Codex 只能读取文件；从不请求审批。 |
| 自动编辑但请求审批以运行不可信的命令 | `--sandbox workspace-write --ask-for-approval untrusted` | Codex 可以读取和编辑文件，但在运行不可信的命令之前请求审批。 |
| 危险完全访问 | `--dangerously-bypass-approvals-and-sandbox`（别名：`--yolo`） | [提高风险](https://help.openai.com/articles/20001061) 不使用沙箱；无审批 _(不推荐)_ |

`--full-auto` 是 `--sandbox workspace-write --ask-for-approval on-request` 的便捷别名。

使用 `--ask-for-approval untrusted`，Codex 仅自动运行已知安全的读取操作。可以改变状态或触发外部执行路径的命令（例如，破坏性 Git 操作或 Git 输出/配置覆盖标志）需要审批。

#### 在 `config.toml` 中的配置

有关更广泛的配置工作流，请参见 [配置基础](https://developers.openai.com/codex/config-basic)、[高级配置](https://developers.openai.com/codex/config-advanced#approval-policies-and-sandbox-modes) 和 [配置参考](https://developers.openai.com/codex/config-reference)。

```
# 始终请求审批模式
approval_policy = "untrusted"
sandbox_mode    = "read-only"
allow_login_shell = false # 可选的强化：禁止 shell 工具的登录 shell

# 可选：在工作区写入模式下允许网络
[sandbox_workspace_write]
network_access = true

# 可选：细粒度审批政策
# approval_policy = { granular = {
#   sandbox_approval = true,
#   rules = true,
#   mcp_elicitations = true,
#   request_permissions = false,
#   skill_approval = false
# } }
```

您还可以将这些预设保存为配置文件，然后通过 `codex --profile <name>` 选择它们：

```
[profiles.full_auto]
approval_policy = "on-request"
sandbox_mode    = "workspace-write"

[profiles.readonly_quiet]
approval_policy = "never"
sandbox_mode    = "read-only"
```

## 操作系统级沙箱

Codex 根据您的操作系统以不同方式强制执行沙箱：

- **macOS** 使用 Seatbelt 策略，使用带有与您选择的 `--sandbox` 模式相对应的配置文件（`-p`）运行命令。当限制读取访问启用平台默认值时，Codex 将附加优化的 macOS 平台策略（而不是广泛允许 `/System`）以保持常用工具的兼容性。
- **Linux** 默认使用 `bwrap` 和 `seccomp`。
- **Windows** 在 [Windows 子系统 Linux 2 (WSL2)](https://developers.openai.com/codex/windows#windows-subsystem-for-linux) 中运行时使用 Linux 沙箱实现。WSL1 通过 Codex `0.114` 支持；从 `0.115` 开始，Linux 沙箱转移到 `bwrap`，因此不再支持 WSL1。在 Windows 原生运行时，Codex 使用 [Windows 沙箱](https://developers.openai.com/codex/windows#windows-sandbox) 实现。

如果在 Windows 上使用 Codex IDE 扩展，它直接支持 WSL2。在您的 VS Code 设置中设置以下内容，以便在可用时让代理保持在 WSL2 中：

```
{
  "chatgpt.runCodexInWindowsSubsystemForLinux": true
}
```

这确保 IDE 扩展继承 Linux 沙箱语义，用于命令、审批和文件系统访问，即使主操作系统是 Windows。有关详细信息，请参见 [Windows 设置指南](https://developers.openai.com/codex/windows)。

在 Windows 原生运行时，配置 native 沙箱模式 `config.toml`：

```
[windows]
sandbox = "unelevated" # 或 "elevated"
# sandbox_private_desktop = true  # 默认；仅在兼容性方面设置为 false
```

有关详细信息，请参见 [Windows 设置指南](https://developers.openai.com/codex/windows#windows-sandbox)。

当您在如 Docker 的容器化环境中运行 Linux 时，如果主机或容器配置阻止 Codex 需要的命名空间、setuid `bwrap` 或 `seccomp` 操作，沙箱可能无法正常工作。

在这种情况下，配置您的 Docker 容器以提供您所需的隔离，然后在容器内运行 `codex`，并使用 `--sandbox danger-full-access`（或 `--dangerously-bypass-approvals-and-sandbox` 标志）。

### 在 Dev Containers 中运行 Codex

如果您的主机无法直接运行 Linux 沙箱，或者您的组织已经标准化为基于容器的开发，则可以使用 Dev Containers 运行 Codex，让 Docker 提供外部隔离边界。这与 Visual Studio Code Dev Containers 和兼容工具一起使用。

使用 [Codex 安全 devcontainer 示例](https://github.com/openai/codex/tree/main/.devcontainer) 作为参考实现。该示例安装 Codex、常用开发工具、`bubblewrap` 和基于防火墙的出站控制。

Devcontainers 提供了实质性保护，但并不能防止每一种攻击。如果在容器内部运行 Codex 并使用 `--sandbox danger-full-access` 或 `--dangerously-bypass-approvals-and-sandbox`，恶意项目可以提取 devcontainer 内部可用的任何内容，包括 Codex 凭据。仅在可信的存储库中使用这种模式，并监控 Codex 的活动，像对待任何其他提高环境一样。

参考实现包括：

- 一个安装了 Codex 和常用开发工具的 Ubuntu 24.04 基础镜像；
- 一个基于允许列表的防火墙配置文件，用于出站访问；
- VS Code 设置和扩展建议，以便在容器中重新打开工作区；
- 持久挂载用于命令历史记录和 Codex 配置；
- `bubblewrap`，以便在容器授予所需能力时，Codex 仍可以使用其 Linux 沙箱。

要尝试：

1. 安装 Visual Studio Code 和 [Dev Containers 扩展](https://marketplace.visualstudio.com/items?itemName=ms-vscode-remote.remote-containers)。
2. 将 Codex 示例 `.devcontainer` 设置复制到您的存储库中，或直接从 Codex 存储库开始。
3. 在 VS Code 中运行 **Dev Containers: 在容器中打开文件夹…** 并选择 `.devcontainer/devcontainer.secure.json`。
4. 在容器启动后，打开终端，运行 `codex`。

您还可以通过 CLI 启动容器：

```
devcontainer up --workspace-folder . --config .devcontainer/devcontainer.secure.json
```

该示例有三个主要部分：

- `.devcontainer/devcontainer.secure.json` 控制容器设置、能力、挂载、环境变量和 VS Code 扩展。
- `.devcontainer/Dockerfile.secure` 定义基于 Ubuntu 的镜像和安装的工具。
- `.devcontainer/init-firewall.sh` 应用出站网络策略。

参考防火墙故意作为起始点。如果您依赖于域允许列表进行隔离，请实施适合您环境的 DNS 重新绑定和 DNS 刷新保护，例如 TTL  aware 刷新或 DNS-aware 防火墙。

在容器内，选择以下模式之一：

- 如果 Dev Container 配置文件授予 `bwrap` 创建内部沙箱所需的能力，请保持 Codex 的 Linux 沙箱启用。
- 如果容器是您期望的安全边界，请在容器内使用 `--sandbox danger-full-access` 运行 Codex，这样 Codex 就不会尝试创建第二个沙箱层。

## 版本控制

Codex 在版本控制工作流中效果最佳：

- 在功能分支上工作，并在委托之前保持 `git status` 整洁。这使得 Codex 补丁更容易隔离和回滚。
- 优先采用基于补丁的工作流（例如，`git diff` / `git apply`），而不是直接编辑受跟踪的文件。频繁提交以便您可以小范围回滚。
- 将 Codex 的建议视为任何其他 PR：运行针对性验证、审查差异并在提交消息中记录决策以便审计。

## 监控和遥测

Codex 支持通过 OpenTelemetry (OTel) 的自选监控，以帮助团队审核使用情况、调查问题，并满足合规要求，而不削弱本地安全默认设置。遥测默认关闭；在配置中显式启用它。

### 概述

- Codex 默认关闭 OTel 导出，以保持本地运行的独立性。
- 启用时，Codex 发出结构化日志事件，涵盖对话、API 请求、SSE/WebSocket 流活动、用户提示（默认情况下会被编辑）、工具审批决策和工具结果。
- Codex 用 `service.name`（发起者）、CLI 版本和环境标签标记导出的事件，以便区分开发/暂存/生产流量。

### 启用 OTel（可选）

向 Codex 配置中添加 `[otel]` 块（通常在 `~/.codex/config.toml`），选择导出器以及是否记录提示文本。

```
[otel]
environment = "staging"   # dev | staging | prod
exporter = "none"          # none | otlp-http | otlp-grpc
log_user_prompt = false     # 除非政策允许，否则编辑提示文本
```

- `exporter = "none"` 保留仪器活跃但不发送数据到任何地方。
- 要将事件发送到您的收集器，请选择以下之一：

```
[otel]
exporter = { otlp-http = {
  endpoint = "https://otel.example.com/v1/logs",
  protocol = "binary",
  headers = { "x-otlp-api-key" = "${OTLP_TOKEN}" }
}}
```

```
[otel]
exporter = { otlp-grpc = {
  endpoint = "https://otel.example.com:4317",
  headers = { "x-otlp-meta" = "abc123" }
}}
```

Codex 将事件分批，并在关闭时刷新它们。Codex 仅导出其 OTel 模块产生的遥测。

### 事件类别

代表事件类型包括：

- `codex.conversation_starts`（模型、推理设置、沙箱/审批政策）
- `codex.api_request`（尝试、状态/成功、持续时间和错误详情）
- `codex.sse_event`（流事件类型、成功/失败、持续时间，以及 `response.completed` 上的令牌计数）
- `codex.websocket_request` 和 `codex.websocket_event`（请求持续时间加上每条消息的种类/成功/错误）
- `codex.user_prompt`（长度；内容在未明确启用前被编辑）
- `codex.tool_decision`（已批准/拒绝，来源：配置与用户）
- `codex.tool_result`（持续时间、成功、输出片段）

相关的 OTel 指标（计数器加上持续时间直方图对）包括 `codex.api_request`、`codex.sse_event`、`codex.websocket.request`、`codex.websocket.event` 和 `codex.tool.call`（及相应的 `.duration_ms` 仪器）。

有关完整事件目录和配置参考，请参见 [GitHub 上的 Codex 配置文档](https://github.com/openai/codex/blob/main/docs/config.md#otel)。

### 安全性和隐私指导

- 保持 `log_user_prompt = false`，除非政策明确允许存储提示内容。提示可能包含源代码和敏感数据。
- 仅将遥测路由到您控制的收集器；施加保留限制和访问控制，以符合您的合规要求。
- 将工具参数和输出视为敏感。尽可能在收集器或 SIEM 处进行编辑。
- 如果您不希望 Codex 在 `CODEX_HOME` 下保存会话记录，请检查本地数据保留设置（例如，`history.persistence` / `history.max_bytes`）。有关详细信息，请参见 [高级配置](https://developers.openai.com/codex/config-advanced#history-persistence) 和 [配置参考](https://developers.openai.com/codex/config-reference)。
- 如果您在 CLI 中关闭网络访问，则 OTel 导出无法到达您的收集器。要导出，在 `workspace-write` 模式下为 OTel 端点允许网络访问，或通过 Codex 云从您的批准列表中导出收集器域。
- 定期审查事件以获取审批/沙箱更改和意外工具执行。

OTel 是可选的，旨在补充而非替代上述沙箱和审批保护。
