## OpenAI 身份验证

Codex 支持在使用 OpenAI 模型时，通过两种方式登录：

- 使用 ChatGPT 登录以获得订阅访问权限
- 使用 API 密钥登录以获得基于使用量的访问权限

Codex 云要求使用 ChatGPT 登录。Codex CLI 和 IDE 扩展支持两种登录方式。

您的登录方式还决定了适用哪些管理员控制和数据处理政策。

- 使用 ChatGPT 登录时，Codex 使用遵循您的 ChatGPT 工作区权限、RBAC 和 ChatGPT 企业保留和居留设置。
- 使用 API 密钥时，使用遵循您的 API 组织的保留和数据共享设置。

对于 CLI，当没有有效的会话时，以 ChatGPT 登录为默认身份验证路径。

### 使用 ChatGPT 登录

当您从 Codex 应用、CLI 或 IDE 扩展使用 ChatGPT 登录时，Codex 会为您打开一个浏览器窗口以完成登录流程。登录后，浏览器会将访问令牌返回给 CLI 或 IDE 扩展。

### 使用 API 密钥登录

您还可以使用 API 密钥登录 Codex 应用、CLI 或 IDE 扩展。请从 [OpenAI 仪表板](https://platform.openai.com/api-keys) 获取您的 API 密钥。

OpenAI 会通过您的 OpenAI 平台账户按标准 API 费率对 API 密钥的使用进行计费。请查看 [API 定价页面](https://openai.com/api/pricing/)。

依赖于 ChatGPT 余额的功能，例如 [快速模式](https://developers.openai.com/codex/speed)，仅在使用 ChatGPT 登录时可用。如果您使用 API 密钥登录，Codex 将使用标准 API 定价。

推荐在程序化的 Codex CLI 工作流中使用 API 密钥身份验证（例如 CI/CD 作业）。请勿在不受信任或公共环境中暴露 Codex 执行。

## 保护您的 Codex 云账户

Codex 云直接与您的代码库交互，因此需要比许多其他 ChatGPT 功能更强的安全性。启用多因素身份验证 (MFA)。

如果您使用社交登录提供商（谷歌、微软、苹果），则不需要在您的 ChatGPT 账户中启用 MFA，但您可以与您的社交登录提供商一起设置。

有关设置说明，请参见：

- [谷歌](https://support.google.com/accounts/answer/185839)
- [微软](https://support.microsoft.com/en-us/topic/what-is-multifactor-authentication-e5e39437-121c-be60-d123-eda06bddf661)
- [苹果](https://support.apple.com/en-us/102660)

如果您通过单点登录 (SSO) 访问 ChatGPT，您组织的 SSO 管理员应该为所有用户强制实施 MFA。

如果您使用电子邮件和密码登录，则必须在访问 Codex 云之前在您的账户上设置 MFA。

如果您的账户支持多种登录方式，其中一种是电子邮件和密码，则即使您以其他方式登录，也必须在访问 Codex 之前设置 MFA。

## 登录缓存

当您使用 ChatGPT 或 API 密钥登录 Codex 应用、CLI 或 IDE 扩展时，Codex 会缓存您的登录详情，并在您下次启动 CLI 或扩展时重用它们。CLI 和扩展共享相同的缓存登录详情。如果您从其中一个注销，则下次启动 CLI 或扩展时需要重新登录。

Codex 在 `~/.codex/auth.json` 的明文文件中或在您操作系统特定的凭证存储中本地缓存登录详情。

对于使用 ChatGPT 会话的登录，Codex 会在使用期间在令牌过期前自动刷新令牌，因此活跃会话通常会持续进行，而无需再次浏览器登录。

## 凭证存储

使用 `cli_auth_credentials_store` 来控制 Codex CLI 存储缓存凭证的位置：

```
# file | keyring | auto
cli_auth_credentials_store = "keyring"

```

- `file` 将凭证存储在 `CODEX_HOME` 下的 `auth.json`（默认为 `~/.codex`）。
- `keyring` 将凭证存储在您的操作系统凭证存储中。
- `auto` 在可用时使用操作系统凭证存储，否则回退到 `auth.json`。

如果您使用基于文件的存储，请将 `~/.codex/auth.json` 视为密码：它包含访问令牌。请勿提交、粘贴到工单中或在聊天中分享。

## 强制登录方法或工作区

在受管环境中，管理员可能会限制用户被允许的身份验证方式：

```
# 仅允许 ChatGPT 登录或仅允许 API 密钥登录。
forced_login_method = "chatgpt" # 或 "api"

# 在使用 ChatGPT 登录时，限制用户到特定工作区。
forced_chatgpt_workspace_id = "00000000-0000-0000-0000-000000000000"

```

如果活跃凭证与配置的限制不匹配，Codex 会注销用户并退出。

这些设置通常通过受管配置应用，而不是每个用户的设置。请参阅 [受管配置](https://developers.openai.com/codex/enterprise/managed-configuration)。

## 登录诊断

直接运行 `codex login` 会在您配置的日志目录下写入一个专门的 `codex-login.log` 文件。当您需要调试浏览器登录或设备代码故障，或当支持人员要求登录特定日志时，请使用它。

## 自定义 CA 包

如果您的网络使用企业 TLS 代理或私有根 CA，请在登录前将 `CODEX_CA_CERTIFICATE` 设置为 PEM 包。当 `CODEX_CA_CERTIFICATE` 未设置时，Codex 会回退到 `SSL_CERT_FILE`。相同的自定义 CA 设置适用于登录、正常 HTTPS 请求和安全的 websocket 连接。

```
export CODEX_CA_CERTIFICATE=/path/to/corporate-root-ca.pem
codex login

```

## 在无头设备上登录

如果您正在使用 Codex CLI 登录 ChatGPT，则在某些情况下，基于浏览器的登录 UI 可能无法正常工作：

- 您在远程或无头环境中运行 CLI。
- 您的本地网络配置阻止了 Codex 用于将 OAuth 令牌返回到 CLI 的 localhost 回调。

在这些情况下，建议使用设备代码身份验证（测试版）。在交互式登录 UI 中，选择 **使用设备代码登录**，或者直接运行 `codex login --device-auth`。如果设备代码身份验证在您的环境中不起作用，请使用其中一种备用方法。

### 首选：设备代码身份验证（测试版）

1. 在您的 ChatGPT 安全设置（个人账户）或 ChatGPT 工作区权限（工作区管理员）中启用设备代码登录。
2. 在运行 Codex 的终端中，选择以下选项之一：
   - 在交互式登录 UI 中，选择 **使用设备代码登录**。
   - 运行 `codex login --device-auth`。
3. 在浏览器中打开链接，登录，然后输入一次性代码。

如果服务器未启用设备代码登录，Codex 将回退到标准的基于浏览器的登录流程。

### 备用：在本地身份验证并复制您的身份验证缓存

如果您能够在具有浏览器的机器上完成登录流程，可以将缓存凭证复制到无头机器上。

1. 在可以使用基于浏览器的登录流程的机器上，运行 `codex login`。
2. 确认登录缓存存在于 `~/.codex/auth.json`。
3. 将 `~/.codex/auth.json` 复制到无头机器上的 `~/.codex/auth.json`。

将 `~/.codex/auth.json` 视为密码：它包含访问令牌。请勿提交、粘贴到工单中或在聊天中分享。

如果您的操作系统在凭证存储中而不是 `~/.codex/auth.json` 中存储凭证，则此方法可能不适用。请参阅 [凭证存储](https://developers.openai.com/codex/auth#credential-storage) 以了解如何配置基于文件的存储。

通过 SSH 复制到远程机器：

```
ssh user@remote 'mkdir -p ~/.codex'
scp ~/.codex/auth.json user@remote:~/.codex/auth.json

```

或者使用避免 `scp` 的单行命令：

```
ssh user@remote 'mkdir -p ~/.codex && cat > ~/.codex/auth.json' < ~/.codex/auth.json

```

复制到 Docker 容器中：

```
# 用您的容器名称或 ID 替换 MY_CONTAINER。
CONTAINER_HOME=$(docker exec MY_CONTAINER printenv HOME)
docker exec MY_CONTAINER mkdir -p "$CONTAINER_HOME/.codex"
docker cp ~/.codex/auth.json MY_CONTAINER:"$CONTAINER_HOME/.codex/auth.json"

```

有关在受信任的 CI/CD 运行程序上使用相同模式的更高级版本，请参阅 [在 CI/CD 中维护 Codex 账户身份验证（高级）](https://developers.openai.com/codex/auth/ci-cd-auth)。该指南解释了如何让 Codex 在正常运行期间刷新 `auth.json`，然后在下一个作业中保持更新的文件。API 密钥仍然是推荐的默认值。

### 备用：通过 SSH 转发 localhost 回调

如果您可以在本地机器和远程主机之间转发端口，则可以通过隧道 Codex 的本地回调服务器（默认 `localhost:1455`）使用标准的基于浏览器的流程。

1. 从本地机器启动端口转发：

```
ssh -L 1455:localhost:1455 user@remote

```

2. 在该 SSH 会话中，运行 `codex login` 并按照本地机器上打印的地址进行操作。

## 替代模型提供商

当您在配置文件中定义 [自定义模型提供商](https://developers.openai.com/codex/config-advanced#custom-model-providers) 时，可以选择以下身份验证方法之一：

- **OpenAI 身份验证**：设置 `requires_openai_auth = true` 以使用 OpenAI 身份验证。然后，您可以使用 ChatGPT 或 API 密钥登录。这在您通过 LLM 代理服务器访问 OpenAI 模型时非常有用。当 `requires_openai_auth = true` 时，Codex 会忽略 `env_key`。
- **环境变量身份验证**：设置 `env_key = "<ENV_VARIABLE_NAME>"` 以使用来自名为 `<ENV_VARIABLE_NAME>` 的本地环境变量的特定于提供者的 API 密钥。
- **无身份验证**：如果您不设置 `requires_openai_auth`（或将其设置为 `false`），而且不设置 `env_key`，Codex 假定提供商不需要身份验证。这在本地模型中是有用的。
