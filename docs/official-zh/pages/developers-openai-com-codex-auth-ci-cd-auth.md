本指南显示了如何在受信任的 CI/CD 运行器上保持 ChatGPT 管理的 Codex 身份验证有效，而无需自己调用 OAuth 令牌端点。

自动化的正确认证方式是使用 API 密钥。仅在您特别需要作为 Codex 帐户运行工作流时使用此指南。

模式如下：

1. 在受信任的机器上使用 `codex login` 创建一次 `auth.json`。
2. 将该文件放在运行器上。
3. 正常运行 Codex。
4. 让 Codex 在会话失效时刷新。
5. 保留更新后的 `auth.json` 以供下次运行。

这是一个企业和其他受信任私有自动化的高级工作流。对于大多数 CI/CD 作业，API 密钥仍然是推荐的选项。

将 `~/.codex/auth.json` 当作密码对待：它包含访问令牌。不要提交它、粘贴到票据中或在聊天中共享。不要在公共或开源代码库中使用此工作流。

## 为什么这有效

Codex 已经知道如何刷新 ChatGPT 管理的会话。

根据当前开源客户端：

- Codex 从 `auth.json` 加载本地授权缓存
- 如果 `last_refresh` 超过约 8 天，Codex 会在运行继续之前刷新令牌束
- 在成功刷新后，Codex 将新的令牌和新的 `last_refresh` 写回 `auth.json`
- 如果请求收到 `401`，Codex 也有内置的刷新和重试路径

这意味着支持的 CI/CD 策略不是“自行调用刷新 API”。而是“运行 Codex 并持久化更新后的 `auth.json`”。

## 何时使用此指南

仅在以下所有情况下使用此指南：

- 您需要 ChatGPT 管理的 Codex 身份验证而不是 API 密钥
- `codex login` 无法在远程运行器上运行
- 运行器是受信任的私有基础设施
- 您可以在运行之间保持更新后的 `auth.json`
- 仅一个机器或串行作业流将使用给定的 `auth.json` 副本

该指南适用于 Codex 管理的 ChatGPT 身份验证（`auth_mode: "chatgpt"`）。

不适用于：

- API 密钥身份验证
- 外部令牌主机集成（`auth_mode: "chatgptAuthTokens"`）
- Codex 以外的通用OAuth客户端

如果您的凭据存储在操作系统密钥环中，请先切换到文件支持的存储。请参见 [凭据存储](https://developers.openai.com/codex/auth#credential-storage)。

## 一次性种子 `auth.json`

在可以进行浏览器登录的受信任机器上：

1. 配置 Codex 将凭据存储在文件中：

```
cli_auth_credentials_store = "file"

```

2. 运行：

```
codex login

```

3. 验证文件看起来像是管理的 ChatGPT 身份验证：

```
AUTH_FILE="${CODEX_HOME:-$HOME/.codex}/auth.json"

jq '{
  auth_mode,
  has_tokens: (.tokens != null),
  has_refresh_token: ((.tokens.refresh_token // "") != ""),
  last_refresh
}' "$AUTH_FILE"

```

仅在以下情况下继续：

- `auth_mode` 是 `"chatgpt"`
- `has_refresh_token` 为 `true`

然后将 `auth.json` 的内容放入您的 CI/CD 秘密管理器中，或将其复制到受信任的持久运行器。

## 推荐模式：在自托管运行器上使用 GitHub Actions

最简单的完全自动化设置是一个具有持久 `CODEX_HOME` 的自托管 GitHub Actions 运行器。

为什么这个模式效果很好：

- 运行器可以在作业之间将 `auth.json` 保持在磁盘上
- Codex 可以原地刷新文件
- 后续作业自动获取更新的令牌
- 您只需要原始机密进行引导或重新种子

关键细节是仅在缺失时种子 `auth.json`。如果您在每次运行时从原始机密重写文件，您将丢弃 Codex 刚刚写入的更新的令牌。

示例定时工作流：

```
name: Keep Codex auth fresh

on:
  schedule:
    - cron: "0 9 * * 1"
  workflow_dispatch:

jobs:
  keep-codex-auth-fresh:
    runs-on: self-hosted
    steps:
      - name: Bootstrap auth.json if needed
        shell: bash
        env:
          CODEX_AUTH_JSON: ${{ secrets.CODEX_AUTH_JSON }}
        run: |
          export CODEX_HOME="${CODEX_HOME:-$HOME/.codex}"
          mkdir -p "$CODEX_HOME"
          chmod 700 "$CODEX_HOME"

          if [ ! -f "$CODEX_HOME/auth.json" ]; then
            printf '%s' "$CODEX_AUTH_JSON" > "$CODEX_HOME/auth.json"
            chmod 600 "$CODEX_HOME/auth.json"
          fi

      - name: Run Codex
        shell: bash
        run: |
          codex exec --json "Reply with the single word OK." >/dev/null

```

这段代码的作用：

- 第一次运行种子 `auth.json`
- 后续运行重用同一文件
- 一旦缓存的会话足够旧，Codex 就会在正常的 `codex exec` 步骤中刷新它
- 刷新的文件将保留在磁盘上以供下次工作流运行

每周计划通常足够，因为 Codex 在当前开源客户端中会在约 8 天后将会话视为过期。

## 短暂运行器：恢复、运行 Codex、持久化更新的文件

如果您使用 GitHub 托管的运行器、GitLab 共享运行器或任何其他短暂环境，运行器的文件系统在每个作业后会消失。在该设置中，您需要进行一个往返：

1. 从安全存储恢复当前的 `auth.json`
2. 运行 Codex
3. 将更新的 `auth.json` 写回安全存储

通用 GitHub Actions 模板：

```
name: Run Codex with managed auth

on:
  workflow_dispatch:

jobs:
  codex-job:
    runs-on: ubuntu-latest
    steps:
      - name: Restore auth.json
        shell: bash
        run: |
          export CODEX_HOME="${CODEX_HOME:-$HOME/.codex}"
          mkdir -p "$CODEX_HOME"
          chmod 700 "$CODEX_HOME"

          # 将此替换为您的秘密管理器或安全存储命令。
          my-secret-cli read codex-auth-json > "$CODEX_HOME/auth.json"
          chmod 600 "$CODEX_HOME/auth.json"

      - name: Run Codex
        shell: bash
        run: |
          codex exec --json "summarize the failing tests"

      - name: Persist refreshed auth.json
        if: always()
        shell: bash
        run: |
          # 将此替换为您的秘密管理器或安全存储命令。
          my-secret-cli write codex-auth-json < "$CODEX_HOME/auth.json"

```

关键要求是写回步骤存储的是 Codex 在运行期间生成的更新文件，而不是原始种子。

## 您不需要单独的刷新命令

任何正常的 Codex 运行都可以刷新会话。

这意味着您有两个不错的选择：

- 让您现有的 CI/CD Codex 作业自然刷新文件
- 添加一个轻量级的定时维护作业，例如上面的 GitHub Actions 示例，前提是您的实际作业运行得不够频繁

会话首次变得过期后，Codex 的第一次运行将刷新 `auth.json`。

## 重要的操作规则

- 每个运行器或每个串行工作流流使用一个 `auth.json`。
- 不要在并发作业或多台机器之间共享同一个文件。
- 不要在每次运行时从原始种子覆盖持续运行器的刷新文件。
- 不要将 `auth.json` 存储在代码库、日志或公共工件存储中。
- 如果内置刷新停止工作，则从受信任的机器重新种子。

## 刷新停止工作时该怎么办

此流程减少了手动工作，但不保证相同的会话会持续到永远。

如果：

- Codex 开始返回 `401`，并且运行器无法再刷新
- 刷新令牌被撤销或过期
- 另一台机器或并发作业首先旋转了令牌
- 您的安全存储往返失败并恢复了旧文件

请重新种子：

1. 在受信任的机器上运行 `codex login`。
2. 替换存储的 CI/CD 版本的 `auth.json`。
3. 让下一个运行器作业继续使用 Codex 内置的刷新流。

## 验证运行器是否维护会话

检查运行器是否仍然拥有管理的授权令牌，以及 `last_refresh` 是否存在：

```
AUTH_FILE="${CODEX_HOME:-$HOME/.codex}/auth.json"

jq '{
  auth_mode,
  last_refresh,
  has_access_token: ((.tokens.access_token // "") != ""),
  has_id_token: ((.tokens.id_token // "") != ""),
  has_refresh_token: ((.tokens.refresh_token // "") != "")
}' "$AUTH_FILE"

```

如果您的运行器是持久的，您应该看到同一文件在运行之间继续存在。如果您的运行器是短暂的，请确认您的写回步骤正在存储来自最后一个作业的更新文件。
