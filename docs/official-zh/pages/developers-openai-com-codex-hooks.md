钩子是 Codex 的扩展框架。它们允许您将自己的脚本注入到代理循环中，实现以下功能：

- 将对话发送到自定义日志/分析引擎
- 扫描团队的提示，以阻止意外粘贴 API 密钥
- 自动总结对话以创建持久记忆
- 在对话回合停止时运行自定义验证检查，以强制执行标准
- 在特定目录中自定义提示

钩子在 `config.toml` 中处于特性标志下：

```
[features]
codex_hooks = true

```

需要记住的运行时行为：

- 来自多个文件的匹配钩子均会运行。
- 针对同一事件的多个匹配命令钩子会并发启动，因此一个钩子不能阻止另一个匹配钩子的启动。
- `PreToolUse`、`PermissionRequest`、`PostToolUse`、`UserPromptSubmit` 和 `Stop` 在回合范围内运行。

## Codex 查找钩子的地方

Codex 在以下两种形式的活动配置层旁发现钩子：

- `hooks.json`
- `config.toml` 中的内联 `[hooks]` 表

在实践中，最有用的四个位置是：

- `~/.codex/hooks.json`
- `~/.codex/config.toml`
- `<repo>/.codex/hooks.json`
- `<repo>/.codex/config.toml`

如果存在多个钩子源，Codex 将加载所有匹配的钩子。较高优先级的配置层不会替换较低优先级的钩子。如果单个层同时包含 `hooks.json` 和内联 `[hooks]`，Codex 会将它们合并并在启动时发出警告。每个层应优先选择一种表示方式。

项目本地钩子仅在信任项目的 `.codex/` 层时加载。在不受信任的项目中，Codex 仍然会从其自身活动配置层加载用户和系统钩子。

## 配置形状

钩子分为三个层次：

- 钩子事件，例如 `PreToolUse`、`PostToolUse` 或 `Stop`
- 匹配器组，用于决定何时匹配该事件
- 一个或多个钩子处理程序，在匹配器组匹配时运行

```
{
  "hooks": {
    "SessionStart": [\
      {\
        "matcher": "startup|resume",\
        "hooks": [\
          {\
            "type": "command",\
            "command": "python3 ~/.codex/hooks/session_start.py",\
            "statusMessage": "加载会话笔记"\
          }\
        ]\
      }\
    ],
    "PreToolUse": [\
      {\
        "matcher": "Bash",\
        "hooks": [\
          {\
            "type": "command",\
            "command": "/usr/bin/python3 \"$(git rev-parse --show-toplevel)/.codex/hooks/pre_tool_use_policy.py\"",\
            "statusMessage": "检查 Bash 命令"\
          }\
        ]\
      }\
    ],
    "PermissionRequest": [\
      {\
        "matcher": "Bash",\
        "hooks": [\
          {\
            "type": "command",\
            "command": "/usr/bin/python3 \"$(git rev-parse --show-toplevel)/.codex/hooks/permission_request.py\"",\
            "statusMessage": "检查审批请求"\
          }\
        ]\
      }\
    ],
    "PostToolUse": [\
      {\
        "matcher": "Bash",\
        "hooks": [\
          {\
            "type": "command",\
            "command": "/usr/bin/python3 \"$(git rev-parse --show-toplevel)/.codex/hooks/post_tool_use_review.py\"",\
            "statusMessage": "审核 Bash 输出"\
          }\
        ]\
      }\
    ],
    "UserPromptSubmit": [\
      {\
        "hooks": [\
          {\
            "type": "command",\
            "command": "/usr/bin/python3 \"$(git rev-parse --show-toplevel)/.codex/hooks/user_prompt_submit_data_flywheel.py\""\
          }\
        ]\
      }\
    ],
    "Stop": [\
      {\
        "hooks": [\
          {\
            "type": "command",\
            "command": "/usr/bin/python3 \"$(git rev-parse --show-toplevel)/.codex/hooks/stop_continue.py\"",\
            "timeout": 30\
          }\
        ]\
      }\
    ]
  }
}

```

注意：

- `timeout` 以秒为单位。
- 如果省略 `timeout`，Codex 会使用 `600` 秒。
- `statusMessage` 是可选的。
- 命令在会话 `cwd` 下运行，作为其工作目录。
- 对于仓库本地钩子，建议从 git 根目录解决，而不是使用相对路径，例如 `.codex/hooks/...`。Codex 可能是在子目录中启动的，并且基于 git 根目录的路径可以保持钩子位置的稳定。

等价的内联 TOML 展示在 `config.toml` 中：

```
[features]
codex_hooks = true

[[hooks.PreToolUse]]
matcher = "^Bash$"

[[hooks.PreToolUse.hooks]]
type = "command"
command = '/usr/bin/python3 "$(git rev-parse --show-toplevel)/.codex/hooks/pre_tool_use_policy.py"'
timeout = 30
statusMessage = "检查 Bash 命令"

[[hooks.PostToolUse]]
matcher = "^Bash$"

[[hooks.PostToolUse.hooks]]
type = "command"
command = '/usr/bin/python3 "$(git rev-parse --show-toplevel)/.codex/hooks/post_tool_use_review.py"'
timeout = 30
statusMessage = "审核 Bash 输出"

```

## 从 `requirements.toml` 管理的钩子

企业管理的需求也可以在 `[hooks]` 下定义钩子。这在管理员希望强制执行钩子配置的情况下非常有用，同时通过 MDM 或其他设备管理系统提供实际脚本。

```
[features]
codex_hooks = true

[hooks]
managed_dir = "/enterprise/hooks"
windows_managed_dir = 'C:\enterprise\hooks'

[[hooks.PreToolUse]]
matcher = "^Bash$"

[[hooks.PreToolUse.hooks]]
type = "command"
command = "python3 /enterprise/hooks/pre_tool_use_policy.py"
timeout = 30
statusMessage = "检查管理的 Bash 命令"

```

管理钩子的注意事项：

- `managed_dir` 在 macOS 和 Linux 上使用。
- `windows_managed_dir` 在 Windows 上使用。
- Codex 不会分发 `managed_dir` 中的脚本；您的企业工具必须单独安装和更新它们。
- 管理钩子命令应在配置的管理目录下使用绝对脚本路径。

## 匹配器模式

`matcher` 字段是过滤钩子触发的正则表达式字符串。使用 `"*"`, `""`，或完全省略 `matcher` 以匹配每个支持事件的出现。

当前，只有某些 Codex 事件认可 `matcher`：

| 事件 | `matcher` 过滤内容 | 备注 |
| --- | --- | --- |
| `PermissionRequest` | 工具名称 | 支持包括 `Bash`、`apply_patch`\* 和 MCP 工具名称 |
| `PostToolUse` | 工具名称 | 支持包括 `Bash`、`apply_patch`\* 和 MCP 工具名称 |
| `PreToolUse` | 工具名称 | 支持包括 `Bash`、`apply_patch`\* 和 MCP 工具名称 |
| `SessionStart` | 启动源 | 当前运行时值包括 `startup`、`resume` 和 `clear` |
| `UserPromptSubmit` | 不支持 | 此事件将忽略任何配置的 `matcher` |
| `Stop` | 不支持 | 此事件将忽略任何配置的 `matcher` |

\*对于 `apply_patch`，匹配器也可以使用 `Edit` 或 `Write`。

示例：

- `Bash`
- `^apply_patch$`
- `Edit|Write`
- `mcp__filesystem__read_file`
- `mcp__filesystem__.*`
- `startup|resume|clear`

## 常见输入字段

每个命令钩子在 `stdin` 上接收一个 JSON 对象。

这些是您通常会使用的共享字段：

| 字段 | 类型 | 意义 |
| --- | --- | --- |
| `session_id` | `string` | 当前会话或线程 ID。 |
| `transcript_path` | `string | null` | 会话成绩单文件的路径（如果有） |
| `cwd` | `string` | 会话的工作目录 |
| `hook_event_name` | `string` | 当前钩子事件名称 |
| `model` | `string` | 活动模型缩写 |

回合范围的钩子在其事件特定表中列出 `turn_id`。

如果您需要完整的电缆格式，请参见 [Schemas](https://developers.openai.com/codex/hooks#schemas)。

## 常见输出字段

`SessionStart`、`UserPromptSubmit` 和 `Stop` 支持这些共享 JSON 字段：

```
{
  "continue": true,
  "stopReason": "可选",
  "systemMessage": "可选",
  "suppressOutput": false
}

```

| 字段 | 效果 |
| --- | --- |
| `continue` | 如果为 `false`，标记该钩子运行已停止 |
| `stopReason` | 记录停止原因 |
| `systemMessage` | 在 UI 或事件流中显示的警告 |
| `suppressOutput` | 今日解析但未实现 |

退出 `0` 且没有输出被视为成功，Codex 继续。

`PreToolUse` 和 `PermissionRequest` 支持 `systemMessage`，但 `continue`、`stopReason` 和 `suppressOutput` 当前不支持这些事件。

`PostToolUse` 支持 `systemMessage`、`continue: false` 和 `stopReason`。`suppressOutput` 被解析但当前不支持该事件。

## 钩子

### SessionStart

`matcher` 应用于该事件的 `source`。

除了 [常见输入字段](https://developers.openai.com/codex/hooks#common-input-fields) 之外的字段：

| 字段 | 类型 | 意义 |
| --- | --- | --- |
| `source` | `string` | 会话开始的方式：`startup` 或 `resume` |

在 `stdout` 上的纯文本作为额外的开发者上下文添加。

在 `stdout` 上的 JSON 支持 [常见输出字段](https://developers.openai.com/codex/hooks#common-output-fields) 和此钩子特定形状：

```
{
  "hookSpecificOutput": {
    "hookEventName": "SessionStart",
    "additionalContext": "在编辑之前加载工作区约定。"
  }
}

```

该 `additionalContext` 文本被添加为额外的开发者上下文。

### PreToolUse

`PreToolUse` 可以拦截 Bash、通过 `apply_patch` 执行的文件编辑和 MCP 工具调用。它仍然是一个保护措施，而不是完整的执行边界，因为 Codex 通常可以通过另一种支持的工具路径执行等效工作。

这尚未拦截所有的 shell 调用，只是简单的调用。更新的 `unified_exec` 机制允许丰富的流处理 stdin/stdout 的 shell，但拦截仍然不完整。同样，这尚未拦截 `WebSearch` 或其他非 shell、非 MCP 工具调用。

`matcher` 应用于 `tool_name` 和匹配器别名。通过 `apply_patch` 进行文件编辑时，匹配器可以使用 `apply_patch`、`Edit` 或 `Write`；钩子输入仍然报告 `tool_name: "apply_patch"`。

除了 [常见输入字段](https://developers.openai.com/codex/hooks#common-input-fields) 之外的字段：

| 字段 | 类型 | 意义 |
| --- | --- | --- |
| `turn_id` | `string` | Codex 特定扩展。活动 Codex 回合 ID |
| `tool_name` | `string` | 典型的钩子工具名称，例如 `Bash`、`apply_patch`、或 MCP 名称如 `mcp__fs__read` |
| `tool_use_id` | `string` | 此调用的工具调用 ID |
| `tool_input` | `JSON 值` | 工具特定输入。`Bash` 和 `apply_patch` 使用 `tool_input.command`，而 MCP 工具发送所有参数。 |

在 `stdout` 上的纯文本会被忽略。

在 `stdout` 上的 JSON 可以使用 `systemMessage` 并可以使用此钩子特定形状阻止 Bash 命令：

```
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "钩子阻止了破坏性命令。"
  }
}

```

Codex 也接受此较旧的阻止形状：

```
{
  "decision": "block",
  "reason": "钩子阻止了破坏性命令。"
}

```

您还可以使用退出代码 `2` 并将阻止原因写入 `stderr`。

`permissionDecision: "allow"` 和 `"ask"`、传统的 `decision: "approve"`、`updatedInput`、`additionalContext`、`continue: false`、`stopReason` 和 `suppressOutput` 已解析但尚未支持，因此它们失败时是开放的。

### PermissionRequest

`PermissionRequest` 在 Codex 即将请求批准（例如，提权或受管理网络批准）时运行。它可以允许请求、拒绝请求，或者拒绝决定并让正常的批准提示继续。它不针对不需要批准的命令运行。

`matcher` 应用于 `tool_name` 和匹配器别名。当前典型值包括 `Bash`、`apply_patch` 和 MCP 工具名称，例如 `mcp__server__tool`；`apply_patch` 也匹配 `Edit` 和 `Write`。

除了 [常见输入字段](https://developers.openai.com/codex/hooks#common-input-fields) 之外的字段：

| 字段 | 类型 | 意义 |
| --- | --- | --- |
| `turn_id` | `string` | Codex 特定扩展。活动 Codex 回合 ID |
| `tool_name` | `string` | 典型的钩子工具名称，例如 `Bash`、`apply_patch`，或 MCP 名称如 `mcp__fs__read` |
| `tool_input` | `JSON 值` | 工具特定输入。`Bash` 和 `apply_patch` 使用 `tool_input.command`，而 MCP 工具发送所有参数。 |
| `tool_input.description` | `string | null` | 可人读的批准原因，当 Codex 拥有一个时 |

在 `stdout` 上的纯文本会被忽略。

要批准请求，返回：

```
{
  "hookSpecificOutput": {
    "hookEventName": "PermissionRequest",
    "decision": {
      "behavior": "allow"
    }
  }
}

```

要拒绝请求，返回：

```
{
  "hookSpecificOutput": {
    "hookEventName": "PermissionRequest",
    "decision": {
      "behavior": "deny",
      "message": "被仓库政策阻止。"
    }
  }
}

```

如果多个匹配钩子返回决策，则任何 `deny` 优先。如没有匹配的钩子做出决策，Codex 使用正常的批准流程。

不要为 `PermissionRequest` 返回 `updatedInput`、`updatedPermissions` 或 `interrupt`；这些字段保留给未来行为且今天无法使用。

### PostToolUse

`PostToolUse` 在支持的工具生成输出后运行，包括 Bash、`apply_patch` 和 MCP 工具调用。对于 Bash，它还会在以非零状态退出的命令后运行。它无法撤销已经运行的工具的副作用。

这尚未拦截所有的 shell 调用，只是简单的调用。更新的 `unified_exec` 机制允许丰富的流处理 stdin/stdout 的 shell，但拦截仍然不完整。同样，这尚未拦截 `WebSearch` 或其他非 shell、非 MCP 工具调用。

`matcher` 应用于 `tool_name` 和匹配器别名。通过 `apply_patch` 进行文件编辑时，匹配器可以使用 `apply_patch`、`Edit` 或 `Write`；钩子输入仍然报告 `tool_name: "apply_patch"`。

除了 [常见输入字段](https://developers.openai.com/codex/hooks#common-input-fields) 之外的字段：

| 字段 | 类型 | 意义 |
| --- | --- | --- |
| `turn_id` | `string` | Codex 特定扩展。活动 Codex 回合 ID |
| `tool_name` | `string` | 典型的钩子工具名称，例如 `Bash`、`apply_patch` 或 MCP 名称如 `mcp__fs__read` |
| `tool_use_id` | `string` | 此调用的工具调用 ID |
| `tool_input` | `JSON 值` | 工具特定输入。`Bash` 和 `apply_patch` 使用 `tool_input.command`，而 MCP 工具发送所有参数。 |
| `tool_response` | `JSON 值` | 工具特定输出。对于 MCP 工具，这是 MCP 调用结果。 |

在 `stdout` 上的纯文本会被忽略。

在 `stdout` 上的 JSON 可以使用 `systemMessage` 和此钩子特定形状：

```
{
  "decision": "block",
  "reason": "Bash 输出需要审核后继续。",
  "hookSpecificOutput": {
    "hookEventName": "PostToolUse",
    "additionalContext": "该命令更新了生成的文件。"
  }
}

```

该 `additionalContext` 文本被添加为额外的开发者上下文。

对于此事件，`decision: "block"` 并不会撤销已完成的 Bash 命令。相反，Codex 会记录反馈，使用该反馈替换工具结果，并从钩子提供的消息继续处理模型。

您还可以使用退出代码 `2` 并将反馈原因写入 `stderr`。

要在命令已经运行后停止原始工具结果的正常处理，返回 `continue: false`。Codex 将使用您的反馈或停止文本替换工具结果，并从那里继续。

`updatedMCPToolOutput` 和 `suppressOutput` 已解析但不支持，因此它们失败时是开放的。

### UserPromptSubmit

`matcher` 当前未在此事件中使用。

除了 [常见输入字段](https://developers.openai.com/codex/hooks#common-input-fields) 之外的字段：

| 字段 | 类型 | 意义 |
| --- | --- | --- |
| `turn_id` | `string` | Codex 特定扩展。活动 Codex 回合 ID |
| `prompt` | `string` | 即将发送的用户提示 |

在 `stdout` 上的纯文本作为额外的开发者上下文添加。

在 `stdout` 上的 JSON 支持 [常见输出字段](https://developers.openai.com/codex/hooks#common-output-fields) 和此钩子特定形状：

```
{
  "hookSpecificOutput": {
    "hookEventName": "UserPromptSubmit",
    "additionalContext": "在编辑文件之前请求更清晰的复制。"
  }
}

```

该 `additionalContext` 文本被添加为额外的开发者上下文。

要阻止提示，返回：

```
{
  "decision": "block",
  "reason": "在执行之前请求确认。"
}

```

您也可以使用退出代码 `2` 并将阻止原因写入 `stderr`。

### Stop

`matcher` 当前未在此事件中使用。

除了 [常见输入字段](https://developers.openai.com/codex/hooks#common-input-fields) 之外的字段：

| 字段 | 类型 | 意义 |
| --- | --- | --- |
| `turn_id` | `string` | Codex 特定扩展。活动 Codex 回合 ID |
| `stop_hook_active` | `boolean` | 此回合是否已通过 `Stop` 被继续 |
| `last_assistant_message` | `string | null` | 最新的助手消息文本（如果可用） |

`Stop` 在退出 `0` 时期待 `stdout` 上的 JSON。纯文本输出对该事件无效。

在 `stdout` 上的 JSON 支持 [常见输出字段](https://developers.openai.com/codex/hooks#common-output-fields)。要让 Codex 继续，返回：

```
{
  "decision": "block",
  "reason": "对失败的测试进行一次额外的处理。"
}

```

您也可以使用退出代码 `2` 并将继续原因写入 `stderr`。

对于此事件，`decision: "block"` 并不会拒绝该回合。相反，它告诉 Codex 继续并自动创建一个新的继续提示，该提示用您的 `reason` 作为该提示文本。

如果任何匹配的 `Stop` 钩子返回 `continue: false`，则优先于其他匹配的 `Stop` 钩子中的继续决策。
